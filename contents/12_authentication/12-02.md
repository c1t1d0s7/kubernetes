## 2. 역할 기반 접근 제어

### 1) 인증, 권한 그리고 접근 제어
쿠버네티스 클러스터의 API 서버에 접근하기 위해서는 사용자 자격증명 정보를 제공하여 인증을 해야한다. 인증을 한 후에는 API를 사용할 권한이 있는지 확인하게 된다.

하나의 쿠버네티스 클러스터를 여러 사용자가 같이 사용하는 경우, 요청하는 API 또는 네임스페이스, 리소스 별로 권한을 구분하여 접근을 제어할 수 있다.

A 사용자는 자신만 리소스를 관리할 수 있는 권한을 가진 네임스페이스를 가질 수 있으며, B 사용자는 해당 네임스페이스의 파드 리소스에 대해서 목록 및 정보를 확인만 할 수 있도록 권한을 구성할 수 있다.

지금까지 사용했던 컨텍스트의 권한은 클러스터 전체에 모든 권한을 가지고 있는 클러스터 관리자다.

쿠버네티스에서는 이러한 권한을 가지고 리소스에 대한 접근을 제어 할 수 있는데, 두가지 방법의 접근 제어 방식을 제공한다.
- 속성 기반의 접근 제어(ABAC: Attributed-based Access Control)
- 역할 기반의 접근 제어(RBAC: Role-based Access Control)

ABAC는 쿠버네티스 초기부터 사용하던 접근 제어 방식이다. 속성을 이용한 권한을 관리하며, 사용자, API 요청 경로, 요청 동사(Verb), 네임스페이스, 리소스 등을 선언해 제어할 수 있다. 그러나 지금은 잘 사용하지 않는데, 권한을 변경하려면 마스터 노드에 접근할 수 있어야 하며, 마스터에 접근해 권한 파일을 변경하고, API 서버를 재시작 해야한다. API 서버를 재시작하면 재시작이 완료될때 까지 다른 작업을 할 수 없을뿐만 아니라, 퍼블릭 클라우드의 관리형 쿠버네티스 서비스는 마스터에 접근할 수 없다.

현재는 일반적으로 RBAC 접근 제어 방식을 사용하며, 역할에 권한을 부여하고, 역할을 사용자에게 연결(Binding)하여 권한을 할당한다. RBAC는 권한을 변경하더라도 API 서버를 재시작할 필요가 없으며, 마스터에 접근을 할 필요없이 kubectl 명령을 이용해 관리할 수 있다.

#### RBAC 관련 리소스
RBAC는 권한을 부여할 수 있는 롤(Role)과 클러스터 롤(ClusterRole) 리소스가 있다.
역할에는 허용(Allow) 권한만 부여할 수 있으며, 거부(Deny) 권한은 없다. 명시적으로 허용되지 않는 권한은 모두 거부된다.
- 롤: 특정 네임스페이스 내에 권한을 설정한다.
- 클러스터 롤: 클러스터 전체에 권한을 설정한다.

이런 역할은 특정 사용자에게 부여할 수 있다. 역할과 사용자를 연결할 수 있는 롤 바인딩(RoleBinding)과 클러스터 롤 바인딩(ClusterRoleBinding) 리소스가 있다.
- 롤 바인딩: 네임스페이스 내의 롤 과 사용자를 연결한다.
- 클러스터 롤 바인딩: 클러스터 롤과 사용자를 연결한다.

```
롤        <-----롤 바인딩-----> 사용자(서비스 계정)
클러스터 롤 <--클러스터 롤 바인딩--> 사용자(서비스 계정)
```

RBAC 관련 리소스는 YAML 오브젝트를 이용해 생성할 수 있고, kubectl 명령을 이용해 생성할 수도 있다.
- kubectl create role
- kubectl create clusterrole
- kubectl create serviceaccount
- kubectl create rolebinding
- kubectl create clusterrolebinding

RBAC 관련 모든 리소스는 rbac.authorization.k8s.io API 리소스 그룹을 사용하며, 버전은 v1 이다.
아래 실습에서는 YAML 오브젝트를 이용한 방법만을 사용한다.

### 2) 롤

#### (1) 롤 오브젝트 선언
다음은 롤 오브젝트 주요 필드다.
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: role-example
rules:
- apiGroups: [""]
  nonResourceURLs: ["/api","/healthz"]
  resources: ["pods"]
  resourceNames: ["mynapp-pod"]
  verbs: ["get", "list"]
```
- .rules.apiGroups: 권한을 부여할 API 그룹을 지정한다. 지정하지 않으면 모든 API 그룹이 대상이 된다.
- .rules.nonResourceURLs: 특정 리소스를 대상으로 지정할 수 없는 비 리소스의 경우, API 주소 경로를 지정한다.
- .rules.resouces: 리소스 대상을 지정한다. 지정하지 않으면 모든 리소스가 대상이 된다.
- .rules.resourceNames: 특정 리소스 이름을 지정한다. 지정하지 않으면 모든 리소스가 대상이 된다.
- .rules.verbs(필수): API 요청 동사를 지정한다. 지정하지 않으면 모든 동사가 대상이 된다.

| API 요청 동사      | 의미              |
|------------------|------------------|
| get              | 개별 리소스 확인     |
| list             | 여러 리소스 확인     |
| create           | 새로운 리소스 생성   |
| update           | 기존 리소스 변경     |
| patch            | 기존 리소스 일부 변경 |
| watch            | 리소스 실시간 확인   |
| delete           | 개별 리소스 삭제     |
| deletecollection | 여러 리소스 삭제     |

#### (2) 롤 리소스 생성 및 확인
다음은 생성할 롤 리소스다.
> role-read-default-pod.yml
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: read-default-pod
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
```
default 네임스페이스 내에서 모든 API 그룹을 사용할 수 있고, 파드 리소스에 한해 개별/여러 리소스를 확인할 수 있는 권한이다.

```
$ kubectl create -f role-read-default-pod.yml

role.rbac.authorization.k8s.io/read-default-pod created
```

```
$ kubectl get role

NAME               AGE
read-default-pod   3s
```

```
$ kubectl describe role read-default-pod

Name:         read-default-pod
Labels:       <none>
Annotations:  <none>
PolicyRule:
  Resources  Non-Resource URLs  Resource Names  Verbs
  ---------  -----------------  --------------  -----
  pods       []                 []              [get list]
```

### 3) 클러스터 롤

#### (1) 클러스터 롤 오브젝트 선언
다음은 클러스터 롤 오브젝트 주요 필드다.
```yaml
kind: ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: clusterrole-example
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      kubernetes.io/bootstrapping: rbac-defaults
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get","list"]
```
기본적으로 롤 오브젝트의 구성과 거의 동일하다.

- .aggregationRules: 다른 클러스터 롤 여러개를 레이블을 이용해 집계/조합할 수 있다.

### 4) 서비스 계정

#### (1) 기본 서비스 계정 확인
쿠버네티스 클러스터에 기본적으로 생성되어 있는 서비스 계정을 확인해보자.
```
$ kubectl get sa

NAME      SECRETS   AGE
default   1         9d
```
default 라는 이름에 서비스 계정이 존재한다. 서비스 계정은 항상 인증에 사용할 토큰을 가지고 있다. 서비스 계정을 생성하면 시크릿이 생성되며, 해당 시크릿에는 CA 인증서, 네임스페이스 및 토큰이 저장되어 있다.

> 참고  
> default 서비스 계정 및 토큰은 앞서 학습한 시크릿 파트에서 확인 해본적이 있다.  
> 해당 시크릿의 토큰은 파드가 기본적으로 마운트하고 있으며, 파드가 API 서버와 통신할때 인증하기 위함이다. 

default 서비스 계정의 상세 정보를 확인하자.
```

Tokens:              default-token-x4tb2
Events:              <none>
```

토큰이 저장된 시크릿 이름을 확인할 수 있다.

### 5) 롤 바인딩

#### (1) 롤 바인딩 오브젝트 선언
다음은 롤 바인딩 오브젝트 주요 필드다.
```yaml
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: rolebind-example
  namespace: default
subjects:
- kind: ServiceAccount
  name: testuser
  apiGroup: ""
  namespaceclea: default
roleRef:
  kind: Role
  name: role-example
  apiGroup: rbac.authorization.k8s.io
```
- .subjects: 롤이 연결될 주체 (예: 서비스 계정)
- .roleRef: 주체에 부여할 롤 리소스

#### (2) 롤 바인딩 리소스 생성 및 확인
다음은 생성할 롤 바인딩 리소스다.
> rbind-myuser1-read.yml
```yaml
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: myuser1-read-default
  namespace: default
subjects:
- kind: ServiceAccount
  name: myuser1
  apiGroup: ""
  namespaces: default
roleRef:
  kind: Role
  name: read-default-pod
  apiGroup: rbac.authorization.k8s.io
```
myuser1-read-default 롤 바인딩 리소스며, myser1 서비스 계정과, read-default-pod 롤 리소스를 연결해준다.

롤 바인딩 리소스를 생성하자.
```
$ kubectl create -f rbind-myuser1-read.yml

```

롤 바인딩 목록을 확인해보자.
```
$ kubectl get rolebinding

NAME                   AGE
myuser1-read-default   6s
```

rbind-myuser1-read 롤 바인딩의 상세 정보를 확인하자.

read-default-pod 롤이 myuser1 서비스 계정에 연결된 것을 확인할 수 있다.

### 9) 기본 클러스터 롤
많은 기본 클러스터 롤 중 system: 접두사가 없는 클러스터 롤은 사용자를 위해 미리 정의해놓은 클러스터 롤이다.
- cluster-admin: 슈퍼 유저로 모든 관리 권한을 가진다.
- admin: 클러스터 롤/바인딩에 관한 모든 권한이 없으며, 리소스 할댱량 및 네임스페이스 자체에 대한 쓰기 권한이 없다. 이외 모든 권한을 가진다.
- edit: 대부분의 리소스에 대한 읽기/쓰기 권한을 가지지만, (클러스터)롤/바인딩에 관한 모든 권한이 없다.
- view: 대부분의 리소스에 대한 읽기 권한만 가지며, 시크릿은 볼 수 없다.

### 10) 기본 컨텍스트로 전환

```
$ kubectl config use-context kubernetes-admin@cluster.local

Switched to context "kubernetes-admin@cluster.local".
```
