## 3. 역할 기반 접근 제어
RBAC는 권한을 부여할 수 있는 롤(Role)과 클러스터 롤(ClusterRole) 리소스가 있다.
역할에는 허용(Allow) 권한만 부여할 수 있으며, 거부(Deny) 권한은 없다. 명시적으로 허용되지 않는 권한은 모두 거부된다.
- 롤: 특정 네임스페이스 내 권한 부여
- 클러스터 롤: 클러스터 전체 권한 부여

이런 역할은 특정 사용자에게 부여할 수 있다. 역할과 사용자를 연결할 수 있는 롤 바인딩(RoleBinding)과 클러스터 롤 바인딩(ClusterRoleBinding) 리소스가 있다.
- 롤 바인딩: 네임스페이스 내의 롤과 사용자 연결
- 클러스터 롤 바인딩: 클러스터 롤과 사용자 연결

```
롤        <-----롤 바인딩-----> 사용자
클러스터 롤 <--클러스터 롤 바인딩--> 사용자
```

RBAC 관련 리소스는 YAML 오브젝트를 이용해 생성할 수 있고, kubectl 명령을 이용해 생성할 수도 있다.
- kubectl create role
- kubectl create clusterrole
- kubectl create rolebinding
- kubectl create clusterrolebinding

RBAC 관련 모든 리소스는 rbac.authorization.k8s.io API 리소스 그룹을 사용하며, 버전은 v1 이다.

### 1) 롤

#### (1) 롤 오브젝트 선언
다음은 롤 오브젝트 주요 필드다.
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: role-example
rules:
- apiGroups: [""]
  nonResourceURLs: ["/api","/healthz"]
  resources: ["pods"]
  resourceNames: ["mynapp-pod"]
  verbs: ["get", "list"]
```
- .rules.apiGroups: 권한을 부여할 API 그룹
- .rules.nonResourceURLs: 비 리소스의 경우 API 주소 경로 지정
- .rules.resouces: 리소스 지정
- .rules.resourceNames: 특정 리소스 이름
- .rules.verbs(필수): API 요청 동사 지정

| API 요청 동사      | 의미              |
|------------------|------------------|
| get              | 개별 리소스 확인     |
| list             | 여러 리소스 확인     |
| create           | 새로운 리소스 생성   |
| update           | 기존 리소스 변경     |
| patch            | 기존 리소스 일부 변경 |
| watch            | 리소스 실시간 확인   |
| delete           | 개별 리소스 삭제     |
| deletecollection | 여러 리소스 삭제     |

#### (2) 롤 리소스 생성 및 확인
다음은 생성할 롤 리소스다.
> role-read-pod.yaml
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: read-pod
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
```
default 네임스페이스 내에서 모든 API 그룹을 사용할 수 있고, 파드 리소스에 한해 개별/여러 리소스를 확인할 수 있는 권한이다.

```
$ kubectl create -f role-read-pod.yaml

role.rbac.authorization.k8s.io/read-pod created
```

```
$ kubectl get role read-pod

NAME       CREATED AT
read-pod   2021-03-05T03:08:05Z
```

```
$ kubectl describe role read-pod

Name:         read-pod
Labels:       <none>
Annotations:  <none>
PolicyRule:
  Resources  Non-Resource URLs  Resource Names  Verbs
  ---------  -----------------  --------------  -----
  pods       []                 []              [get list]
```

### 2) 클러스터 롤

#### (1) 기본 클러스터 롤
많은 클러스터 롤 중 system: 접두사가 없는 클러스터 롤은 사용자를 위해 사전 정의된 클러스터 롤이다.
- cluster-admin: 슈퍼 유저로 모든 관리 권한을 가진다.
- admin: 클러스터 롤/바인딩에 관한 모든 권한이 없으며, 리소스 할댱량 및 네임스페이스 자체에 대한 쓰기 권한이 없다. 이외 모든 권한을 가진다.
- edit: 대부분의 리소스에 대한 읽기/쓰기 권한을 가지지만, (클러스터)롤/바인딩에 관한 모든 권한이 없다.
- view: 대부분의 리소스에 대한 읽기 권한만 가지며, 시크릿은 볼 수 없다.

#### (1) 클러스터 롤 오브젝트 선언
다음은 클러스터 롤 오브젝트 주요 필드다.
```yaml
kind: ClusterRole
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: clusterrole-example
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      kubernetes.io/bootstrapping: rbac-defaults
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get","list"]
```
기본적으로 롤 오브젝트의 구성과 거의 동일하다.

- .aggregationRules: 다른 클러스터 롤 여러 개를 레이블을 이용해 집계 할 수 있다.

#### (2) 클러스터 롤 리소스 생성 및 확인
다음은 생성할 클러스터 롤 리소스다.
> crole-read-pod.yaml
```yaml

  name: read-pod

- apiGroups: [""]
  resources: ["pods"]

```
클러스터 전체에서 모든 API 그룹을 사용할 수 있고, 파드 리소스에 한해 개별/여러 리소스를 확인할 수 있는 권한이다.

클러스터 롤 리소스를 생성하자.

### 3) 롤 바인딩

#### (1) 롤 바인딩 오브젝트 선언
다음은 롤 바인딩 오브젝트 주요 필드다.
```yaml
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: rolebind-example
  namespace: default
subjects:
- kind: ServiceAccount
  name: testuser
  apiGroup: ""
  namespaceclea: default
roleRef:
  kind: Role
  name: role-example
  apiGroup: rbac.authorization.k8s.io
```
- .subjects: 롤이 연결될 주체 (예: 사용자)
- .roleRef: 주체에 부여할 롤 리소스

#### (2) 롤 바인딩 리소스 생성 및 확인
다음은 생성할 롤 바인딩 리소스다.
> rbind-myuser1.yaml
```yaml
kind: RoleBinding
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  name: myuser1-read-pod
  namespace: default
subjects:
- kind: ServiceAccount
  name: myuser1
  apiGroup: ""
  namespace: default
roleRef:
  kind: Role
  name: read-pod
  apiGroup: rbac.authorization.k8s.io
```
myuser1-read-pod 롤 바인딩 리소스며, myuser1 서비스 어카운트와, read-pod 롤 리소스를 연결해준다.

롤 바인딩 리소스를 생성하자.
```
$ kubectl create -f rbind-myuser1.yaml

rolebinding.rbac.authorization.k8s.io/myuser1-read-pod created
```

롤 바인딩 목록을 확인해보자.
```
$ kubectl get rolebindings myuser1-read-pod

NAME                        ROLE                    AGE
myuser1-read-pod            Role/read-pod           24s
```

myuser1-read-pod 롤 바인딩의 상세 정보를 확인하자.
```
$ kubectl describe rolebinding myuser1-read-pod

Name:         myuser1-read-pod
Labels:       <none>
Annotations:  <none>
Role:
  Kind:  Role
  Name:  read-pod
Subjects:
  Kind            Name     Namespace
  ----            ----     ---------
  ServiceAccount  myuser1  default
```
read-pod 롤이 myuser1 서비스 어카운트에 연결된 것을 확인할 수 있다.

### 7) 기본 컨텍스트로 전환
마지막으로 원래 슈퍼유저 권한을 가지고 있는 kubernetes-admin@cluster.local 컨텍스트로 전환하자.
```
$ kubectl config use-context kubernetes-admin@cluster.local

Switched to context "kubernetes-admin@cluster.local".
```
