## 1. 인증 개요
모든 쿠버네티스 클러스터를 사용하는 사용자는 kubectl 클라이언트나 HTTP REST 요청을 통해 API 서버(kube-apiserver)에 접근한다. 해당 사용자는 제공한 인증 정보를 바탕으로 API 접근 권한을 인가 받을 수 있다. 

![인증-인가-승인제어(출처: kubernetes.io)](img/authentication.svg)

- 인증(Authentication): 클라이언트 식별 및 인증
- 인가(Authorization): 접근 권한 부여
- 승인 제어(Admission Control): 유효성 검사, 접근 제어, 오브젝트의 생성, 삭제, 수정에 대한 제한(읽기 X)

### 1) 인증

#### (1) 사용자
쿠버네티스 클러스터에는 쿠버네티스에서 관리하는 서비스 어카운트과 일반 사용자 어카운트가 있다.
- 서비스 어카운트(Service Account)
- 일반 사용자 어카운트(Normal User Account)

서비스 어카운트는 쿠버네티스 API로 관리되는 쿠버네티스에 존재하는 사용자다. 서비스 어카운트는 시크릿에 자격증명 토큰이 저장되어 있으며, API 요청시 해당 자격증명으로 인증을 수행하게 된다. 일반적으로 파드가 사용하기 위한 사용자다.

일반 사용자 어카운트(Normal User Account)는 쿠버네티스 자체적으로 관리하지 않는, 쿠버네티스 클러스터 외부의, 외부 계정 관리 시스템(AWS Identity and Access Management(IAM), OpenStack Keystone, LDAP, Kerberos, Active Directory, Google 계정 등 OAuth2 공급자)과 연동되어 인증에 사용되는 사용자를 말한다.

#### (2) 그룹
사용자는 하나 이상의 그룹에 속할 수 있다. 

#### (3) 인증 방법

- X.509 클라이언트 인증서
  * Root CA가 서명한 클라이언트 인증서
- 정적 토큰
  * HTTP Bearer 토큰 
  * HTTP 요청 헤더: ```Authorization: Bearer [TOKEN]```
- 부트 스트랩 토큰
  * 새 클러스터에 대해 부트 스트랩을 허용
  * HTTP Bearer 토큰
  * 컨트롤 플레인의 컨트롤러 관리자(kube-controller-manager)는 만료된 부트 스트랩 토큰을 삭제하는 TokenCleaner 컨트롤러 포함
- 서비스 어카운트 토큰
  * Root CA가 서명한 JWT(JSON Web Token) 토큰
  * 시크릿에 저장됨
- OpenID Connect(OIDC) 토큰
  * OAuth2 공급자(Provider)
  * AWS IAM, Microsoft Active Directory, OpenStack Keystone, LDAP 등
  * JWT 토큰

### 2) 인가
쿠버네티스 클러스터의 API 서버에 접근하기 위해서는 사용자 자격증명 정보를 제공하여 인증을 해야한다. 인증을 한 후에는 API를 사용할 권한이 있는지 확인하게 되는데 이것을 인가라 한다.

하나의 쿠버네티스 클러스터를 여러 사용자가 같이 사용하는 경우, 요청하는 API 또는 네임스페이스, 리소스 별로 권한을 구분하여 접근을 제어할 수 있다.

A 사용자는 자신만 리소스를 관리할 수 있는 권한을 가진 네임스페이스를 가질 수 있으며, B 사용자는 해당 네임스페이스의 파드 리소스에 대해서 목록 및 정보를 확인만 할 수 있도록 권한을 구성할 수 있다.

지금까지 사용했던 컨텍스트의 권한은 클러스터 전체에 모든 권한을 가지고 있는 클러스터 관리자다.

쿠버네티스에서는 이러한 권한을 가지고 리소스에 대한 접근을 제어 할 수 있는데, 두가지 방법의 접근 제어 방식을 제공한다.
- 속성 기반의 접근 제어(ABAC: Attributed-based Access Control)
- 역할 기반의 접근 제어(RBAC: Role-based Access Control)

ABAC는 쿠버네티스 초기부터 사용하던 접근 제어 방식이다. 속성을 이용한 권한을 관리하며, 사용자, API 요청 경로, 요청 동사(Verb), 네임스페이스, 리소스 등을 선언해 제어할 수 있다. 그러나 지금은 잘 사용하지 않는데, 권한을 변경하려면 마스터 노드에 접근할 수 있어야 하며, 마스터에 접근해 권한 파일을 변경하고, API 서버를 재시작 해야한다. API 서버를 재시작하면 재시작이 완료될때 까지 다른 작업을 할 수 없을뿐만 아니라, 퍼블릭 클라우드의 관리형 쿠버네티스 서비스는 마스터에 접근할 수 없다.

현재는 일반적으로 RBAC 접근 제어 방식을 사용하며, 역할에 권한을 부여하고, 역할을 사용자에게 연결(Binding)하여 권한을 할당한다. RBAC는 권한을 변경하더라도 API 서버를 재시작할 필요가 없으며, 마스터에 접근을 할 필요없이 kubectl 명령을 이용해 관리할 수 있다.
