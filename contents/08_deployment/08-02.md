## 2. 디플로이먼트
지금까지 확인했던 파드를 제어하는 컨트롤러보다 더 상위 수준의 컨트롤러 리소스가 있다. 그것이 디플로이먼트(Deployment) 리소스다.

디플로이먼트는 애플리케이션(컨트롤러, 파드)을 배포하고 선언적으로 업데이트를 수행하는 리소스이다. 디플로이먼트 리소스는, 하위에 레플리카셋 컨트롤러를 제어하고, 그리고 레플리카셋 컨트롤러가 하위 파드를 제어하는 구조이다.

디플로이먼트 리소스가 관리하는 하위 레플리카셋 컨트롤러를 직접 관리하지 말아야 한다. 디플로이먼트 리소스를 이용해 전체를 제어한다.

### 1) 디플로이먼트 정의

#### (1) 디플로이먼트 기본 사양
다음은 디플로이먼트 리소스의 YAML 오브젝트다.
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: mynapp-deploy
  labels:
    app: mynapp-deploy
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-deploy
  template:
    metadata:
      labels:
        app: mynapp-deploy
    spec:
      containers:
      - image: c1t1d0s7/myweb:v1
        name: mynapp
        ports:
        - containerPort: 8080
```
디플로이먼트 리소스의 필드는 레플리카셋의 필드와 특별히 다르지 않다.

#### (2) 디플로이먼트 전략
디플로이먼트 리소스에 만 적용할 수 있는 배로 전략 관련 필드는 다음과 같다.
```yaml
spec:
  strategy:
    type: [RollingUpdate | Recreate]
    rollingupdate:
      maxUnavailable: <정수 | 비율>
      maxSurge:
```

- RollingUpdate: 롤링업데이트 방식(기본값)
  * maxUnavailable: 롤링업데이트 프로세스 중에 사용할 수 없는 최대 파드의 수(기본값: 25%)
    - 정수: 예) 3
    - 비율: 예) 10%
  * maxSurge: 생성할 수 있는 최대 파드의 수(기본값: 25%)
    - 정수: 예) 3
    - 비율: 예) 10%
- Recreate: 재생성(기존 파드는 모두 삭제 되고, 새로운 파드 생성)

#### (3) 컨테이너 최소 대기 시간
롤링업데이트를 수행할 때 새로운 파드가 배포되고 배포된 파드의 컨테이너가 실제 준비되기 대기할 시간을 지정할 수 있다.
```yaml
spec:
  minReadySeconds: <초>
```
- minReadySeconds: 새로 배포된 컨테이너가 준비되기 까지 대기할 시간(기본값: 0초)

### 2) 디플로이먼트 생성

#### (1) 디플로이먼트 생성
다음은 디플로이먼트 리소스 오브젝트다.
> mynapp-deploy-v1.yml 파일
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: mynapp-deploy
  labels:
    app: mynapp-deploy
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 1
      maxSurge: 1
  minReadySeconds: 20
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-deploy
  template:
    metadata:
      labels:
        app: mynapp-deploy
    spec:
      containers:
      - image: c1t1d0s7/myweb:v1
        name: mynapp
        ports:
        - containerPort: 8080
```

디플로이먼트 리소스를 생성하자. --record 옵션을 붙여두자. 이 옵션은 뒤에서 다뤄보자.
```
$ kubectl create -f mynapp-deploy-v1.yml --record

deployment.apps/mynapp-deploy created
```

확인하기 위한 서비스도 생성하자.
```
$ kubectl create -f mynapp-svc-deploy.yml

service/mynapp-svc-deploy created
```

로드 밸런서 서비스의 외부 IP를 확인하자.
```
$ kubectl get services mynapp-svc-deploy

NAME                TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)        AGE
mynapp-svc-deploy   LoadBalancer   10.233.21.66   192.168.56.200   80:32724/TCP   90s
```

#### (2) 디플로이먼트 모터링 준비
새로운 터미널을 하나 추가로 열어 컨트롤러 및 파드의 상태를 확인한다.
```
$ watch -n1 -d kubectl get all
```

새로운 터미널을 하나 추가로 열어 웹 요청 및 응답을 모니터링 한다.
```
$ watch -n1 -d curl http://192.168.56.200
```

### 3) 디플로이먼트 롤아웃 상태 확인
디플로이먼트에서는 모든 배포 및 롤링업데이트를 롤아웃(Rollout)이라고 한다.

롤아웃 상태를 확인하는 방법은 다음과 같다.
```
$ kubectl rollout status deployment mynapp-deploy

deployment "mynapp-deploy" successfully rolled out
```
mynapp-deploy 디플로이먼트가 정상적으로 배포된 것을 확인할 수 있다.

롤아웃 히스토리를 확인하자.
```
$ kubectl rollout history deployment mynapp-deploy

deployment.apps/mynapp-deploy
REVISION  CHANGE-CAUSE
1         kubectl create --filename=mynapp-deploy-v1.yml --record=true
```
앞서 디플로이먼트 리소스 생성 시 --record 옵션을 사용하여 생성하면 변경 사유(CHANGE-CAUSE)에 해당 명령이 기록된다. 그리고 디플로이먼트의 리비전(REVISION)이 남게 되는게 이를 이용해 무엇 때문에 리소스가 변경되었는지, 필요하다면 롤백(Rollback)을 할 수 있다.


### 4) 디플로이먼트 롤링업데이트

#### (1) 롤링업데이트 명령 형식
```
kubectl set image deployment <디플로이먼트 이름> <컨테이너 이름>=<새 이미지>
kubectl set image deployment -f <디플로이먼트 파일> <컨테이너 이름>=<새 이미지>
```

#### (2) 버전1 -> 버전2
버전1에서 버전2로 업데이트 하자.
```
$ kubectl set image deployment mynapp-deploy mynapp=c1t1d0s7/myweb:v2

deployment.apps/mynapp-deploy image updated
```

> 참고  
> 디플로이먼트 롤아웃은 kubectl 클라이언트가 직접 실행하지 않는 것을 알 수 있다.  

롤아웃 상태를 확인 해보자.
```
$ kubectl rollout status deployment mynapp-deploy

Waiting for deployment "mynapp-deploy" rollout to finish: 2 out of 3 new replicas have been updated...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 out of 3 new replicas have been updated...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 out of 3 new replicas have been updated...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 out of 3 new replicas have been updated...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 out of 3 new replicas have been updated...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 of 3 updated replicas are available...
Waiting for deployment "mynapp-deploy" rollout to finish: 2 of 3 updated replicas are available...
deployment "mynapp-deploy" successfully rolled out
```
롤아웃이 정상적으로 완료 되었다.


롤아웃 히스토리를 확인 해보자.
```
$ kubectl rollout history deployment mynapp-deploy

deployment.apps/mynapp-deploy
REVISION  CHANGE-CAUSE
1         kubectl create --filename=mynapp-deploy-v1.yml --record=true
2         kubectl create --filename=mynapp-deploy-v1.yml --record=true
```
kubectl set image 명령으로 업데이트 할 때 --record 옵션을 사용하지 않아 어떤 이유에서 변경됬는지 기록이 되지 않았다. 이는 나중에 추적을 어렵게 만들 수 있다.

> 참고  
> 리소스를 수정하는 방법  
> - kubectl edit: 편집기를 이용해 오브젝트 수정
> - kubectl patch: 리소스의 개별 속성 수정
> - kubectl replace: 리소스를 새로운 오브젝트로 변경
> - kubectl apply: 리소스 생성 및 변경
> - kubectl set image: 리소스의 이미지만 변경

#### (3) 버전2 -> 버전3
버전2에서 버전3으로 업데이트를 하자. 꼭 --record 옵션을 사용한다.
```
$ kubectl set image deployment mynapp-deploy mynapp=c1t1d0s7/myweb:v3 --record

deployment.apps/mynapp-deploy image updated
```

롤아웃 히스토리를 확인 해보자.
```
$ kubectl rollout history deployment mynapp-deploy

deployment.apps/mynapp-deploy
REVISION  CHANGE-CAUSE
1         kubectl create --filename=mynapp-deploy-v1.yml --record=true
2         kubectl create --filename=mynapp-deploy-v1.yml --record=true
3         kubectl set image deployment mynapp-deploy mynapp=c1t1d0s7/myweb:v3 --record=true
```
변경 사유가 기록된 것을 확인할 수 있다.

#### (4) 버전3 -> 버전2
버전3에서 버전2로 롤백 해보자.
```
$ kubectl rollout undo deployment mynapp-deploy --to-revision=2

deployment.apps/mynapp-deploy rolled back
```

롤아웃 히스토리를 확인해 보자.
```
$ kubectl rollout history deployment mynapp-deploy

deployment.apps/mynapp-deploy
REVISION  CHANGE-CAUSE
1         kubectl create --filename=mynapp-deploy-v1.yml --record=true
3         kubectl set image deployment mynapp-deploy mynapp=c1t1d0s7/myweb:v3 --record=true
4         kubectl create --filename=mynapp-deploy-v1.yml --record=true
```
리비전 2는 업어졌고, 리비전 2의 상태가 현재 리비전 4로 반영되었다.

#### (5) 버전2 --> 버전3
이번에는 새로 디플로이먼트 오브젝트를 생성해서 롤링업데이트를 수행해보자.
> mynapp-deploy-v3.yml 파일
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: mynapp-deploy
  labels:
    app: mynapp-deploy
  annotations:
    kubernetes.io/change-cause: My Node App Version 3
spec:
  strategy:
    type: RollingUpdate
    rollingupdate:
      maxUnavailable: 1
      maxSurge: 1
  minReadySeconds: 5
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-deploy
  template:
    metadata:
      labels:
        app: mynapp-deploy
    spec:
      containers:
      - image: c1t1d0s7/myweb:v3
        name: mynapp
        ports:
        - containerPort: 8080
        readinessProbe:
          periodSeconds: 1
          httpGet:
            path: /
            port: 8080
```
여기서 주요한 필드는 어노테이션이다. deployment.metadata.annotations.kubernetes.io/change-cause 필드에 디플로이먼트 리소스의 변경 사유에 대해 명령이 아닌 문자열로 기록할 수 있다.

kubectl apply로 변경된 리소스를 적용 해보자.
```
$ kubectl apply -f mynapp-deploy-v3.yml

deployment.apps/mynapp-deploy configured
```

롤아웃 상태를 확인해보자.
```
$ kubectl rollout status deployment mynapp-deploy

deployment "mynapp-deploy" successfully rolled out
```

롤아웃 히스토리를 확인해보자.
```
$ kubectl rollout history deployment mynapp-deploy

deployment.apps/mynapp-deploy
REVISION  CHANGE-CAUSE
1         kubectl create --filename=mynapp-deploy-v1.yml --record=true
3         kubectl set image deployment mynapp-deploy mynapp=c1t1d0s7/myweb:v3 --record=true
4         kubectl create --filename=mynapp-deploy-v1.yml --record=true
5         My Node App Version 3
```
변경 사유에 명령이 아닌 어노테이션의 문자열이 기록된 것을 알 수 있다.

#### (6) 롤백과 레플리카셋 컨트롤러
디플로이먼트는 이전 버전의 레플리카셋 컨트롤러를 삭제하지 않는다.

레플리카셋 목록을 확인해보자.
```
$ kubectl get replicaset -o wide

NAME                       DESIRED   CURRENT   READY   AGE     CONTAINERS   IMAGES              SELECTOR
mynapp-deploy-65c8fdc6cf   0         0         0       7m40s   mynapp       c1t1d0s7/myweb:v3   app=mynapp-deploy,pod-template-hash=65c8fdc6cf
mynapp-deploy-6b64fd6f5    3         3         3       5m15s   mynapp       c1t1d0s7/myweb:v3   app=mynapp-deploy,pod-template-hash=6b64fd6f5
mynapp-deploy-859bd5fbd    0         0         0       15m     mynapp       c1t1d0s7/myweb:v2   app=mynapp-deploy,pod-template-hash=859bd5fbd
mynapp-deploy-8c668bdcc    0         0         0       16m     mynapp       c1t1d0s7/myweb:v1   app=mynapp-deploy,pod-template-hash=8c668bdcc
```

이는 롤백하기 위한 롤아웃 히스토리 목록과 같으며, 롤백 시 다시 이전 레플리카셋을 그대로 사용하기 위함이다.

### 5) 리소스 삭제
```
$ kubectl delete deployments.apps mynapp-deploy

deployment.apps "mynapp-deploy" deleted
```

```
$ kubectl delete service mynapp-svc-deploy

service "mynapp-svc-deploy" deleted
```
