## 1. 레플리케이션 컨트롤러 롤링업데이트
쿠버네티스 시스템에서 파드가 실행하고 있는 애플리케이션을 업데이트 해보자.

애플리케이션 업데이트 시 가장 쉬운 형태는 애플리케이션이 동작하고 있는 파드 세트를 모두 삭제하고, 그런 다음 새로운 애플리케이션이 배포된 이미지를 이용하여 파드 세트를 생성하는 방법이다. 그러나 이런 방법은 새로운 파드 세트가 생성되고 준비될 때 까지 다운타임이 발생할 수밖에 없다.

쿠버네티스 시스템은 애플리케이션 업데이트 시 이전 파드 세트를 모두 삭제하고, 새로운 파드 세트를 생성하는 것이 아니라, 이전 파드와 새 파드를 점진적으로 교체 해나가는 전략을 사용한다.

그리고 수 많은 명령어를 이용해 수작업으로 하는 것이 아닌 자동으로 롤링 업데이트를 수행한다.

| RC v1 | RC v2 |
|:-----:|:-----:|
| A B C | X X X |
| A B X | A X X |
| A X X | A B X |
| X X X | X X X |

> 참고  
> 지금 진행하는 롤링업데이트는 레플리케이션 컨트롤러만 가능하다.(DEPRECATED)  
> 레플리카셋은 컨트롤러의 상위 수준인 디플로이먼트 리소스에서 롤링업데이트를 수행한다.  

### 1) 롤링업데이트를 위한 리소스 준비
롤링업데이트를 위한 레플리케이션 컨트롤러와 서비스를 준비하자.

> 참고  
> 롤링업데이트에 사용할 이미지는 index.js에 버전을 나타내는 텍스트만 추가한 간단한 이미지를 준비한다.  
> c1t1d0s7/myweb:v1  
> c1t1d0s7/myweb:v2  
> c1t1d0s7/myweb:v3  

#### (1) 롤링업데이터를 위한 컨트롤러 및 서비스 준비
롤링업데이트를 위한 버전1 이미지를 가지고 있는 레플리케이션 컨트롤러다. 이미지의 태그가 v1이다.
> mynapp-rc-v1.yml 파일
```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: mynapp-rc-v1
spec:
  replicas: 3
  template:
    metadata:
      labels:
        app: mynapp-rc
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb:v1
        ports:
        - containerPort: 8080
```

웹 애플리케이션을 확인하기 위한 로드 밸런서 서비스다.
> mynapp-svc-roll.yml 파일
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-roll
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: mynapp-rc
```

레플리케이션 컨트롤러와 서비스를 생성하자.
```
$ kubectl create -f mynapp-rc-v1.yml -f mynapp-svc-roll.yml

replicationcontroller/mynapp-rc-v1 created
service/mynapp-svc-roll created
```

#### (2) 모니터링 준비
생성한 레플리케이션 컨트롤러가 작동하고 있는지 확인하자.
```
$ kubectl get replicationcontrollers

NAME           DESIRED   CURRENT   READY   AGE
mynapp-rc-v1   3         3         3       99s
```

로드 밸런서 서비스의 외부 IP를 확인하자.
```
$ kubectl get services mynapp-svc-roll

NAME              TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)        AGE
mynapp-svc-roll   LoadBalancer   10.233.50.78   192.168.56.200   80:30911/TCP   117s
```

새로운 터미널을 하나 추가로 열어 컨트롤러 및 파드의 상태를 확인한다.
```
$ watch -n1 -d kubectl get all
```

새로운 터미널을 하나 추가로 열어 웹 요청 및 응답을 모니터링 한다.
```
$ watch -n1 -d curl http://192.168.56.200
```

### 2) 레플리케이션 컨트롤러 롤링업데이트
레플리케이션 컨트롤러의 롤링업데이트 명령은 다음 형식을 가지고 있다. (자동 완성 기능은 없음)
```
kubectl rolling-update <현재 컨트롤러> <새 컨트롤러> --image=<새 이미지>
kubectl rolling-update <현재 컨트롤러> <새 컨트롤러> -f <새 오브젝트 파일>
```

버전1에서 버전2로 업데이트 하자. 업데이트 도중 모니터링 터미널을 유심히 관찰한다.
```
$ kubectl rolling-update mynapp-rc-v1 mynapp-rc-v2 --image=c1t1d0s7/myweb:v2
```

파드의 라벨을 확인해보자.
```
$ kubectl get pods --show-labels

NAME                 READY   STATUS        RESTARTS   AGE     LABELS
mynapp-rc-v2-79f5g   1/1     Running       0          2m14s   app=mynapp-rc,deployment=130a7eac772721aab74e77cd0e13cbd4
mynapp-rc-v2-br2xn   1/1     Running       0          3m17s   app=mynapp-rc,deployment=130a7eac772721aab74e77cd0e13cbd4
mynapp-rc-v2-fpcwx   1/1     Running       0          68s     app=mynapp-rc,deployment=130a7eac772721aab74e77cd0e13cbd4
```
원래 이전 버전의 파드에는 app 레아블만 있었다. deployment 레이블이 롤링업데이트의해 자동으로 생성된 것을 확인할 수 있다.

컨트롤러도 확인해보자.
```
$ kubectl get replicationcontroller -o wide

NAME           DESIRED   CURRENT   READY   AGE     CONTAINERS   IMAGES              SELECTOR
mynapp-rc-v2   3         3         3       4m38s   mynapp       c1t1d0s7/myweb:v2   app=mynapp-rc,deployment=130a7eac772721aab74e77cd0e13cbd4
```
같은 deployment 레이블 셀렉터가 있는것을 확인할 수 있다. 이 레이블은 롤링업데이트를 수행할 때 리소스를 구분하기 위해 붙인 해시 값이다. 롤링업데이트는 컨트롤러와 파드를 deployment 레이블을 이용하여 연결하게 된다.

### 3) 레플리케이션 컨트롤러의 롤링업데이트를 더 이상 사용하지 않는 이유
kubectl rolling-update 명령을 더 이상 사용하지 않는 이유는 다음과 같다.

#### (1) 리소스의 임의 변경
비록 deployment 라벨을 추가하는 것이지만 정의한 리소스를 쿠버네티스 클러스터가 임의로 변경(레이블 추가)하기 때문이다.

#### (2) 클라이언트가 업데이트를 수행
쿠버네티스의 클라이언트가 롤링업데이트를 수행하고 있다. kubectl 명령은 쿠버네티스 클라이언트이다. 즉, 여러가지 이유로 쿠버네티스 클라이언트가 쿠버네티스 클러스터 서버와 연결이 끊어지면 롤링업데이트가 중단된다. 클라이언트가 롤링업데이트의 단계적 요청을 할 수 없기 때문이다.

그러나 레플리케이션 컨트롤러를 아직 사용하고 있거나, 꼭 레플리케이션 컨트롤러를 사용해야겠다면 이 방법이 유일하다.

### 4) 리소스 삭제
```
$ kubectl delete replicationcontrollers mynapp-rc-v2

replicationcontroller "mynapp-rc-v2" deleted
```

```
$ kubectl delete services mynapp-svc-roll

service "mynapp-svc-roll" deleted
```
