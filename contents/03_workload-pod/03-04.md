## 4. 네임스페이스

### 1) 네임스페이스 소개
모든 오브젝트에 레이블을 부여할 수 있다. 그러나 이는 레이블 셀렉터를 이용해서 식별 및 검색을 할 수 있을 뿐, 논리적으로 분리되어 있지 않다. 즉, 많은 오브젝트를 관리할 때 논리적으로 오브젝트의 분리가 필요하다.

특히 멀티테넌트나 사용자 및 관리자가 여려명인 환경의 경우 오브젝트와 리소스를 완전히 분리시킬 필요가 있다. 또는 팀별이나 개발, QA, 프로덕션 등 필요한 용도 별로 오브젝트를 분리 시킬 필요가 있다.

네임스페이스는 오브젝트를 논리적으로 분리할 수 있는 논리적 파티션이다. 

> 참고  
> 도커 컨테이너의 핵심 요소인 리눅스 네임스페이스와는 관련 없다.

### 2) 네임스페이스 확인
```
$ kubectl get namespaces

NAME              STATUS   AGE
default           Active   125m
ingress-nginx     Active   123m
kube-node-lease   Active   125m
kube-public       Active   125m
kube-system       Active   125m
metallb-system    Active   123m
rook-ceph         Active   63m
```

지금까지 우리가 생성한 오브젝트는 모두 기본 네임스페이스인 default 네임스페이스를 사용했다. 즉, 오브젝트 생성 시 특정 네임스페이스를 지정하 지 않으면 기본 네임스페이스를 사용한다.

네임스페이스의 목록은 쿠버네티스 클러스터의 배포 방법/도구나 쿠버네티스 버전에 따라 기본 제공되는 네임스페이스 목록이 조금씩은 다를 수 있다.

기본적인 네임스페이스의 용도는 다음과 같다.
- default
  * 기본 네임스페이스
- kube-node-lease
  * 쿠버네티스 노드의 가용성을 체크하기 위한 네임스페이스
  * 해당 네임스페이스에는 하트비트를 위한 리스(Leases) 오브젝트가 있음
  * 쿠버네티스 1.14 이상
- kube-public
  * 모든 사용자(인증 받지 않은 사용자 포함)가 읽기 권한으로 접근할 수 있음
  * 관례적으로 만들어져 있지만 아무 리소스도 없고, 꼭 사용해야하는 것은 아님
- kube-system
  * 쿠버네티스 클러스터의 핵심 리소스 배치

추가적인 네임스페이스:
- ingress-nginx
  * 쿠버네티스 인그레스(Ingress) 리소스를 위한 Nginx 인그레스 컨트롤러 배치
- rook-ceph
  * Rook Ceph 스토리지 관린 리소스 배치

### 4) 네임스페이스 생성
YAML 파일을 생성하지 않고도 간단히 명령을 이용하여 네임스페이스를 생성할 수 있다.
```
$ kubectl create namespace development

namespace/development created
```

네임스페이스를 확인해보자.
```
$ kubectl get namespaces

NAME              STATUS   AGE
default           Active   134m
development       Active   6s
ingress-nginx     Active   132m
kube-node-lease   Active   134m
kube-public       Active   134m
kube-system       Active   134m
metallb-system    Active   132m
rook-ceph         Active   72m
```

YAML 파일을 이용하여 네임스페이스를 생성할 수 있다. 다음은 네임스페이스를 구성하기 위한 YAML 파일이다.

> qa-ns.yaml
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: quality-assurance
```

YAML 파일을 선언해 quality-assurance 네임스페이스를 만든다.
```
$ kubectl create -f qa-ns.yaml

namespace/quality-assurance created
```

네임스페이스가 생성되었는지 확인한다.
```
$ kubectl get namespaces

NAME                STATUS   AGE
default             Active   137m
development         Active   3m31s
ingress-nginx       Active   135m
kube-node-lease     Active   137m
kube-public         Active   137m
kube-system         Active   137m
metallb-system      Active   135m
quality-assurance   Active   10s
rook-ceph           Active   76m
```

### 5) 특정 네임스페이스에서 파드(및 오브젝트) 생성
오브젝트 생성 시 특정 네임스페이스를 지정하면 해당 네임스페이스에 오브젝트를 생성할 수 있다.

```
$ kubectl create -f myapp-pod.yaml -n development

pod/myapp-pod created
```

development 네임스페이스를 확인해보자.
```
$ kubectl get pods -n development

NAME        READY   STATUS    RESTARTS   AGE
myapp-pod   1/1     Running   0          11s
```

YAML 파일의 pods.metadata.namespaces 필드에 직접 네임스페이스도 선언 가능하다.
> myapp-pod-ns.yaml
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  namespace: development
spec:
  containers:
  - image: ghcr.io/c1t1d0s7/go-myweb
    name: myapp
    ports:
    - containerPort: 8080
      protocol: TCP
```

### 6) 리소스 삭제
리소스의 삭제는 크게 세 가지 방법으로 가능하다.
- 오브젝트 이름으로 삭제
- 오브젝트 정의 파일(YAML 및 JSON)로 삭제
- 오브젝트 레이블로 삭제

#### (1) 오브젝트 이름으로 파드 삭제
```
$ kubectl delete pod myapp-pod

pod "myapp-pod" deleted
```
네임스페이스를 지정하지 않았기 때문에 default 네임스페이스의 myapp-pod 파드가 삭제 된다.

#### (2) YAML 파일로 파드 삭제
```
$ kubectl delete -f myapp-pod-ns.yaml

pod "myapp-pod" deleted
```
네임스페이스는 YAML 파일 내에 지정되어 있고 development 네임스페이스의 myapp-pod 파드가 삭제 된다.

#### (3) 오브젝트 레이블을 이용한 오브젝트 삭제
```
$ kubectl delete pods -l tier=frontend

pod "myapp-pod-label" deleted
```
default 네임스페이스에서 tier=frontend 레이블을 가지고 있는 모든 오브젝트를 삭제한다.

#### (4) 네임스페이스 삭제
명령을 이용하여 development 네임스페이스 삭제한다.
```
$ kubectl delete namespaces development

namespace "development" deleted
```

정의된 YAML을 이용하여 quality-assurance 네임스페이스를 삭제한다.
```
$ kubectl delete -f qa-ns.yaml

namespace "quality-assurance" deleted
```

#### (5) 삭제된 네임스페이스 확인
development 및 quality-assurance 네임스페이스 적절하게 삭제되었는지 확인한다.
```
$ kubectl get namespaces

NAME              STATUS   AGE

kube-node-lease   Active   141m
kube-public       Active   141m

```
