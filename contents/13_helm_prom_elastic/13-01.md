## 1. Helm 패키지 관리자

지금까지 리소스를 생성하기 위해 간단한 명령으로도 가능했지만, 수많은 YAML 오브젝트 파일을 생성하고 수정하고 관리하였다. 웹 애플리케이션 하나를 구성하기 위해 컨트롤러, 서비스, 인그레스, PVC, 서비스 계정, 롤, 롤 바인딩, 컨피그맵, 시크릿 등 여러 YAML 오브젝트 파일이 필요하다고 가정하면, 일일이 구성하고 관리하는 것이 쉽지 않았을 것이다.

Helm은 이런 관리의 복잡함을 Helm 차트라는 구성으로 여러 YAML 오브젝트 리소스를 하나의 패키지로 구성, 배포 및 관리를 쉽게 할 수 있다. 리눅스 시스템의 APT 및 YUM 패키지 관리자와 비슷하다.

Helm Classic 프로젝트는 2015년 KubeCon에 소개되면서 알려지게 되었고, 2016년 1월 쿠버네티스 프로젝트와 통합되었으며, 2016년 말 Helm v2가 릴리즈되었으며 Tiller라는 쿠버네티스 클러스터에 설치되는 서버측 구성요소를 두었다. 2018년 6월 쿠버네티스 하위 프로젝트에서 CNCF 정식 프로젝트로 승격되었다. 2019년 11월 Helm v3을 출시하였고, 그동안 문제가 되었던 Tiller(복잡하고 과도한 권한이 부여되어야 되는 문제) 구성요소를 제거하여, 원래의 목적인 클라이언트 도구로 탈바꿈하였다.

> 참고: Helm v2 vs. Helm v3
> Helm v2와 Helm v3는 명령어 사용법이 다르다.  
> Helm v2는 Tiller라는 서버측 구성요소가 있어, 클라언트인 Helm 명령과 gRPC로 통신하며, Tiller가 쿠버네티스 API 서버와 통신하는 구조였다.  
> Helm v3는 별도의 서버측 구성요소가 없고, 클라이언트인 Helm 명령과 쿠버네티스 API 서버가 통신한다.

### 1) Helm 설치
Helm은 kubectl 명령과 적절한 kubeconfig 파일이 있는 호스트에 설치해야 한다.

#### (1) 바이너리
각 운영체제의 Helm 바이너리는 https://github.com/helm/helm/releases 에서 수동으로 다운로드 받을 수 있다.

현재 최신버전인 v3.X.X를 다운 받는다. 아직 v2 버전도 릴리즈되고 있으니 잘 확인하자.
```
$ wget https://get.helm.sh/helm-v3.1.2-linux-amd64.tar.gz
```

압축을 푼다.
```
$ tar xf helm-v3.1.2-linux-amd64.tar.gz
```

파일을 바로 실행할 수 있도록 바이너리 디렉토리로 복사하자.
```
$ sudo sudo cp linux-amd64/helm /usr/local/bin/helm
```

실행되는지 확인해보자.
```
$ helm version

version.BuildInfo{Version:"v3.1.2", GitCommit:"d878d4d45863e42fd5cff6743294a11d28a9abce", GitTreeState:"clean", GoVersion:"go1.13.8"}
```

#### (2) macOS
macOS에서는 Homebrew 패키지 관리자로 간단히 설치할 수 있다.
```
$ brew install helm
```

#### (3) Windows
윈도우에서는 Chocolatey 패키지 관리자로 간단히 설치할 수 있다.
```
> choco install kubernetes-helm
```

#### (4) Ubuntu
우분투 리눅스는 Snap 패키지 관리자로 간단히 설치할 수 있다.
```
$ sudo snap install helm --classic
```

### 2) Helm 기본 개념

#### (1) Helm 주요 용어
- 차트(Chart): Helm 패키지
- 저장소(Repository): Helm 차트가 저장된 저장소
- 릴리즈(Release): 차트가 클러스터에 배포된 인스턴스 단위

#### (2) Helm 차트 구조
Helm 차트는 쿠버네티스 리소스를 모아놓은 패키지다. TAR Gzip으로 아카이브 및 압축되어 있으며, 아카이브 내에는 표준화된 구조를 가지고 있다.

Helm 차트는 공개되어 있는 저장소에서 얻을수도 있고, 직접 차트를 개발하거나, 사내에 사설 저장소를 운영할수도 있다. 공개되어있는 Helm 저장소는 Helm Hub(https://hub.helm.sh/) 저장소와, 구글에서 제공하는 저장소 등 여러 저장소가 있다.

다음은 Helm 차트의 필수적인 구조만 나타내었다.
```
mychart/
  Chart.yaml
  values.yaml
  templates/
  ...
```
- mychart 디렉토리는 차트 패키지의 최상위 디렉토리로 패키지의 이름을 나타낸다.
- Chart.yaml 파일은 차트에 대한 설명을 포함하고 있다. Helm 차트의 버전, 설명, 작성자 등의 정보를 정의한다.
- values.yaml 파일은 변수와 변수의 기본값을 정의하는 파일로, templates/ 디렉토리의 YAML 오브젝트 파일에 해당 값을 제공하며, 사용자가 값을 변경해 사용자화 할 수 있다.
- templates/ 디렉토리는 템플릿 파일을 위치하는 디렉토리로, 실제 API 서버에 제공할 YAML 오브젝트 파일을 배치한다.

templates 디렉토리의 YAML 오브젝트 파일에는 사용자화 할 수 있도록, 변수를 참조 하도록 구성하고, 변수와 값은 values.yaml 파일에 정의해 사용자화를 쉽게할 수 있도록 구성한다.

### 3) Helm 사용

#### (1) 사전 요구사항
앞서 설명했지만 Helm은 쿠버네티스 클러스터의 클라이언트로 접속가능한 쿠버네티스 클러스터가 준비되어 있어야 한다. 또한 kubectl 명령과 API 서버에 적절히 접근하기 위한 kubeconfig 파일이 있어야 Helm도 쿠버네티스 클러스터에 접근할 수 있다.

- 접근 가능한 쿠버네티스 클러스터
- helm이 설치될 시스템에 kubectl이 설치되어 있어야 한다.
- kubectl은 적절한 kubeconfig(~/.kube/config)파일로 쿠버네티스 클러스터에 접근할 수 있어야 한다.

#### (2) Helm 차트 검색 및 저장소
기본적으로 Helm 허브 저장소에 접근할 수 있으며, Helm 허브외의 저장소는 별도로 저장소를 추가해야 사용할 수 있다.

- ```helm search hub```: Helm 허브에서 차트를 검색한다.
- ```helm search repo```: ```helm repo add``` 명령으로 추가한 저장소에서 차트를 검색한다.

> <참고>  
> Helm v3는 처음부터 기본 저장소가 없다.  
> Helm v2는 처음부터 stable이란 이름의 기본 저장소가 있다.

##### Helm 허브 저장소 검색
다음은 Helm 허브에서 mysql 차트를 검색하는 명령이다.
```
$ helm search hub mysql

URL                                        CHART VERSION	APP VERSION	DESCRIPTION
...
https://hub.helm.sh/charts/stable/mysql    1.6.2        	5.7.28     	Fast, reliable...
...
```

##### Helm 차트 저장소 추가
다음은 Helm 허브 이외에 차트 저장소를 추가하는 방법이다.
- ```helm repo add [NAME] <URL>```
  * NAME: 저장소 이름
  * URL: 저장소 주소 

현재 저장소의 목록을 출력해보자.
```
$ helm repo list

Error: no repositories to show
```
아무 저장소도 없다고 보고하고 있다.

helm repo add 명령을 사용해 구글에서 제공하는 저장소를 추가해보자.
```
$ helm repo add stable https://kubernetes-charts.storage.googleapis.com/

"stable" has been added to your repositories
```
저장소 이름은 원하는 이름으로 지정할 수 있다. 여기서는 stable 이란 이름으로 추가했다.

> 참고  
> Helm 차트 안정(Stable)버전 저장소: https://kubernetes-charts.storage.googleapis.com/

```
$ helm repo list

NAME  	URL
stable	https://kubernetes-charts.storage.googleapis.com/
```

Helm 허브가 아닌 추가된 저장소를 helm search repo 명령으로 mysql 차트를 검색해보자.

##### Helm 차트 저장소 최신 목록 업데이트
차트 저장소의 차트 목록을 업데이트 하려면 helm repo update 명령을 내린다.
```
$ helm repo update

Hang tight while we grab the latest from your chart repositories...
...Successfully got an update from the "stable" chart repository
Update Complete. ⎈ Happy Helming!⎈
```

##### Helm 차트 저장소 삭제
Helm 차트 저장소를 삭제하기 위해서는 helm repo remove 명령을 사용한다. 여기서 저장소를 삭제하지 말자.
```
$ helm repo remove stable
```

##### Helm 차트 저장소의 전체 차트 목록 확인
Helm Hub 전체 차트 목록 확인하는 방법은 검색할 차트 이름을 지정하지 않는 것이다.
```
$ helm search hub
```

Helm 저장소 전체 차트 목록 확인역시 같다.
```
$ helm search repo
```

Helm 추가 저장소는 여러개가 있을수 있으며, stable 저장소에서만 전체 차트 목록 확인하는 방법은 다음과 같다.
```
$ helm search repo stable
```

#### (3) Helm 차트 정보 확인
Helm 차트의 정보를 확인하기 위해서는 helm show 명령을 사용한다.
- ```helm show chart [CHART]```: 차트 정보(버전, 홈페이지, 키워드, 소스 주소 등)
- ```helm show readme [CHART]```: 차트 설명(사용법, 파라미터 등)
- ```helm show values [CHART]```: 차트 설정 파라미터
- ```helm show all [CHART]```: Chart, Readme, Values

Helm 차트를 다운로드 받지 않아도 확인할 수 있다.

helm show chart 명령으로 차트의 정보를 확인할 수 있다.
```
$ helm show chart stable/mysql

  system.

- mysql

- email: o.with@sportradar.com
```

#### (4) Helm 차트 설치 및 관리

##### Helm 차트 설치
Helm 차트 설치는 helm install 명령으로 설치할 수 있다.
- ```helm install [NAME] [CHART] [FLAGS]```
  * NAME: 릴리즈 이름
  * CHART: 차트 이름
  * FLAGS: 옵션

mydb라는 릴리즈 이름으로 stable 저장소의 mysql 차트를 설치해보자.
```
$ helm install mydb stable/mysql

NAME: mydb
LAST DEPLOYED: Fri Mar  6 14:53:40 2020
NAMESPACE: default
STATUS: deployed
REVISION: 1
NOTES:
MySQL can be accessed via port 3306 on the following DNS name from within your cluster:
mydb-mysql.default.svc.cluster.local
...
```

#### (4) Helm 차트 설치전 사용자화
차트 구조에서 확인했던 values.yaml을 수정해 사용자화를 해보자.

##### Helm 차트 기본 설치
다시 mydb 릴리즈 이름으로 stable 저장소의 mysql을 설치해보자.
```
$ helm install mydb stable/mysql

NAME: mydb
LAST DEPLOYED: Fri Mar 27 12:26:59 2020
NAMESPACE: default
STATUS: deployed
REVISION: 1
NOTES:
MySQL can be accessed via port 3306 on the following DNS name from within your cluster:
mydb-mysql.default.svc.cluster.local
...
```

##### 배포된 데이터베이스 root 패스워드 확인
차트 설치 명령을 실행한 후 나오는 출력에 MySQL 데이터베이스 root 사용자의 패스워드를 확인하는 방법에 대해 확인할 수 있다.

아래 시크릿 정보를 BASE64로 디코딩하여 패스워드를 확인할 수 있다.
```
$ kubectl get secret --namespace default mydb-mysql -o jsonpath="{.data.mysql-root-password}" | base64 --decode; echo

pzHn4ZEtuv
```

mydb 릴리즈를 삭제하자.
```
$ helm uninstall mydb
```

##### Helm 차트의 구성 가능한 옵션 확인
Helm 차트에서 사용자화 가능한 구성 옵션을 확인해보자.
```
$ helm show values stable/mysql

## mysql image version
## ref: https://hub.docker.com/r/library/mysql/tags/
##
image: "mysql"
imageTag: "5.7.28"

strategy:
  type: Recreate

busybox:
  image: "busybox"
  tag: "1.29.3"
...
## Specify password for root user
##
## Default: random 10 character string
# mysqlRootPassword: testing
...
```
여러 변수들이 정의되어 있지만 "mysqlRootPassword" 변수를 확인할 수 있다. 이 변수에 값을 지정해 root 패스워드가 원하는 패스워드로 사용자화 되는지 확인해보자.

##### Helm 차트의 root 패스워드 구성 수정 및 배포
사용자화 하는 방법은 values.yaml 파일을 직접 작성하는 방법이 있다. values.yaml 파일의 필요한 부분만 수정해 전체를 다 제공하는 방법이 있고, 필요한 변수만 별도의 파일에 선언해서 제공하는 방법이 있다.

또는 변수 몇 개만 변경할것이라면, 차트 설치시 해당 변수/값 쌍만 제공하는 방법이 있다.

- ```--values / -f```: 덮어쓸 YAML 파일 지정
- ```--set```: 덮어쓸 값 지정 (name=value)

간단하게 root 패스워드를 변경하기 위한 custom.yml 파일을 생성해보자.
```
$ echo "mysqlRootPassword: P@ssw0rd" > custom.yml
```

##### 사용자화를 위한 Helm 차트 설치
사용자화를 위한 custom.yml 파일을 제공해 mydb-custom 릴리즈를 생성해보자.
```
$ helm install mydb-custom stable/mysql --values custom.yml
```

##### root 패스워드 확인
시크릿에 저장된 패스워드를 확인해보자.
```
$ kubectl get secret --namespace default mydb-custom-mysql -o jsonpath="{.data.mysql-root-password}" | base64 --decode; echo

P@ssw0rd
```
정상적으로 변경한 패스워드를 확인할 수 있다.

#### (5) Helm 릴리즈 업그레이드 및 롤백

##### mydb-custom 서비스 포트 확인
mydb-custom 릴리즈의 서비스 포트를 확인해보자.
```
$ kubectl get svc

NAME                TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
mydb-custom-mysql   ClusterIP   10.96.171.124   <none>        3306/TCP   2m30s
```
MySQL의 기본 서비스 포트인 3306 포트로 지정되어 있다.

##### Helm 릴리즈 업그레이드 (서비스 포트 변경)
Helm의 장점중 하나는 릴리즈를 업그레이드할 수 있다는 것이다.

릴리즈의 업그레이드 명령은 다음과 같다.
- ```helm upgrade [NAME] [CHART]```
  * NAME: 릴리즈 이름
  * CHART: 차트 이름

이번에는 간단하게 변경할 변수를 지정해 사용자화 및 업그레이드 해보자.
```
$ helm upgrade mydb-custom stable/mysql --set service.port=13306

Release "mydb-custom" has been upgraded. Happy Helming!
NAME: mydb-custom
LAST DEPLOYED: Fri Mar 27 12:41:33 2020
NAMESPACE: default
STATUS: deployed
REVISION: 2
...
```
service.port 변수의 값을 13306 포트로 변경한다는 의미다. 원래 기본 값은 당연하게도 3306 포트로 되어있다.

참고로 리비전(REVISION) 번호가 1에서 2로 변경된 것을 확인할 수 있다.

##### Helm 릴리즈 업그레이드 확인
릴리즈 목록을 확인해보자.
```
$ helm list

NAME       	NAMESPACE	REVISION	UPDATED                            	STATUS  	CHART      	APP VERSION
mydb-custom	default  	2       	2020-03-09 19:56:29.12312 +0900 KST	deployed	mysql-1.6.2	5.7.28
```
배포가 완료되었고 리비전이 2 버전인 것을 확인할 수 있다.

실제 서비스 리소스 목록을 확인해보자.
```
$ kubectl get svc

NAME                TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)     AGE
mydb-custom-mysql   ClusterIP   10.96.171.124   <none>        13306/TCP   9m14s
```
포트가 13306 포트로 변경된것을 확인할 수 있다.

##### Helm 릴리즈 롤백
릴리즈의 롤백은 다음 명령으로 할 수 있다.
- ```helm rollback [NAME] [REVISION]```

mydb-custom 릴리즈의 리비전 1로 롤백해보자.
```
$ helm rollback mydb-custom 1
```

릴리즈 목록을 확인해 보자.
```
$ helm list

NAME       	NAMESPACE	REVISION	UPDATED         	STATUS  	CHART      	APP VERSION
mydb-custom	default  	3       	2020-03-09 19:58	deployed	mysql-1.6.2	5.7.28
```
배포가 완료되었고 롤백되어 리비전 3으로 변경되었다.

실제 서비스 리소스를 확인해보자.
```
$ kubectl get svc

NAME                TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)    AGE
mydb-custom-mysql   ClusterIP   10.96.171.124   <none>        3306/TCP   13m
```
포트가 원래대로 3306 포트로 변경되었다.

##### Helm 릴리즈 히스토리 확인
릴리즈 히스토리 목록은 다음 명령으로 확인할 수 있다.
- ```helm history [NAME]```

mydb-custom 릴리즈의 히스토리를 확인해보자.
```
$ helm history mydb-custom

REVISION	UPDATED                 	STATUS    	CHART      	APP VERSION	DESCRIPTION
1       	Mon Mar  9 19:48:03 2020	superseded	mysql-1.6.2	5.7.28     	Install complete
2       	Mon Mar  9 19:53:20 2020	superseded	mysql-1.6.2	5.7.28     	Upgrade complete
3       	Mon Mar  9 19:58:38 2020	deployed  	mysql-1.6.2	5.7.28     	Rollback to 1
```
최초 설치, 업그레이드, 롤백 순서대로 확인할 수 있다.

### 4) Helm 릴리즈 삭제
생성했던 릴리즈를 삭제하자.
```
$ helm uninstall mydb-custom

release "mydb-custom" uninstalled
```
