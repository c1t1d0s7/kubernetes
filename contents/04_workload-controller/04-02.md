## 2. 레플리케이션 컨트롤러
쿠버네티스의 컨트롤러는 파드를 올바르게 동작하기 위해 특정 상태를 보장 하기 위한 기능이다. 특정 상태는 컨트롤러에 따라 동작하는 방식 및 정의하는 상태는 조금씩 다르다.

### 1) 레플리케이션 컨트롤러 소개
레플리케이션 컨트롤러(ReplicationController)는 파드가 특정 개수만큼이 복제되고 동작하는 것을 보장한다.

노드의 문제가 발생했거나 파드가 문제가 생겨 원하는 수의 파드가 동작하지 않는다면 자동으로 스케쥴러에 의해 새로운 노드나 기존 노드에 다시 새로운 파드를 생성해 원하는 수의 파드를 복제해서 동작시킨다.

심지어 일반적인 형태에서 발생할 일은 많이 없지만, 파드가 원하는 수보다 많은 경우에도, 원하는 파드 수로 맞춘다. 즉, 불필요한 파드를 제거해 원하는 수를 맞춘다.

원하는 수의 복제본 보다 더 많은 복제본이 많아 질 수 있는 경우:
- 수동으로 동일한 형식을 파드를 생성
- 기존의 파드의 유형을 변경
- 원하는 수의 파드 복제본 수를 줄인 경우

즉, 실행중인 파드의 목록을 주기적으로 모니터링 하고, 원하는 수의 파드 수와 항상 일치시킨다.

레플리케이션 컨트롤러를 구성하기 위해서는 세 가지 요소가 있다.
- 파드를 지정하는 레이블셀렉터
- 새로운 파드의 복제본을 만들기 위한 파드 템플릿
- 복제본 수

레플리케이션 컨트롤러가 제공하는 기능은 다음과 같다.
- 원하는 복제본 수의 파드가 없는 경우 파드 템플릿을 이용하여 파드를 생성한다.
- 노드에 장애가 발생하면 장애가 발생한 노드에서 실행 중이던 파드를 다른 노드에 복제본을 생성한다.
- 수동이나 자동으로 파드를 수평 스케일링 할 수 있다.

### 2) 레플리케이션 컨트롤러 생성
레플리케이션 컨트롤러를 앞서 살펴봤듯이 세 가지 요소가 필요하다.

> mynapp-rc.yml
```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: mynapp-rc
spec:
  replicas: 3
  selector:
    app: mynapp-rc
  template:
    metadata:
      labels:
        app: mynapp-rc
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        ports:
        - containerPort: 8080
```
복제본 수는 세 개로 구성하고, 레플리케이션이 관리할 파드의 레이블 셀렉터는 app=mynapp-rc 레이블이다. 또한 생성할 파드는 템플릿에 정의하였고, 당연히 레이블은 app=mynapp-rc로 구성하였다.

레플리케이션 컨트롤러를 생성해 보자.
```
$ kubectl create -f mynapp-rc.yml

replicationcontroller/mynapp-rc created
```

### 3) 레플리케이션 컨트롤러 확인
레플리케이션 컨트롤러의 목록 및 상태를 확인해보자.

```
$ kubectl get replicationcontrollers

NAME        DESIRED   CURRENT   READY   AGE
mynapp-rc   3         3         0       4s
```
원하는(DESIRED) 복제본의 개수는 3개였고, 현재(CURRENT) 존재하는 파드는 3개, 준비(READY)된 파드는 없다는 의미이다. 이는 당연하게도 파드를 생성 중이기 때문이다.

파드의 목록을 확인해보자.
```
$ kubectl get po

NAME              READY   STATUS              RESTARTS   AGE
mynapp-rc-gj9qc   0/1     ContainerCreating   0          8s
mynapp-rc-kxqhn   0/1     ContainerCreating   0          8s
mynapp-rc-sndzk   0/1     ContainerCreating   0          8s
```
파드의 컨테이너를 생성하고 있는 중이다. 성능에 따라서 이미 만들어져 있을 수도 있다.

레플리케이션 컨트롤러가 처음 만들어졌을 때는 app=mynapp-rc 레이블을 가지고 있는 파드가 하나도 없기 때문에, 복제본 3개를 생성하게 된다.

파드에 구성된 레이블을 같이 살펴보자.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE    LABELS
mynapp-rc-gj9qc   1/1     Running   0          2m6s   app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running   0          2m6s   app=mynapp-rc
mynapp-rc-sndzk   1/1     Running   0          2m6s   app=mynapp-rc
```
조금 시간이 지난 뒤 다시 확인해 보면 원하는 복제본 세 개가 정상적으로 동작하는 것을 확인할 수 있다.

파드가 모두 준비된 것을 확인 할 수 있고, 레플리케이션 컨트롤러의 상태도 확인 해보자.
```
$ kubectl get replicationcontrollers

NAME        DESIRED   CURRENT   READY   AGE
mynapp-rc   3         3         3       3m44s
```
모든 파드가 요구 복제본 개수만큼 만들어져 있다.

레플리케이션 컨트롤러의 자세한 정보를 확인해보자. kubectl describe 명령으로 확인할 수 있다.
```
$ kubectl describe replicationcontrollers mynapp-rc

Namespace:    default

Labels:       app=mynapp-rc
Annotations:  <none>

Pods Status:  3 Running / 0 Waiting / 0 Succeeded / 0 Failed
Pod Template:
  Labels:  app=mynapp-rc
  Containers:
   mynapp:
    Image:        c1t1d0s7/myweb
    Port:         8080/TCP
    Host Port:    0/TCP
    Environment:  <none>
    Mounts:       <none>
  Volumes:        <none>
Events:
  Type    Reason            Age    From                    Message
  ----    ------            ----   ----                    -------
  Normal  SuccessfulCreate  4m11s  replication-controller  Created pod: mynapp-rc-sndzk
  Normal  SuccessfulCreate  4m11s  replication-controller  Created pod: mynapp-rc-gj9qc
  Normal  SuccessfulCreate  4m11s  replication-controller  Created pod: mynapp-rc-kxqhn
```

### 4) 레플리케이션 컨트롤러와 파드
레플리케이션 컨트롤러에서 노드의 장애 발생을 가정하고 파드를 삭제 했을때 어떻게 되는지 확인해보자.

```
$ kubectl get pods

NAME              READY   STATUS    RESTARTS   AGE
mynapp-rc-gj9qc   1/1     Running   0          5m3s
mynapp-rc-kxqhn   1/1     Running   0          5m3s
mynapp-rc-sndzk   1/1     Running   0          5m3s
```

마지막 파드(mynapp-rc-sndzk)를 삭제해 본다.
```
$ kubectl delete pods mynapp-rc-sndzk

pod "mynapp-rc-sndzk" deleted
```

파드가 삭제되고 있는 도중 다른 터미널에서 파드의 목록을 확인해 본다.
```
$ kubectl get pods

NAME              READY   STATUS        RESTARTS   AGE
mynapp-rc-gj9qc   1/1     Running       0          5m40s
mynapp-rc-kxqhn   1/1     Running       0          5m40s
mynapp-rc-mmz9p   1/1     Running       0          16s
mynapp-rc-sndzk   1/1     Terminating   0          5m40s
```
mynapp-rc-sndzk 파드는 삭제되고 있는 중이며(또는 이미 삭제 되었을 수 있음), mynapp-rc-mmz9p 새로운 파드가 생성되었다. 레플리케이션 컨트롤러는 파드의 목록을 확인하고 있다가, 하나의 파드가 없어진 것을 확인하고, 파드의 복제본 수와 맞추기 위해 하나를 더 생성한 것이다.

레플리케이션 컨트롤러의 자세한 정보를 확인해보면 정확하게 어떤 파드가 생성되었는지 확인 할 수 있다.
```
$ kubectl describe replicationcontrollers mynapp-rc
...
Name:         mynapp-rc
...
Events:
  Type    Reason            Age    From                    Message
  ----    ------            ----   ----                    -------
  Normal  SuccessfulCreate  7m25s  replication-controller  Created pod: mynapp-rc-sndzk
  Normal  SuccessfulCreate  7m25s  replication-controller  Created pod: mynapp-rc-gj9qc
  Normal  SuccessfulCreate  7m25s  replication-controller  Created pod: mynapp-rc-kxqhn
  Normal  SuccessfulCreate  2m1s   replication-controller  Created pod: mynapp-rc-mmz9p
```

> 참고  
> 파드를 삭제하기 전 watch 명령이나 --watch 옵션을 사용하여 실시간으로 확인할 수도 있다.

### 5) 레플리케이션 컨트롤러와 레이블
레플리케이션 컨트롤러는 관리할 파드를 파드의 레이블과 레플리케이션 컨트롤러의 레이블 셀렉터를 이용하여, 관리할 파드를 지정하고 관리하게 된다.

#### (1) 기존 파드에 레이블 추가
그러면 기존의 파드의 레이블을 추가하면 어떻게 될지 살펴보자. 우선 파드의 목록 및 레이블을 확인한다.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE     LABELS
mynapp-rc-gj9qc   1/1     Running   0          9m14s   app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running   0          9m14s   app=mynapp-rc
mynapp-rc-mmz9p   1/1     Running   0          3m50s   app=mynapp-rc
```

mynapp-rc-mmz9p 파드에 env=dev 레이블을 추가해 보자.
```
$ kubectl label pods mynapp-rc-mmz9p env=dev

pod/mynapp-rc-mmz9p labeled
```

다시 파드 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE     LABELS
mynapp-rc-gj9qc   1/1     Running   0          10m     app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running   0          10m     app=mynapp-rc
mynapp-rc-mmz9p   1/1     Running   0          4m48s   app=mynapp-rc,env=dev
```
결과는 기존 파드에 추가 레이블만 구성되었고 변경된 사항은 없다.

#### (2) 기존 파드의 레이블 변경
그러면 기존 파드의 레이블을 변경해 보자, 즉 app=mynapp을 다른 레이블로 변경하면 어떤 결과가 발생하는지 살펴보자.

mynapp-rc-mmz9p 파드의 app 레이블의 값을 test로 변경 해보자.
```
$ kubectl label pods mynapp-rc-mmz9p app=test --overwrite

pod/mynapp-rc-mmz9p labeled
```

파드의 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE   LABELS
mynapp-rc-6hvbc   1/1     Running   0          24s   app=mynapp-rc
mynapp-rc-gj9qc   1/1     Running   0          19m   app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running   0          19m   app=mynapp-rc
mynapp-rc-mmz9p   1/1     Running   0          13m   app=test,env=dev
```

mynapp-rc-mmz9p 파드의 레이블을 app=test로 변경한 이후, 레플리케이션 컨트롤러는 app=mynapp-rc 레이블을 가진 파드가 두 개밖에 없기 때문에, 새로운 mynapp-rc-6hvbc 파드를 생성하였다. 

이제부터 mynapp-rc-mmz9p 파드는 더이상 mynapp-rc 레플리케이션 컨트롤러가 관리하는 파드가 아니다.

다시 원래대로 mynapp-rc-mmz9p 파드의 레이블을 app=mynapp-rc로 변경해 보자.
```
$ kubectl label pods mynapp-rc-mmz9p app=mynapp-rc --overwrite

pod/mynapp-rc-mmz9p labeled
```


파드의 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS        RESTARTS   AGE     LABELS
mynapp-rc-6hvbc   1/1     Terminating   0          2m25s   app=mynapp-rc
mynapp-rc-gj9qc   1/1     Running       0          21m     app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running       0          21m     app=mynapp-rc
mynapp-rc-mmz9p   1/1     Running       0          15m     app=mynapp-rc,env=dev
```

mynapp-rc-mmz9p 파드의 레이블이 원래대로 app=mynapp-rc로 변경되었기 때문에, mynapp-rc 레플리케이션 컨트롤러의 관리 파드로 지정 되었다. 그러나 레플리케이션 컨트롤러는 관리 할 파드가 네 개가 되었으므로, 파드 하나를 삭제한다.

삭제되는 파드의 기준은 가장 최근(AGE)에 생성된 파드가 대상이다.

위 명령의 결과에는 종료중(Terminating)으로 나오지만, 시간에 따라 이미 종료되어 삭제되었을 수 있다.

### 6) 파드의 수평 스케일링
레플리케이션 컨트롤러의 복제본 수만큼 파드를 만들고 관리한다는 것을 확인하였다. 그러면 복제본 수를 변경하여 수평 스케일링을 해보자.

#### (1) 명령을 이용한 스케일링
앞서 살펴보았듯이 kubectl scale 명령으로 스케일링이 가능하다. 복제본 개수를 4로 스케일링 해보자.

```
$ kubectl scale replicationcontroller mynapp-rc --replicas=4

replicationcontroller/mynapp-rc scaled
```

mynapp-rc 레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME        DESIRED   CURRENT   READY   AGE
mynapp-rc   4         4         4       23m
```
모든 상태가 네 개로 구성되었다. 시간에 따라 준비된 파드가 세 개일 수도 있다. 이는 파드를 만드는 중이이다.

파드의 목록을 살펴보자.
```
$ kubectl get pods

NAME              READY   STATUS    RESTARTS   AGE
mynapp-rc-gj9qc   1/1     Running   0          23m
mynapp-rc-kxqhn   1/1     Running   0          23m
mynapp-rc-mmz9p   1/1     Running   0          18m
mynapp-rc-sv5gj   1/1     Running   0          20s
```
mynapp-rc-sv5gj 파드가 추가로 생성되었다. 이는 kubectl describe 명령의 결과중 Events 필드를 확인해도 어떠한 파드가 생성되었는지 확인할 수 있다.

#### (2) 오브젝트 수정을 이용한 스케일링
kubectl edit 명령으로 온라인 상태의 YAML 내용을 수정할 수 있다. 기본적 으로 vi 에디터로 수정이 가능하다. 복제본 개수를 하나 더 추가해 보자.

```
$ kubectl edit replicationcontrollers mynapp-rc

...
spec:
  replicas: 5
...

replicationcontroller/mynapp-rc edited
```
spec 항목에 replicas 항목을 찾아서 값을 변경한다.

> 참고  
> kubectl edit 명령은 스케일링 이외 일부 필드를 온라인 상태에서 변경 가능하다. 변경한 내용이 문법 오류일 경우 저장이 되지 않고, 변경사항이 반영이 되지도 않는다. 저장할 때 출력에 "XXX edited" 라고 나오는지 반드시 확인한다.

> 참고  
> kubectl edit 명령은 KUBE_EDITOR 환경 변수에 에디터를 지정하여 vi/vim 이외의 원하는 에디터로 수정할 수 있다.

> 참고  
> kubectl edit 명령은 실제 YAML 파일의 내용을 변경하지는 않는다.

mynapp-rc 레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME        DESIRED   CURRENT   READY   AGE
mynapp-rc   5         5         5       27m
```

파드의 목록을 살펴보자.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE     LABELS
mynapp-rc-c8fx7   1/1     Running   0          2m21s   app=mynapp-rc
mynapp-rc-gj9qc   1/1     Running   0          27m     app=mynapp-rc
mynapp-rc-kxqhn   1/1     Running   0          27m     app=mynapp-rc
mynapp-rc-mmz9p   1/1     Running   0          22m     app=mynapp-rc,env=dev
mynapp-rc-sv5gj   1/1     Running   0          4m16s   app=mynapp-rc
```
mynapp-rc-sv5gj 파드가 생성되었으며, 총 다섯 개의 파드가 존재한다.

#### (3) 오브젝트 파일 수정을 이용한 스케일링
앞서 mynapp-rc.yml 오브젝트 파일을 생성했었다. 현재 파드는 스케일링을 통해 다섯 개가 구성되었다. 그러나 YAML 파일을 수정한 것은 아니기 때문에 YAML 파일 내의 복제본 수는 세 개로 정의되어 있다.

YAML 파일을 이용한 스캐일링 방법은, YAML 파일을 적절하게 변경한 후, kubectl replace 명령을 사용하는 것이다. 이는 기존 생성된 오브젝트의 구성과 YAML 파일의 구성 정보를 비교한 후 변경된 사항에 대해서만 반영을 하게 된다.

```
$ kubectl replace -f mynapp-rc.yml

replicationcontroller/mynapp-rc replaced
```
파일을 변경하지 않고(복제본: 3) 실행했다면, 당연하겠지만 복제본은 세 개가 되며, 다섯 개 중 두 개는 삭제될 것이다.

레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME        DESIRED   CURRENT   READY   AGE
mynapp-rc   3         3         3       29m
```

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME              READY   STATUS    RESTARTS   AGE
mynapp-rc-gj9qc   1/1     Running   0          30m
mynapp-rc-kxqhn   1/1     Running   0          30m
mynapp-rc-mmz9p   1/1     Running   0          25m
```
세 개의 파드만 남았으며, 또는 두 개의 파드가 종료 중인 것을 확인할 수 있다.

### 7) 레플리케이션 컨트롤러 및 파드 삭제
앞서 살펴보았듯이 컨트롤러에 의해 관리되는 파드들은 파드 자체를 삭제 하더라도 복제본 조건에 따라 다시 생성하게 된다. 그래서 파드를 삭제하면 안되고, 컨트롤러를 직접 삭제해야 컨트롤러에 의해 관리되는 파드도 같이 삭제가 된다.

컨트롤러 삭제 명령은 다음과 같다.
```
$ kubectl delete replicationcontrollers mynapp-rc

replicationcontroller "mynapp-rc" deleted
```

레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

No resources found in default namespace.
```
해당 리소스는 이미 삭제되었기 때문에 존재하지 않는다.

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME              READY   STATUS        RESTARTS   AGE
mynapp-rc-gj9qc   1/1     Terminating   0          31m
mynapp-rc-kxqhn   1/1     Terminating   0          31m
mynapp-rc-mmz9p   1/1     Terminating   0          26m
```
컨트롤러를 삭제한지 얼마 되지 않았기 때문에 파드가 종료 중이다. 시간이 지나면 파드 또한 목록에 보이지 않을 것이다.
