## 4. 잡

### 1) 잡 소개
잡(Job) 컨트롤러는 파드(컨테이너)의 애플리케이션(잡)의 실행이 완료/종료되는 것에 초점을 맞춘 컨트롤러이다. 

레플리케이션 컨트롤러, 레플리카셋, 데몬셋은 파드의 애플리케이션이 계속적으로 잘 동작하는 것을 초점으로 맞춘 것에 대비된다. 즉, 애플리케이션이 실행되고 실행이 완료되면 파드의 할 일이 끝난 것으로 간주하고 파드를 종료시킨다. 

잡 컨트롤러는 임시(Ad-hoc) 작업 및 배치(Batch) 작업에 유용하게 사용될 수 있다.

만약 애플리케이션이 실행되고 있는중에 노드가 죽거나 파드의 실행이 완료 되지 않았다면, 파드를 다시 스케줄링하여 재실행하게 구성할 수 있다.

### 2) 잡 컨트롤러의 레이블 셀렉터
지금까지 확인했던 컨트롤러와 다르게, 잡/크론잡 컨트롤러는 일반적으로 파드 레이블 셀렉터(`.spec.selector`)를 직접 지정해 사용하지 않는다.

클러스터가 잡/크론잡 컨트롤러를 생성할 때 컨트롤러에 부여된 UID 기반의 레이블을 셀렉터로 사용한다.
```
controller-uid=cb95c6f9-711c-4602-b8c8-1098d0c153e0
```

일반적으로 레이블 셀렉터를 사용하지 않는 이유는, 셀렉터에 의해 관련없는 파드가 선택되어 잡이 이미 실행되고 있다고 판단하거나, 잡이 이미 완료되었다고 판단해 잡을 삭제하는 경우 등 예상하지 못한 결과를 발생할 수 있기 대문이다.

### 3) 잡 컨트롤러 생성
> myapp-job.yaml
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: myapp-job
spec:
  template:
    metadata:
      labels:
        app: myapp-job
    spec:
      restartPolicy: OnFailure
      containers:
      - name: myapp
        image: busybox
        command: ["sleep", "60"]
```
잡 오브젝트의 API는 batch 그룹의 v1 버전을 사용한다. 오브젝트의 종류는 Job 이다.

잡 컨트롤러는 레플리케이션 컨트롤러, 레플리카셋, 데몬셋처럼 계속 동작 하는 방식이 아니기 때문에, 재시작 정책(restartPolicy)을 기본값이 Always가 아닌 Onfailure 또는 Never로 명시적으로 선언해야 한다. 즉, 잡 컨트롤러의 파드 오브젝트는 재실행하지 않는다.

job.spec.template.spec.restartPolicy 재시작 정책:
- Always: (기본값) 종료/실패시 항상 재시작
- Onfailure: 실패시 재시작 (정상 종료시 재시작하지 않음)
- Never: 종료 또는 오류시 절대 재시작 하지 않음

이미지는 busybox 리눅스 이미지를 사용하였고, sleep 60 명령이 실행되도록 구성하였다. 즉, sleep 애플리케이션은 60초 동안 아무것도 하지않는 잡이며, 60초가 지나면 애플리케이션이 완료(Completed)된다.

잡 오브젝트를 생성해보자.
```
$ kubectl create -f myapp-job.yaml

job.batch/myapp-job created
```

### 4) 잡 컨트롤러 및 파드 확인
kubectl get 명령으로 잡 상태를 확인해보자.
```
$ kubectl get jobs.batch

NAME        COMPLETIONS   DURATION   AGE
myapp-job   0/1           12s        12s
```
하나의 파드가 있고 아직 잡이 완료되지 않았다는 의미이다.

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME              READY   STATUS    RESTARTS   AGE
myapp-job-7vqdx   1/1     Running   0          30s
```
현재 파드는 동작중이다.

시간이 약 1분 지난 후 다시 파드 상태를 확인해보자.
```
$ kubectl get pods
NAME              READY   STATUS      RESTARTS   AGE

myapp-job-7vqdx   0/1     Completed   0          68s
```
파드의 상태(STATUS)가 완료됨(Completed) 상태로 보고하고 있다. 실행중(Running) 상태가 아니다.

역시 잡 컨트롤러를 확인해 보면 잡이 완료된 것을 확인할 수 있다.
```
$ kubectl get jobs.batch

NAME        COMPLETIONS   DURATION   AGE
myapp-job   1/1           67s        88s
```


### 6) 다중 잡 컨트롤러 생성
잡 컨트롤러에서 다중 잡을 구성할 수 있다.

> myapp-job-comp.yaml\
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: myapp-job-comp
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: myapp-job-comp
    spec:
      restartPolicy: OnFailure
      containers:
      - name: myapp
        image: busybox
        command: ["sleep", "60"]
```

job.spec.completions: 작업의 완료 개수 지정

job.spec.completions 필드를 정의해 잡이 한번으로 완료되지 않고 작업을 여러번 순차적으로 실행 할 수 있다. 즉, 하나의 파드가 생성되어 잡이 실행되고 완료되면, 두 번째 파드가 생성되고 잡이 완료되고, 마지막으로 세 번째 파드가 생성되고 잡이 완료되어야 모든 잡이 완료된 것이다.

다중 잡 컨트롤러를 생성해 보자.
```
$ kubectl create -f myapp-job-comp.yaml

job.batch/myapp-job-comp created
```

### 9) 병렬 다중 잡 컨트롤러 생성
앞서 살펴본 다중 잡 컨틀롤러는 순서대로 직렬 형태로 실행되지만, 이를 병렬 형태로 동시에 실행 할 수도 있다.

> myapp-job-para.yaml
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: myapp-job-para
spec:
  completions: 3
  parallelism: 3
  template:
    metadata:
      labels:
        app: myapp-job-para
    spec:
      restartPolicy: OnFailure
      containers:
      - name: myapp
        image: busybox
        command: ["sleep", "60"]
```

job.spec.parallelism: 동시에 실행할 잡의 개수 지정

job.spec.parallelism 필드를 정의해 병렬로 동시에 실행할 잡의 개수를 지정할 수 있다. 총 세 개의 잡을 실행해야 하며, 동시에 병렬로 세 개의 잡을 실행한다는 의미다.

병렬 다중 잡 컨트롤러를 생성해 보자.
```
$ kubectl create -f myapp-job-para.yaml

job.batch/myapp-job-para created
```

### 12) 잡의 종료
잡은 파드의 실패나 컨테이너의 오류가 아닌 이상 애플리케이션이 종료되어야 잡이 종료된다. 정상적으로 잡을 종료하는 것이 아닌, 잡의 종료를 제어할 수 있는 방법이 존재한다.

#### (1) 파드 백오프(backoff)
잡 컨트롤러는 잡의 오류가 있을때 기본적으로 6번의 재시도 후 잡을 실패로 만든다.

.spec.backoffLimit: 잡을 실패로 만들기 전 재시도 회수 (기본: 6)

#### (2) 유효 데드라인
잡의 애플리케이션이 종료되기 전 유효 데드라인을 설정해 파드를 종료할 수 있다. 데드라인에 의해 종료된 파드는 실패한 것으로 간주하고, 재시작하지 않는다.

.spec.activeDeadlineSeconds: 잡이 시작된 후 작동할 유효한 시간

잡이 실행중이고 유효 데드라인이 설정된 경우, 백오프가 지정된 값까지 도달하지 않더라도 유효 데드라인 시간이 우선순위가 높다.

다음은 유효 데드라인이 설정된 오브젝트다.
> myapp-job-atvdl.yaml
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: myapp-job-atvdl
spec:
  backoffLimit: 3
  activeDeadlineSeconds: 30
  template:
    metadata:
      labels:
        app: myapp-job-atvdl
    spec:
      restartPolicy: OnFailure
      containers:
      - name: myapp
        image: busybox
        command: ["sleep", "60"]
```
백오프는 3번으로 제한을 하였고, 유효 데드라인은 30초로 설정하였다. 

유효 데드라인이 설정된 잡 오브젝트를 생성하자.
```
$ kubectl create -f myapp-job-atvdl.yaml

job.batch/myapp-job-atvdl created
```

파드의 목록 및 상태를 모니터링 하자.
```
$ kubectl get po --watch

NAME                    READY   STATUS    RESTARTS   AGE
myapp-job-atvdl-ctqgd   1/1     Running   0          10s
myapp-job-atvdl-ctqgd   1/1     Terminating   0          30s
myapp-job-atvdl-ctqgd   0/1     Terminating   0          60s
```
약 30초 후 파드가 종료되는 것을 확인할 수 있다.

잡 컨트롤러를 확인 해보자.
```
$ kubectl get jobs.batch

NAME              COMPLETIONS   DURATION   AGE
myapp-job-atvdl   0/1           66s        66s
```
잡은 여전히 완료되지 않고있다. 더이상 잡을 실행할 파드가 생성되지는 않는다.

잡 컨트롤러의 상세 정보를 확인 해보자.
```
$ kubectl describe jobs.batch myapp-job-atvdl

Name:                     myapp-job-atvdl
...
Start Time:               Tue, 02 Mar 2021 23:55:57 +0000
Active Deadline Seconds:  30s
Pods Statuses:            0 Running / 0 Succeeded / 1 Failed
...
Events:
  Type     Reason            Age    From            Message
  ----     ------            ----   ----            -------
  Normal   SuccessfulCreate  2m55s  job-controller  Created pod: myapp-job-atvdl-ctqgd
  Normal   SuccessfulDelete  2m25s  job-controller  Deleted pod: myapp-job-atvdl-ctqgd
  Warning  DeadlineExceeded  2m25s  job-controller  Job was active longer than specified deadline
```
파드의 상태는 Failed로 보고하고 있고, 컨트롤러의 이벤트는 DeadlineExceeded로 보고하고 있다.
