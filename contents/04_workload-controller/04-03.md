## 3. 레플라카셋

### 1) 레플리카셋 소개
쿠버네티스가 처음 나왔을 때 파드를 복제하고 파드가 실패했을 때 새 파드를 만드는 컨트롤러는 앞서 살펴본 레플리케이션 컨트롤러가 유일한 컨트롤러였다. 그러나 레플리케이션 컨트롤러의 몇몇 문제점(?) 및 기능 개선을 위해 레플라카셋 이라는 컨트롤러를 추가하였다.

레플리카셋(ReplicaSet)은 레플리케이션 컨트롤러가 제공하는 기능과 거의 비슷하다. 기본적으로 요구하는 복제본 개수만큼 파드를 복제하고 모니터링하고 관리한다.

### 2) 레플리카셋과 레플리케이션 컨트롤러의 비교
레플라카셋의 기본 목적은 레플리케이션 컨트롤러와 같지만 더 개선된 기능을 가지고 있다.

개선된 기능은 다음과 같다.
- 파드의 다중 레이블 지원
- 파드에 설정된 레이블의 키만 선택 가능

즉, 레플리케이션 컨트롤러는 파드의 레이블 중 하나의 레이블만 레이블 셀렉터로 관리 파드를 지정할 수 있고, 반드시 레이블의 “키=값“이 모두 맞아야만 파드 를 지정할 수 있었다. 그러나 레플리카셋은 다중 레이블을 지정할 수 있고, 레이블의 ”키“ 값만 가지고도 레이블을 지정할 수 있다.

### 3) 레플리카셋 생성
지금까지 생성한 오브젝트(파드 및 레플케이션 컨트롤러)의 API 버전은 v1이었지만, 지금 생성하는 레플리카셋은 apps/v1 API를 사용한다. 즉, 레플리카셋은 apps API 그룹에 속하며 v1 버전을 사용한다.

> 참고  
> 레플라카셋의 API 그룹과 버전 정보는 다음 명령으로 확인할 수 있다.  
> kubectl api-resources  
> kubectl api-versions  

레플리케이션 컨트롤러에서 생성한 mynapp-rc.yml 파일과 똑같은 기능을 하는 레플리카셋을 생성해보자.

> mynapp-rs.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-rs
  template:
    metadata:
      labels:
        app: mynapp-rs
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        ports:
        - containerPort: 8080
```

앞서 설명한 대로 apiVersion은 apps/v1를 사용하며 생성할 종류는 ReplicaSet이다. 또한 레플리케이션 컨트롤러에서는 레이블 셀렉터 항목에 레이블을 직접 지정했지만, 레플리카셋은 matchLabels 및 matchExpressions 필드로 레이블을 선택한다.

> 참고  
> 레플라카셋의 레이블 셀렉터  
> kubectl explain replicaset.spec.selector

matchLabels는 레플리케이션 컨트롤러와 같이 특정 레이블의 키=값을 매칭하게 된다. 기능상 완전하게 똑같이 작동한다.

레플리카셋을 생성해 보자.

```
$ kubectl create -f mynapp-rs.yml

replicaset.apps/mynapp-rs created
```

### 5) 레플리카셋의 레이블 셀렉터 사용

#### (1) matchLabels 레이블 셀렉터
```yaml
spec:
  selector:
    matchLabels:
      key: value
```
matchLabels로 레이블 셀렉터를 사용하는 경우는 레플리케이션 컨트롤러와 똑같이 동작한다.

#### (2) matchExpressions 레이블 셀렉터
```yaml
spec:
  selector:
    matchExpressions:
    - key: <string>
      operator: <In | NotIn | Exists | DoesNotExist>
      values:
      - <string>
```
matchExpressions 필드는 레플리카셋의 기능이다. key는 선택할 레이블의 키 이름을 지정하는 필드이고, values는 레이블의 값을 지정하는 필드이다. operator 필드는 키, 값의 연산을 담당하는 부분으로 네 가지 중에 하나를 선택해서 매칭 할 수 있다.

연산자는 다음과 같이 동작한다.
- In: 레이블의 키와 값이 지정된 값으로 일치해야 함
- NotIn: 레이블의 키와 값이 지정된 값과 일치하지 않아야 함
- Exists: 레이블의 키가 포함되어야 함 (값은 해당사항 없음)
- DoesNotExist: 레이블의 키가 포함되지 않아야 함

Exists와 DoesNotExist는 레이블의 키 값만 매칭 한다. 즉, values 항목을 설 정하지 않는다.

### 6) matchExpressions를 사용한 레플리카셋 생성

> mynapp-rs-exp.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-exp
spec:
  replicas: 3
  selector:
    matchExpressions:
      - key: app
        operator: In
        values:
          - mynapp-rs-exp
      - key: env
        operator: Exists
  template:
    metadata:
      labels:
        app: mynapp-rs-exp
        env: dev
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        ports:
        - containerPort: 8080
```
matchExpressions 레이블 셀렉터는 반드시 YAML의 리스트로 표현해야 한다. 매칭하고자 하는 셀렉터는 app 키에 mynapp-rs-exp 값이 있어야 하며, env 키가 있어야 한다. 두 조건 다 만족해야 하는 AND 연산이다.

파드 템플릿의 레이블은 app=mynapp-rs-exp 및 env=dev로 설정하였다.

### 7) 레플리카셋 확인
```
$ kubectl get replicasets.apps -o wide

NAME        DESIRED   CURRENT   READY   AGE   CONTAINERS   IMAGES           SELECTOR
mynapp-rs   3         3         3       5m    mynapp       c1t1d0s7/myweb   app=mynapp-rs
```

kubectl get 명령에 -o wide 옵션을 사용하면 조금 더 자세한 정보를 확인 할 수 있다. 여기서는 컨트롤러의 SELECTOR(레이블 셀렉터) 필드를 보기위해 옵션을 사용하였다.

레플리카셋 컨트롤러에 의해 생성된 파드의 목록을 확인한다.
```
$ kubectl get pods --show-labels

NAME              READY   STATUS    RESTARTS   AGE     LABELS
mynapp-rs-88dgr   1/1     Running   0          5m41s   app=mynapp-rs
mynapp-rs-k8rpm   1/1     Running   0          5m41s   app=mynapp-rs
mynapp-rs-qn49r   1/1     Running   0          5m41s   app=mynapp-rs
```

레플리카셋을 정리하면 레플리케이션 컨트롤러와 기능이 같지만, 다중 레이블 셀렉터 및 레이블 키 만 선택할 수 있는 기능이 추가 되었다.

추후 레플리케이션 컨트롤러는 더 이상 사용되지 않을 것으로, 레플리카셋이 대체 할 것이다.

### 8) 레플리카셋 삭제
레플리카셋을 삭제한다.
```
$ kubectl delete replicasets.apps mynapp-rs

replicaset.apps "mynapp-rs" deleted
```
