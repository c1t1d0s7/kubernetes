## 5. 잡

### 1) 잡 소개
잡(Job) 컨트롤러는 파드(컨테이너)의 애플리케이션(잡)의 실행이 완료되는 것에 초점을 맞춘 컨트롤러이다. 레플리케이션 컨트롤러, 레플리카셋, 데몬셋은 파드의 애플리케이션이 계속으로 잘 동작하는 것을 초점으로 맞춘 것에 대비된다. 즉, 애플리케이션이 실행되고 실행이 완료되면 파드의 할 일이 끝난 것으로 간주하고 파드를 종료시킨다. 잡 컨트롤러는 임시(Ad-hoc) 작업 및 배치(Batch) 작업에 유용하게 사용될 수 있다.

만약 애플리케이션이 실행되고 있는중에 노드가 죽거나 파드의 실행이 완료 되지 않았다면, 파드를 다시 스케줄링하여 재실행하게 구성할 수 있다.

### 2) 잡 컨트롤러 생성
> mynapp-job.yml 파일
```yaml
apiVersion: batch/v1beta1
kind: Job
metadata:
  name: mynapp-job
spec:
  template:
    metadata:
      labels:
        app: mynapp-job
    spec:
      restartPolicy: OnFailure
      containers:
      - name: mynapp
        image: busybox
        command: ["sleep", "60"]
```
잡 오브젝트의 API는 batch 그룹의 v1 버전을 사용한다. 오브젝트의 종류는 Job 이다.

잡 컨트롤러는 레플리케이션 컨트롤러, 레플리카셋, 데몬셋처럼 계속 동작 하는 방식이 아니기 때문에, 재시작 정책(restartPolicy)을 기본값이 Always가 아닌 Onfailure 또는 Never로 명시적으로 선언해야 한다. 즉, 잡 컨트롤러의 파드 오브젝트는 재실행하지 않는다.

job.spec.template.spec.restartPolicy 재시작 정책 필드:
- Always: (기본값) 종료/실패시 항상 재시작
- Onfailure: 실패시 재시작 (정상 종료시 재시작하지 않음)
- Never: 종료 또는 오류시 절대 재시작 하지 않음

이미지는 busybox 리눅스 이미지를 사용하였고, sleep 60 명령이 실행되도록 구성하였다. 즉, sleep 애플리케이션은 60초 동안 아무것도 하지않는 잡이며, 60초가 지나면 애플리케이션이 완료(Completed)된다.

잡 오브젝트를 생성해보자.
```
$ kubectl create -f mynapp-job.yml

job.batch/mynapp-job created
```

### 3) 잡 컨틀로러 및 파드 확인
kubectl get 명령으로 잡 상태를 확인해 보자.
```
$ kubectl get jobs.batch

NAME         COMPLETIONS   DURATION   AGE
mynapp-job   0/1           6s         6s
```
하나의 파드가 있고 아직 잡이 완료되지 않았다는 의미이다.

파드의 목록을 확인해 보자.
```
$ kubectl get pods

NAME               READY   STATUS    RESTARTS   AGE
mynapp-job-vnp48   1/1     Running   0          16s
```
현재 파드는 동작중이다.

시간이 약 1분 지난 후 다시 파드 상태를 확인해 보자.
```
$ kubectl get pods

NAME               READY   STATUS      RESTARTS   AGE
mynapp-job-vnp48   0/1     Completed   0          67s
```
파드의 상태(STATUS)가 완료됨(Completed) 상태로 보고하고 있다. 실행중(Running) 상태가 아니다.

역시 잡 컨트롤러를 확인해 보면 잡이 완료된 것을 확인할 수 있다.
```
$ kubectl get jobs.batch

NAME         COMPLETIONS   DURATION   AGE
mynapp-job   1/1           66s        70s
```

### 4) 잡 컨트롤러 삭제
완료된 잡 컨트롤러를 삭제한다.
```
$ kubectl delete -f mynapp-job.yml

job.batch "mynapp-job" deleted
```

### 5) 다중 잡 컨트롤러 생성
잡 컨트롤러에서 다중 잡을 구성할 수 있다.

> mynapp-job-comp.yml 파일
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: mynapp-job-comp
spec:
  completions: 3
  template:
    metadata:
      labels:
        app: mynapp-job-comp
    spec:
      restartPolicy: OnFailure
      containers:
      - name: mynapp
        image: busybox
        command: ["sleep", "60"]
```
job.spec.completions 필드를 사용하면 잡이 한번으로 완료되지 않고 세 번의 작 업을 순차적으로 실행 할 수 있다. 즉, 하나의 파드가 생성되어 잡이 실행되고 완료되면, 두 번째 파드가 생성되고 잡이 완료되고, 마지막으로 세 번째 파드가 생성되고 잡이 완료되어야 모든 잡이 완료된 것이다.

다중 잡 컨트롤러를 생성해 보자.
```
$ kubectl create -f mynapp-job-comp.yml

job.batch/mynapp-job-comp created
```

### 6) 다중 잡 컨트롤러 확인
다중 잡 컨트롤러를 확인해 보자.
```
$ kubectl get jobs.batch

NAME              COMPLETIONS   DURATION   AGE
mynapp-job-comp   0/3           5s         5s
```
세 개의 잡이 있으며 아직 완료된 잡은 없다.

파드의 상태 및 목록을 시간을 두고 확인해 보자.
```
$ kubectl get pods --watch
NAME                    READY   STATUS    RESTARTS   AGE
mynapp-job-comp-8g8bk   1/1     Running   0          58s
mynapp-job-comp-8g8bk   0/1     Completed   0          66s
mynapp-job-comp-bxxq7   0/1     Pending     0          0s
mynapp-job-comp-bxxq7   0/1     ContainerCreating   0          0s
mynapp-job-comp-bxxq7   1/1     Running             0          6s
mynapp-job-comp-bxxq7   0/1     Completed           0          66s
mynapp-job-comp-6spf4   0/1     Pending             0          0s
mynapp-job-comp-6spf4   0/1     ContainerCreating   0          0s
mynapp-job-comp-6spf4   1/1     Running             0          5s
mynapp-job-comp-6spf4   0/1     Completed           0          65s
```
시간이 지남에 따라 순차적으로 세 개의 잡을 실행하고 완료하는 것을 확인할 수 있다.

잡 컨트롤러도 확인해 보자.
```
$ kubectl get jobs.batch

NAME              COMPLETIONS   DURATION   AGE
mynapp-job-comp   3/3           3m17s      3m27s
```
약 3분 동안 진행되었으며, 세 개의 잡이 완료된 것을 확인할 수 있다.

### 7) 다중 잡 컨트롤러 삭제
다중 잡 컨틀로러를 삭제한다.
```
$ kubectl delete -f mynapp-job-comp.yml

job.batch "mynapp-job-comp" deleted
```

### 7) 병렬 다중 잡 컨트롤러 생성
앞서 살펴본 다중 잡 컨틀롤러는 순서대로 직렬 형태로 실행되지만, 이를 병렬 형태로 동시에 실행 할 수도 있다.

> mynapp-job-comp-parallel.yml 파일
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: mynapp-job-para
spec:
  completions: 3
  parallelism: 3
  template:
    metadata:
      labels:
        app: mynapp-job-para
    spec:
      restartPolicy: OnFailure
      containers:
      - name: mynapp
        image: busybox
        command: ["sleep", "60"]
```
job.spec.parallelism 필드를 사용하여 병렬로 동시에 실행할 잡의 개수를 지정할 수 있다. 총 세 개의 잡을 실행해야 하며, 동시에 병렬로 세 개의 잡을 실행한다는 의미이다.

병렬 다중 잡 컨트롤러를 생성해 보자.
```
$ kubectl create -f mynapp-job-comp-parallel.yml

job.batch/mynapp-job-para created
```

### 8) 병렬 다중 잡 컨트롤러 확인
잡 컨트롤러의 상태를 확인해보자.
```
$ kubectl get jobs.batch

NAME              COMPLETIONS   DURATION   AGE
mynapp-job-para   0/3           4s         4s
```
세 개의 잡이 실행되어야 하며, 아직 완료된 잡은 없다.

파드의 상태 및 목록을 시간을 두고 확인해 보자.
```
$ kubectl get pods --watch

NAME                    READY   STATUS    RESTARTS   AGE
mynapp-job-para-b2z4c   1/1     Running   0          8s
mynapp-job-para-f7qj7   1/1     Running   0          8s
mynapp-job-para-wk5vr   1/1     Running   0          8s
mynapp-job-para-f7qj7   0/1     Completed   0          65s
mynapp-job-para-wk5vr   0/1     Completed   0          65s
mynapp-job-para-b2z4c   0/1     Completed   0          65s
```
동시에 세 개의 파드가 실행되고 완료된 것을 확인할 수 있다.

다시 잡 컨트롤러의 상태를 살펴보자.
```
$ kubectl get jobs.batch

NAME              COMPLETIONS   DURATION   AGE
mynapp-job-para   3/3           65s        74s
```
세 개의 파드(잡)이 완료되었다.

### 9) 병렬 다중 잡 컨트롤러 삭제
```
$ kubectl delete -f mynapp-job-comp-parallel.yml

job.batch "mynapp-job-para" deleted
```
