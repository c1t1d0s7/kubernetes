## 6. 크론잡

### 1) 크론잡 소개
잡 컨트롤러는 잡이 한번만 실행이 되지만 크론잡(CronJob) 컨트롤러는 주기적으로 잡을 반복적으로 실행할 필요가 있을 때 사용하는 컨트롤러이다. 리눅스 시스템의 crontab 도구와 기능이 같다.

### 2) 크론잡 컨트롤러 생성
크론잡 컨트롤러는 아직 베타 기능이며 크론잡 API 역시 batch 그룹을 사용하며 버전은 v1beta1을 사용한다. 역시 오브젝트 종류는 CronJob이다.

> mynapp-cjob.yml 파일
```yaml
apiVersion: batch/v1beta1
kind: CronJob
metadata:
  name: mynapp-cjob
spec:
  schedule: "*/2 * * * *"
  jobTemplate:
    spec:
      template:
        metadata:
           labels:
             app: mynapp-cjob
        spec:
          restartPolicy: OnFailure
          containers:
          - name: mynapp
            image: busybox
            command: ["sleep", "60"]
```
가장 중요한 부분은 스케줄을 지정하는 필드다. cronjob.spec.schedule 필드를 사용 하여 주기적인 시간을 구성한다. 스케줄 구성하는 값은 다섯 개의 필드로 구분되어 지며 다음과 같다.

ronjob.spec.schedule 스케줄러 필드:
- 분
- 시
- 일
- 월
- 요일(0-일요일, 1-월요일, 6-토요일)

스케줄링은 2분마다 실행하도록 구성하였다.

크론잡 컨트롤러를 생성해보자.
```
$ kubectl create -f mynapp-cjob.yml

cronjob.batch/mynapp-cjob created
```

### 3) 크론잡 컨트롤러 및 파드 확인
크론잡 오브젝트의 상태를 확인한다.
```
$ kubectl get cronjobs.batch

NAME          SCHEDULE      SUSPEND   ACTIVE   LAST SCHEDULE   AGE
mynapp-cjob   */2 * * * *   False     0        <none>          5s
```
최초 생성 후 활성화된(ACTIVE) 잡은 없으며, 마지막으로 동작한 잡(LAST SCHEDULE)도 현재까지 없다. 약 2분의 시간이 흐른 후 다시 확인해 보면 최초의 잡이 실행된 것을 확인할 수 있다.

약 2분이 흐른뒤 다시 확인해 보자
```
$ kubectl get cronjobs.batch

NAME          SCHEDULE      SUSPEND   ACTIVE   LAST SCHEDULE   AGE
mynapp-cjob   */2 * * * *   False     1        48s              5s
```

파드의 목록 및 상태를 확인해 보자.
```
$ kubectl get pods --watch

NAME                           READY   STATUS    RESTARTS   AGE
mynapp-cjob-1584420840-sdq82   0/1     Pending   0          0s
mynapp-cjob-1584420840-sdq82   0/1     ContainerCreating   0          0s
mynapp-cjob-1584420840-sdq82   1/1     Running             0          5s
mynapp-cjob-1584420840-sdq82   0/1     Completed           0          65s
```
파드가 실행되었고, 약 1분 후 완료된 것을 확인할 수 있다.

크론잡 상태를 다시 확인해 보자.
```
$ kubectl get cronjobs.batch
NAME          SCHEDULE      SUSPEND   ACTIVE   LAST SCHEDULE   AGE
mynapp-cjob   */2 * * * *   False     0        94s             2m11s
```
현재 활성화된 잡은 없으며, 94초전 마지막 스케줄이 실행되었다고 보고하고 있다.

### 4) 크론잡 컨트롤러 삭제
```
$ kubectl delete -f mynapp-cjob.yml

cronjob.batch "mynapp-cjob" deleted
```

### 5) 크론잡 컨트롤러의 제한사항 소개
크론잡 컨트롤러는 특정 스케줄링된 시간마다 하나의 잡을 실행한다. 그러나 특정 환경에서 잡이 실행되지 않거나 잡이 두개 이상 실행되는 경우도 있을 수 있다.

이를 제어하기 위해 크론잡 컨트롤러의 잡의 제한사항을 설정할 수 있다.

cronjob.spec.startingDeadlineSeconds 시작 데드라인 시간 필드:
- 초: 어떤 이유든 예정된 잡 시작시간에 대한 데드라인
  * 100초: 지금으로 부터 100초 안에 일정을 놓친 잡이 있는지 확인

cronjob.spec.concurrencyPolicy 동시 실행 정책 필드:
- Allow: (기본) 잡이 동시에 실행될 수 있음
- Forbid: 동시 실행 금지, 잡이 아직 완료되지 않은 경우 다음 잡은 건너뜀
- Replace: 현재 실행 중인 잡을 취소하고 새 잡으로 교체

만약 startingDeadlineSecounds가 100초 이고 concurrencyPolicy가 Allow 이면, 이전에 놓친 잡이 있으면 해당 잡을 실행시키게 된다. 또한 concurrencyPolicy가 Forbid 이면, 이전 놓친 잡은 실행하지 않는다.

> mynapp-cjob-deadline.yml 파일
```yaml
apiVersion: batch/v1beta1
kind: CronJob
metadata:
  name: mynapp-cjob-dl
spec:
  schedule: "*/2 * * * *"
  startingDeadlineSeconds: 10
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      template:
        metadata:
           labels:
             app: mynapp-cjob-dl
        spec:
          restartPolicy: OnFailure
          containers:
          - name: mynapp
            image: busybox
            command: ["sleep", "60"]
```
