## 1. 라이브니스 프로브
앞에서 살펴본 파드는 직접 수동으로 생성, 모니터링, 삭제 등 관리하는 방법을 살펴 보았다. 그러나 이런 “관리되지 않는” 파드는 실제 프로덕션 환경에서 사용하기 좋지 않다. 모든 것을 수동으로 관리해 줘야하기 때문이다.

실제 환경에서는 수동으로 개입하지 않더라도 항상 정상적이고 안정적인 상태가 유지되기를 바랄 것이고, 쿠버네티스 시스템은 이런 바람에 부합하게 동작한다. 바로 라이브니스 프로브를 사용하는 것이다.

### 1) 라이브니스 프로브 소개
라이브니스 프로브(Liveness Probe)는 파드에 의해 컨테이너가 동작하게 되고, 동작하고 있는 컨테이너가 잘 동작하는지 주기적으로 모니터링 한다. 만약 모니터링 도중 파드의 오류가 발생한다면 해당 컨테이너를 다시 시작 하게 된다.

라이브니스 프로브는 세 가지 메커니즘을 가지고 컨테이너의 상태를 모니터링 한다.
- HTTP GET 프로브
  * 특정 경로에 HTTP GET 요청
  * HTTP 응답코드가 2XX 또는 3XX 인지 확인 함
- TCP 소켓 프로브
  * 특정 TCP 포트 연결을 시도 함
- Exec 프로브
  * 컨테이너 내부의 바이너리(명령)을 실행하고 종료 코드 확인

### 2) 라이브니스 프로브 생성
라이브니스 프로브는 livenessProbe 지시어를 사용하여 정의한다.

기본적인 형식은 다음과 같다.
```yaml
...
spec:
  containers:
  - livenessProbe:
      httpGet:
        ...
      tcpSocket:
        ...
      exec:
        ...
...
```

> 참고  
$ kubectl explain pod.spec.containers.livenessProbe

#### (1) 라이브니스 프로브 생성 - 정상
> mynapp-pod-liveness.yml 파일
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-liveness
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
    ports:
    - containerPort: 8080
      protocol: TCP
    livenessProbe:
      httpGet:
        path: /
        port: 8080
```
HTTP GET 프로브를 사용하였으며, 경로는 / 포트는 8080 이다. 즉, 결론적으로 200 코드 응답이 돌아올 것으로 예상된다.

파드 오브젝트를 생성한다.
```
$ kubectl create -f mynapp-pod-liveness.yml

pod/mynapp-pod-liveness created
```

파드 상태를 주기적으로 확인한다.
```
$ kubectl get pods --watch

NAME                  READY   STATUS    RESTARTS   AGE
mynapp-pod-liveness   1/1     Running   0          13s
```
계속 파드의 상태를 모니터링을 할 것이므로 다음 작업은 다른 터미널에서 실행한다.

> 참고  
> 유닉스 계열은 watch 명령을 이용하여 모니터링 할 수 있다.
> $ watch -n1 -d <COMMAND>

#### (2) 라이브니스 프로브 생성 - 비정상
> mynapp-pod-liveness-404.yml 파일
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-liveness-404
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
    ports:
    - containerPort: 8081
      protocol: TCP
    livenessProbe:
      httpGet:
        path: /
        port: 8082
```
의도적으로 오류가 나도록 구성한 것으로, 실제 컨테이너는 8080 포트를 사용하고 있고, 쿠버네티스 시스템에서는 접속을 8081로 하도록 정의하였다. 또한 라이브니스 프로브는 8082 포트를 확인하도록 했다.

파드 오브젝트를 생성한다.
```
$ kubectl create -f mynapp-pod-liveness-404.yml

pod/mynapp-pod-liveness-404 created
```

#### (3) 라이브니스 상태 확인
기존의 모니터링을 하고있는 터미널에 표시된 파드 상태를 주기적으로 확인한다.
```
$ kubectl get pods --watch

NAME                      READY   STATUS    RESTARTS   AGE
mynapp-pod-liveness       1/1     Running   0          13s
mynapp-pod-liveness-404   0/1     Pending   0          0s
mynapp-pod-liveness-404   0/1     Pending   0          0s
mynapp-pod-liveness-404   0/1     ContainerCreating   0          0s
mynapp-pod-liveness-404   1/1     Running             0          4s
mynapp-pod-liveness-404   1/1     Running             1          63s
mynapp-pod-liveness-404   1/1     Running             2          2m3s
...
```

시간이 조금 흘러 확인해보면 RESTARTS(재시작) 필드가 0에서 양수로 변경된 것을 확인할 수 있다. 이는 라이브니스 프로브에 의해 건강한 상태가 아님을 감지하고 재시작을 시도 했다는 의미이다.

```
mynapp-pod-liveness-404   1/1     Running   3          3m46s
mynapp-pod-liveness-404   1/1     Running   4          4m4s
mynapp-pod-liveness-404   1/1     Running   5          5m4s
mynapp-pod-liveness-404   0/1     CrashLoopBackOff   5          6m1s
```
시간이 조금더 흘러 확인해 보면 상태가 CrashLoopBackOff 상태인 것을 확인할 수 있다. 또한 RESTART 횟수도 증가하였다.

```
$ kubectl describe pods mynapp-pod-liveness-404
...
Name:         mynapp-pod-liveness-404
...
    Last State:     Terminated
      Reason:       Error
      Exit Code:    137
      Started:      Tue, 17 Mar 2020 00:18:19 +0000
      Finished:     Tue, 17 Mar 2020 00:19:16 +0000
    Ready:          True
    Restart Count:  4
    Liveness:       http-get http://:8082/ delay=0s timeout=1s period=10s #success=1 #failure=3
...
```

Exit Code는 프로세스를 종료하기 위한 코드를 의미한다.
> 137 = 128 + 9  
> 9번 시그널: SIGKILL  
즉 SIGKILL 시그널을 이용해서 종료되었음을 의미한다.

Liveness는 YAML 파일에 정의한 내용이며 추가적으로 delay, timeout, period가 있다. 의미는 기본적으로 파드(컨테이너) 실행 후 바로(delay=0) 모니터링 하면 10초 간격으로 1초 타임아웃으로 동작한다.

파드(컨테이너)가 실행되고 난 이후 애플리케이션에 따라 애플리케이션이 정상 동작하는데 시간이 걸릴 수 있으므로 초기 지연 시간을 설정하는 것이 중요하다. 초기 지연시간은 pod.spec.containers.livenessProve.initialDelaySeconds 필드로 정의할 수 있다.
```yaml
...
    livenessProbe:
      httpGet:
        path: /
        port: 8080
      initialDelaySeconds: 30
...
```

kubectl describe 명령의 결과에서 Events 필드는 파드(컨테이너)의 상태를 시간대 별로 확인할 수 있으며, 라이브니스 프로브의 문제가 있다는 것을 알 수 있다.

불필요한 파드를 삭제한다.
```
$ kubectl delete pods --all

pod "mynapp-pod-liveness" deleted
pod "mynapp-pod-liveness-404" deleted
```
--all 옵션은 모든 리소스를 대상으로 한다.

라이브니스 프로브는 현재 동작중인 파드의 크래시나 애플리케이션 장애를 감지해 재시작 해준다. 그러나 여러 이유에 의해 파드가 삭제되거나 노드가 장애가 발생한 경우 파드를 재시작 해주지는 않는다.

파드의 삭제나 노드의 장애에 대비해 가용성을 높이기 위해서는 다음부터 나오는 레플리케이션 컨트롤러, 레플리카셋, 데몬셋 등 컨트롤러를 사용해야 한다.
