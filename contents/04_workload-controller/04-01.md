## 1. 레플리케이션 컨트롤러
쿠버네티스의 컨트롤러는 파드를 올바르게 동작하기 위해 특정 상태를 보장하기 위한 컨트롤러다. 특정 상태는 컨트롤러에 따라 동작하는 방식 및 정의하는 상태가 조금씩 다르다.

### 1) 레플리케이션 컨트롤러 소개
> 참고
> 레플리케이션 컨트롤러보다 **레플리카셋** 또는 레플리카셋의 상위 개념인 **디플로이먼트**를 권장

레플리케이션 컨트롤러(ReplicationController)는 파드가 특정 개수만큼 복제되고 동작하는 것을 보장한다.

노드의 문제가 발생했거나 파드가 문제가 생겨 원하는 수의 파드가 동작하지 않는다면 자동으로 스케줄러에 의해 새로운 노드나 기존 노드에 다시 새로운 파드를 생성해 원하는 수의 파드를 유지하고자 복제한다.

심지어 일반적인 형태에서 발생할 일은 많이 없지만, 파드가 원하는 수보다 많은 경우에도, 원하는 파드 수로 맞춘다. 즉, 불필요한 파드를 제거해 원하는 수를 맞춘다.

원하는 수의 복제본 보다 더 많은 복제본이 많아질 수 있는 경우:
- 수동으로 동일한 형식을 파드를 생성
- 기존의 파드의 유형을 변경
- 원하는 수의 파드 복제본 수를 줄인 경우

즉, 실행 중인 파드의 목록을 주기적으로 모니터링하고, 원하는 수의 파드 수와 항상 일치시킨다.

레플리케이션 컨트롤러를 구성하기 위해서는 세 가지 요소가 있다.
- 파드를 지정하는 레이블 셀렉터
- 새로운 파드의 복제본을 만들기 위한 파드 템플릿
- 복제본 수(기본값: 1)

레플리케이션 컨트롤러가 제공하는 기능은 다음과 같다.
- 원하는 복제본 수의 파드가 없는 경우 파드 템플릿을 이용하여 파드를 생성한다.
- 노드에 장애가 발생하면 장애가 발생한 노드에서 실행 중이던 파드를 다른 노드에 복제본을 생성한다.
- 수동이나 자동으로 파드를 수평적으로 스케일링할 수 있다.

### 2) 레플리케이션 컨트롤러 생성
레플리케이션 컨트롤러를 앞서 살펴봤듯이 세 가지 요소가 필요하다.

> myapp-rc.yaml
```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: myapp-rc
spec:
  replicas: 3
  selector:
    app: myapp-rc
  template:
    metadata:
      labels:
        app: myapp-rc
    spec:
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
        ports:
        - containerPort: 8080
```
복제본 수는 세 개로 구성하고, 레플리케이션이 관리할 파드의 레이블 셀렉터는 app=myapp-rc 레이블이다. 또한 생성할 파드는 템플릿에 정의하였고, 당연히 레이블은 app=myapp-rc로 구성하였다.

> 참고
> 레이블 셀렉터의 레이블과 파드 템플릿의 레이블이 맞지 않으면 생성되지 않는다.

레플리케이션 컨트롤러를 생성해 보자.
```
$ kubectl create -f myapp-rc.yaml

replicationcontroller/myapp-rc created
```

### 3) 레플리케이션 컨트롤러 확인
레플리케이션 컨트롤러의 목록 및 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME       DESIRED   CURRENT   READY   AGE
myapp-rc   3         3         3       16s
```
원하는(DESIRED) 복제본의 개수는 3개이고, 현재(CURRENT) 존재하는 파드는 3개, 준비(READY) 된 파드도 3개라고 출력하고 있다.

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rc-6vjrq   1/1     Running   0          86s
myapp-rc-79hgq   1/1     Running   0          86s
myapp-rc-n4997   1/1     Running   0          86s
```
파드의 이름은 컨트롤러의 이름(myapp-rc)에 무작위 문자 다섯 자가 추가되어 있다.

레플리케이션 컨트롤러가 처음 만들어졌을 때는 app=mynapp-rc 레이블을 가지고 있는 파드가 하나도 없기 때문에, 복제본 3개를 생성하게 된다.

파드에 구성된 레이블을 같이 살펴보자.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS    RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running   0          2m45s   app=myapp-rc
myapp-rc-79hgq   1/1     Running   0          2m45s   app=myapp-rc
myapp-rc-n4997   1/1     Running   0          2m45s   app=myapp-rc
```

레플리케이션 컨트롤러의 자세한 정보를 확인해보자. kubectl describe 명령으로 확인할 수 있다.


### 4) 레플리케이션 컨트롤러와 파드
레플리케이션 컨트롤러에서 노드의 장애 발생을 가정하고 파드를 삭제했을 때 어떻게 되는지 확인해보자.

```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rc-6vjrq   1/1     Running   0          4m31s
myapp-rc-79hgq   1/1     Running   0          4m31s
myapp-rc-n4997   1/1     Running   0          4m31s
```

마지막 파드(myapp-rc-n4997)를 삭제해 본다.
```
$ kubectl delete pod myapp-rc-n4997

pod "myapp-rc-n4997" deleted
```

다시 파드의 목록을 확인해 본다.
```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rc-6vjrq   1/1     Running   0          5m13s
myapp-rc-79hgq   1/1     Running   0          5m13s
myapp-rc-lwwbz   1/1     Running   0          24s
```
myapp-rc-n4997 파드는 삭제됐고, myapp-rc-lwwbz 이란 새로운 파드가 생성되었다. 레플리케이션 컨트롤러는 레이블 셀렉터에 의해 관리되는 파드를 확인하고 있다가, 하나의 파드가 삭제(없어진) 된 것을 확인하고, 파드의 복제본 수와 맞추기 위해 하나의 파드를 새롭게 생성한 것이다.

레플리케이션 컨트롤러의 자세한 정보를 확인해보면 정확하게 어떤 파드가 생성되었는지 확인할 수 있다.
```
$ kubectl describe replicationcontrollers myapp-rc

Name:         myapp-rc
...
Events:
  Type    Reason            Age    From                    Message
  ----    ------            ----   ----                    -------
  Normal  SuccessfulCreate  7m23s  replication-controller  Created pod: myapp-rc-79hgq
  Normal  SuccessfulCreate  7m23s  replication-controller  Created pod: myapp-rc-n4997
  Normal  SuccessfulCreate  7m23s  replication-controller  Created pod: myapp-rc-6vjrq
  Normal  SuccessfulCreate  2m34s  replication-controller  Created pod: myapp-rc-lwwbz
```

> 참고  
> 파드를 삭제하기 전 watch 명령이나 --watch 옵션을 사용하여 실시간으로 확인할 수도 있다.

### 5) 레플리케이션 컨트롤러와 레이블
레플리케이션 컨트롤러는 관리할 파드를 파드의 레이블과 레플리케이션 컨트롤러의 레이블 셀렉터를 이용하여, 관리할 파드를 지정하고 관리하게 된다.

#### (1) 기존 파드에 레이블 추가
그러면 기존의 파드의 레이블을 추가하면 어떻게 될지 살펴보자. 우선 파드의 목록 및 레이블을 확인한다.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS    RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running   0          7m55s   app=myapp-rc
myapp-rc-79hgq   1/1     Running   0          7m55s   app=myapp-rc
myapp-rc-lwwbz   1/1     Running   0          3m6s    app=myapp-rc
```

myapp-rc-lwwbz 파드에 env=dev 레이블을 추가해 보자.
```
$ kubectl label pod myapp-rc-lwwbz env=dev

pod/myapp-rc-lwwbz labeled
```

다시 파드 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS    RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running   0          8m36s   app=myapp-rc
myapp-rc-79hgq   1/1     Running   0          8m36s   app=myapp-rc
myapp-rc-lwwbz   1/1     Running   0          3m47s   app=myapp-rc,env=dev
```
결과는 기존 파드에 추가 레이블만 구성되었고 변경된 사항은 없다.

#### (2) 기존 파드의 레이블 변경
그러면 기존 파드의 레이블을 변경해 보자, 즉 app=myapp-rc를 다른 레이블로 변경하면 어떤 결과가 발생하는지 살펴보자.

myapp-rc-lwwbz 파드의 app 레이블의 값을 test로 변경해보자.
```
$ kubectl label pod myapp-rc-lwwbz app=test --overwrite

pod/myapp-rc-lwwbz labeled
```

파드의 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS    RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running   0          9m30s   app=myapp-rc
myapp-rc-79hgq   1/1     Running   0          9m30s   app=myapp-rc
myapp-rc-lwwbz   1/1     Running   0          4m41s   app=test,env=dev
myapp-rc-rrwhf   1/1     Running   0          5s      app=myapp-rc
```

myapp-rc-lwwbz 파드의 레이블을 app=test로 변경한 이후, 레플리케이션 컨트롤러는 app=mynapp-rc 레이블을 가진 파드가 두 개밖에 없기 때문에, 새로운 myapp-rc-rrwhf 파드를 생성하였다. 

이제부터 myapp-rc-lwwbz 파드는 더 이상 myapp-rc 레플리케이션 컨트롤러가 관리하는 파드가 아니다.

다시 원래대로 myapp-rc-lwwbz 파드의 레이블을 app=myapp-rc로 변경해 보자.
```
$ kubectl label pod myapp-rc-lwwbz app=myapp-rc --overwrite

pod/myapp-rc-lwwbz labeled
```

파드의 목록을 확인해보자.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS        RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running       0          10m     app=myapp-rc
myapp-rc-79hgq   1/1     Running       0          10m     app=myapp-rc
myapp-rc-lwwbz   1/1     Running       0          5m55s   app=myapp-rc,env=dev
myapp-rc-rrwhf   1/1     Terminating   0          79s     app=myapp-rc
```
myapp-rc-lwwbz 파드의 레이블이 원래대로 app=myapp-rc로 변경되었기 때문에, myapp-rc 레플리케이션 컨트롤러의 관리 파드로 지정되었다. 그러나 레플리케이션 컨트롤러는 관리할 파드가 네 개가 되었으므로, 파드 하나를 삭제한다.

삭제되는 파드의 기준은 가장 최근(AGE)에 생성된 파드인 myapp-rc-rrwhf 파드가 대상이다.

위 명령의 결과에는 종료 중(Terminating)으로 나오지만, 시간에 따라 이미 종료되어 삭제되었을 수 있다.

### 6) 파드의 수평 스케일링
레플리케이션 컨트롤러의 복제본 수만큼 파드를 만들고 관리한다는 것을 확인하였다. 그러면 복제본 수를 변경하여 수평 스케일링을 해보자.

#### (1) 명령을 이용한 스케일링
앞서 살펴보았듯이 kubectl scale 명령으로 스케일링이 가능하다. 복제본 개수를 4로 스케일링해보자.

```
$ kubectl scale replicationcontroller myapp-rc --replicas=4

replicationcontroller/myapp-rc scaled
```

mynapp-rc 레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME       DESIRED   CURRENT   READY   AGE
myapp-rc   4         4         4       12m
```
모든 상태가 네 개로 구성되었다. 시간에 따라 준비된 파드가 세 개일 수도 있다. 이는 파드를 만드는 중이다.

파드의 목록을 살펴보자.
```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rc-6vjrq   1/1     Running   0          12m
myapp-rc-79hgq   1/1     Running   0          12m
myapp-rc-lwwbz   1/1     Running   0          7m19s
myapp-rc-xrjzz   1/1     Running   0          6s
```
myapp-rc-xrjzz 파드가 추가로 생성되었다. 이는 kubectl describe 명령의 결과 중 Events 필드를 확인해도 어떠한 파드가 생성되었는지 확인할 수 있다.

```
$ kubectl describe replicationcontrollers myapp-rc

Name:         myapp-rc
...
Events:
...
  Normal  SuccessfulDelete  2m12s  replication-controller  Deleted pod: myapp-rc-rrwhf
  Normal  SuccessfulCreate  53s    replication-controller  Created pod: myapp-rc-xrjzz
```

#### (2) 오브젝트 수정을 이용한 스케일링
kubectl edit 명령으로 온라인 상태의 YAML 내용을 수정할 수 있다. 기본적으로 vi 에디터로 수정이 가능하다. 복제본 개수를 하나 더 추가해 보자.

```
$ kubectl edit replicationcontrollers myapp-rc

...
spec:
  replicas: 5
...

replicationcontroller/myapp-rc edited
```
spec.replicas 필드를 찾아서 값을 변경한다.

> 참고  
> kubectl edit 명령은 스케일링 이외 일부 필드를 온라인 상태에서 변경 가능하다. 변경한 내용이 문법 오류일 경우 저장이 되지 않고, 변경사항이 반영이 되지도 않는다. 저장할 때 출력에 "XXX edited"라고 나오는지 반드시 확인한다.

> 참고  
> kubectl edit 명령은 KUBE_EDITOR 환경 변수에 에디터를 지정하여 vi/vim 이외의 원하는 에디터로 수정할 수 있다.

myapp-rc 레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME       DESIRED   CURRENT   READY   AGE
myapp-rc   5         5         5       14m
```

파드의 목록을 살펴보자.
```
$ kubectl get pods --show-labels

NAME             READY   STATUS    RESTARTS   AGE     LABELS
myapp-rc-6vjrq   1/1     Running   0          15m     app=myapp-rc
myapp-rc-79hgq   1/1     Running   0          15m     app=myapp-rc
myapp-rc-kcxxg   1/1     Running   0          77s     app=myapp-rc
myapp-rc-lwwbz   1/1     Running   0          10m     app=myapp-rc,env=dev
myapp-rc-xrjzz   1/1     Running   0          3m10s   app=myapp-rc
```
myapp-rc-xrjzz 파드가 생성되었으며, 총 다섯 개의 파드가 존재한다.

#### (3) 오브젝트 파일 수정을 이용한 스케일링
앞서 mynapp-rc.yml 오브젝트 파일을 생성했었다. 현재 파드는 스케일링을 통해 다섯 개가 구성되었다. 그러나 YAML 파일을 수정한 것은 아니기 때문에 YAML 파일 내의 복제본 수는 세 개로 정의되어 있다.

YAML 파일을 이용한 스케일링 방법은, YAML 파일을 적절하게 변경한 후, kubectl replace 명령을 사용하는 것이다. 이는 기존 생성된 오브젝트의 구성과 YAML 파일의 구성 정보를 비교한 후 변경된 사항에 대해서만 반영을 하게 된다.

```
$ kubectl replace -f myapp-rc.yaml

replicationcontroller/myapp-rc replaced
```
파일을 변경하지 않고(복제본: 3) 실행했다면, 당연하겠지만 복제본은 세 개가 되며, 다섯 개 중 두 개는 삭제될 것이다.

레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

NAME       DESIRED   CURRENT   READY   AGE
myapp-rc   3         3         3       15m
```

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rc-6vjrq   1/1     Running   0          16m
myapp-rc-79hgq   1/1     Running   0          16m
myapp-rc-lwwbz   1/1     Running   0          11m
```
세 개의 파드만 남았으며, 또는 두 개의 파드가 종료 중인 것을 확인할 수 있다.

### 7) 레플리케이션 컨트롤러 및 파드 삭제
앞서 살펴보았듯이 컨트롤러에 의해 관리되는 파드들은 파드 자체를 삭제하더라도 복제본 조건에 따라 다시 생성하게 된다. 그래서 파드를 삭제하면 안 되고, 컨트롤러를 직접 삭제해야 컨트롤러에 의해 관리되는 파드도 같이 삭제가 된다.

컨트롤러 삭제 명령은 다음과 같다.
```
$ kubectl delete replicationcontrollers myapp-rc

replicationcontroller "myapp-rc" deleted
```

레플리케이션 컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicationcontrollers

No resources found in default namespace.
```
해당 리소스는 이미 삭제되었기 때문에 존재하지 않는다.

파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME             READY   STATUS        RESTARTS   AGE
myapp-rc-6vjrq   0/1     Terminating   0          16m
myapp-rc-79hgq   0/1     Terminating   0          16m
```
컨트롤러를 삭제한지 얼마 되지 않았기 때문에 파드가 종료 중이다. 시간이 지나면 파드 또한 목록에 보이지 않을 것이다.
