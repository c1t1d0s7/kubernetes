## 6. 동적 볼륨 프로비저닝
정적 볼륨은 관리자가 외부 스토리지와 연결된 PV 리소스를 제공하고, 개발자는 PVC 리소스를 생성해 원하는 PV를 연결해서 파드에 볼륨을 제공할 수 있었다.

매번 개발자가 볼륨을 요청한다면, 그때 마다 적절한 PV를 매번 제공해 줘야 하기 때문에 귀찮은 작업이 아닐 수 없다. 그래서 관리자는 스토리지 기능 및 분류 정보를 담고 있는 스토리지 클래스(StorageClass) 리소스만 정의해 놓고, 개발자가 PVC를 요청하면 스토리지 클래스 리소스에 의해 PV가 자동으로 프로비저닝되어 사용 할수 있도록 동적 프로비저닝 기능을 제공한다.

> 참고  
> 온프레미스(On-Premise)의 베어메탈 또는 가상머신에 배포하거나, 퍼블릭 클라우드의 관리형 쿠버네티스가 아닌 가상머신에 직접 설치한 경우.  
> [Rook](https://rook.io/)  
> 오픈소스를 이용해 쿠버네티스 클러스터 네이티브 스토리지를 구현할 수 있다.  

### 1) 스토리지 클래스
쿠버네티스의 스토리지 클래스 리소스는 스토리지의 클래스(분류, 정책, 종류)에 대한 정보를 정의할 수 있는 리소스다. 쿠버네티스는 자체적으로 클래스에 무엇을 정의해야 하는지에 대한 강제성은 없다. 필요한 스토리지의 여러 정보를 정의할 수 있다.

기본적으로 퍼블릭 클라우드는 기본 스토리지를 하나이상 제공하고 있다.

#### (1) 스토리지 클래스 리소스 정의
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/XXX
parameters:
  XXX:
```
일반적인 스토리지 클래스 리소스의 필드이다.
- storageclass.provisioner 필드
  * 프로비저닝 할 스토리지 PV 
- storageclass.parameters 필드
  * 스토리지의 정보, 분류, 정책, 종류 등 정의
  * 스토리지에 따라 항목은 다름

> 참고: 프로비저닝 할 스토리지 PV  
> https://kubernetes.io/docs/concepts/storage/storage-classes/#provisioner

#### (2) 스토리지 클래스 리소스 정의 예제

> AWS EBS 스토리지 클래스
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: slow
provisioner: kubernetes.io/aws-ebs
parameters:
  type: io1
  iopsPerGB: "10"
  fsType: ext4
```

> AzureDisk 스토리지 클래스
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: slow
provisioner: kubernetes.io/azure-disk
parameters:
  skuName: Standard_LRS
  location: eastus
  storageAccount: azure_storage_account_name
```

> GCP PD 스토리지 클래스
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: slow
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-standard
  replication-type: none
```

### 2) 동적 볼륨을 위한 스토리지 클래스 리소스 확인
Ceph 스토리지의 스토리지 클래스를 확인해보자.
```
$ kubectl get storageclasses.storage.k8s.io

NAME              PROVISIONER                     AGE
csi-cephfs        rook-ceph.cephfs.csi.ceph.com   9s
rook-ceph-block   rook-ceph.rbd.csi.ceph.com      19m
```
- csi-cephfs: CephFS 파일시스템
  * accessModes: RWX,RWO,ROX
- rook-ceph-clock: Ceph RBD 블록
  * accessModes: RWO,ROX

rook-ceph-block 스토리지 클래스의 상세 정보를 확인해보자.
```
$ kubectl describe storageclass rook-ceph-block

Name:                  rook-ceph-block
IsDefaultClass:        No
Annotations:           <none>
Provisioner:           rook-ceph.rbd.csi.ceph.com
Parameters:            clusterID=rook-ceph,csi.storage.k8s.io/...
AllowVolumeExpansion:  True
MountOptions:          <none>
ReclaimPolicy:         Delete
VolumeBindingMode:     Immediate
Events:                <none>
```

csi-cephfs 스토리지 클래스의 상세 정보를 확인해보자.
```
$ kubectl describe storageclasses.storage.k8s.io csi-cephfs

Name:                  csi-cephfs
IsDefaultClass:        No
Annotations:           <none>
Provisioner:           rook-ceph.cephfs.csi.ceph.com
Parameters:            clusterID=rook-ceph,csi.storage.k8s.io/...
AllowVolumeExpansion:  True
MountOptions:          <none>
ReclaimPolicy:         Delete
VolumeBindingMode:     Immediate
Events:                <none>
```

- storageclass.provisioner: Ceph RBD 블록 스토리지 정의
- storageclass.parameters: PV 리소스를 프로비저닝하기 위한 정보 정의

### 3) 동적 볼륨을 위한 PVC 리소스 생성
PVC 리소스 정의 파일이다.
> mynapp-pvc-dynamic.yml
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mynapp-pvc-dynamic
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 1Gi
  storageClassName: csi-cephfs
```
PV 리소스 프로비저닝에 사용할 스토리지 클래스는 csi-cephfs 이고, 요청하는 크기는 1Gi, 접근 모드는 RWX 이다.

PVC 리소스를 생성 해보자.
```
$ kubectl create -f  mynapp-pvc-dynamic.yml

persistentvolumeclaim/mynapp-pvc-dynamic created
```

생성된 PVC 리소스를 확인해보자.
```
$ kubectl get persistentvolumeclaims

NAME                 STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS   AGE
mynapp-pvc-dynamic   Bound    pvc-336294c4-baee-4e2c-ace2-5db6f376f80c   1Gi        RWX            csi-cephfs     6s
```
PV 프로비저닝에 사용한 스토리지 클래스는 csi-cephfs 이고, 상태는 이미 연결된 상태이다. PVC 리소스는 지정한 스토리지 클래스 리소스에 의해 이미 PV 리소스가 프로비저닝 되었고 연결도 되었다는 것을 예상할 수 있다.

PV 리소스를 확인해보자.
```
$ kubectl get persistentvolume

NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                        STORAGECLASS   REASON   AGE
pvc-336294c4-baee-4e2c-ace2-5db6f376f80c   1Gi        RWX            Delete           Bound    default/mynapp-pvc-dynamic   csi-cephfs              7s
```
역시 mynapp-pvc-dynamic PVC 리소스에 요청 되었다는 것을 알 수 있다.

### 4) 동적 볼륨 사용을 위한 레플리카셋 생성 및 확인
mynapp-pvc-dynamic PVC를 볼륨으로 사용하는 레플리카셋 컨틀롤러다.

> mynapp-rs-dynamic.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-dynamic
spec:
  replicas: 2
  selector:
    matchLabels:
      app: mynapp-rs-dynamic
  template:
    metadata:
      labels:
        app: mynapp-rs-dynamic
    spec:
      containers:
      - name: web-server
        image: nginx:alpine
        volumeMounts:
        - name: web-content
          mountPath: /usr/share/nginx/html
        ports:
        - containerPort: 80
      volumes:
      - name: web-content
        persistentVolumeClaim:
          claimName: mynapp-pvc-dynamic
```

레플리카셋 컨트롤러를 생성하자.
```
$ kubectl create -f mynapp-rs-dynamic.yml

replicaset.apps/mynapp-rs-dynamic created
```

컨트롤러 상태를 확인해보자.
```
$ kubectl get replicasets.apps

NAME                DESIRED   CURRENT   READY   AGE
mynapp-rs-dynamic   2         2         2       87s
```

파드의 상태를 확인해보자.
```
$ kubectl get pods

NAME                      READY   STATUS    RESTARTS   AGE
mynapp-rs-dynamic-n424t   1/1     Running   0          92s
mynapp-rs-dynamic-zcl94   1/1     Running   0          92s
```
이상 없이 동작 중이다.

PVC 리소스의 상세 정보를 확인하여 마운트 정보를 확인하자.
```
$ kubectl describe persistentvolumeclaims mynapp-pvc-dynamic
Name:          mynapp-pvc-dynamic
...
VolumeMode:    Filesystem
Mounted By:    mynapp-rs-dynamic-n424t
               mynapp-rs-dynamic-zcl94
...
```
레플리카셋의 파드가 마운트된 것을 확인할 수 있다.

### 5) 기본 스토리지 클래스
PVC 리소스 정의시 PV 프로비저닝에 사용할 스토리지 클래스 리소스를 직접 정의할 수도 있지만, 따로 사용할 스토리지 클래스를 지정하지 않더라도, 기본으로 적용될 스토리지 클래스를 구성할 수 있다.

csi-cephfs 스토리지 클래스를 기본 스토리지 클래스로 설정 해보자.
```
$ kubectl patch storageclasses.storage.k8s.io csi-cephfs \
-p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'

storageclass.storage.k8s.io/csi-cephfs patched
```

위 명령의 의미는 오브젝트의 어노테이션에 "storageclass.kubernetes.io/is-default-class: true"라는 어노테이션을 구성하는 명령이다.

스토리지 클래스 리소스의 상세 정보를 확인해보자.
```
$ kubectl describe storageclasses.storage.k8s.io csi-cephfs

...
Annotations:           storageclass.kubernetes.io/is-default-class=true
...
```
어노테이션이 할당된 것을 확인할 수 있다. 마스터의 API 서비는 해당 어노테이션을 참조하여 기본 스토리지 클래스로 설정한다.

마지막으로 스토리지 클래스 목록을 확인하자.
```
$ kubectl get storageclasses.storage.k8s.io
NAME                   PROVISIONER                     AGE

csi-cephfs (default)   rook-ceph.cephfs.csi.ceph.com   24m
rook-ceph-block        rook-ceph.rbd.csi.ceph.com      44m
```
스토리지 클래스 이름 옆에 (default) 라는 표시가 지정되어 있다.

### 6) 리소스 삭제
```
$ kubectl delete -f mynapp-rs-dynamic.yml -f mynapp-pvc-dynamic.yml

replicaset.apps "mynapp-rs-dynamic" deleted
persistentvolumeclaim "mynapp-pvc-dynamic" deleted
```
