## 5. 정적 볼륨 프로비저닝
정적 볼륨은 PV 및 PVC 리소스를 직접 생성하고 사용하는 방식이다.

### 1) NFS 스토리지 구성

#### (1) NFS 서버 구성(kube-master1)
이 작업은 NFS 서버로 구성할 kube-master1 호스트에서 작업한다.

NFS 서버 패키지를 설치한다.
```
$ sudo apt install -y nfs-kernel-server
```

NFS 공유 디렉토리를 설정한다.
```
$ sudo mkdir /nfs-volume
```

NFS 내보내기를 설정한다.
```
$ echo "/nfs-volume *(rw,sync,no_subtree_check)" | sudo tee /etc/exports
```

NFS 공유 디렉토리에 파드가 사용할 index.html 파일을 생성한다.
```
$ echo "hello NFS volume" | sudo tee /nfs-volume/index.html
```

NFS 공유 디렉토리에 적절한 권한을 부여한다.
```
$ sudo chown -R nobody:nogroup /nfs-volume
$ sudo chmod 777 /nfs-volume
```

NFS 서버 서비스를 재시작 한다.
```
$ sudo systemctl restart nfs-kernel-server
```

NFS 서버 서비스 상태를 확인한다.
```
$ sudo systemctl status nfs-kernel-server

● nfs-server.service - NFS server and services
   Loaded: loaded (/lib/systemd/system/nfs-server.service; enabled; vendor preset: enabled)
   Active: active (exited) since Wed 2020-03-18 02:59:22 UTC; 28min ago
...
```

NFS 서비스 포트인 2049번 포트를 방화벽에서 연다.
```
$ sudo iptables -A INPUT -p tcp --dport 2049 -j ACCEPT
$ sudo iptables -A INPUT -p udp --dport 2049 -j ACCEPT
```

#### (2) NFS 클라우드 구성(kube-node1, kube-node2, kube-node3)
NFS 클라이언트로 설정할 kube-node1, kube-node2, kube-node3 노드에 NFS 클라이언트 패키지를 설치한다.

NFS 클라이언트 패키지인 nfs-common을 설치한다.
```
$ sudo apt install -y nfs-common
```
또는 Ansible을 이용하여 한번에 모든 노드에 설치 할수 있다.
```
$ ansible kube-node -i ~/kubespray/inventory/mycluster/inventory.ini -m apt -a 'name=nfs-common' --become
```

### 2) PV 생성 및 확인
다음은 PV 리소스의 정의 파일이다.

> mynapp-pv-nfs.yml 파일
```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: mynapp-pv-nfs
spec:
  capacity:
    storage: 5Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  nfs:
    path: /nfs-volume
    server: 192.168.56.11 # kube-master1
```

PV 리소스에서 정의할 주요 필드는 다음과 같다.
- .spec.capacity.storage 필드: 스토리지 용량 지정
- .spec.accessMode 필드: 접근방식 관련 옵션 (스토리지에 따라 지원 옵션 다름)
  * ReadWriteOnce: 하나의 파드만 읽기/쓰기 가능
  * ReadWriteMany: 여러 파드 읽기/쓰기 가능
  * ReadOnlyMany: 여러 파드 읽기만 가능
- .spec.persistentVolumeReclaimPolicy 필드: 회수 정책
  * Retain
  * Delete
  * Recycle 
- .spec.nfs 필드: NFS 볼륨
  * 앞서 살펴본 emptyDir, hostPath와 같은 볼륨을 지정
  * 볼륨에 따른 하위 필드는 달라진다.

> 참고: 볼륨 형식에 따른 접근 모드(accessMode) 지원 표  
> https://kubernetes.io/docs/concepts/storage/persistent-volumes/#access-modes

PV 리소스를 생성한다.
```
$ kubectl create -f mynapp-pv-nfs.yml

persistentvolume/mynapp-pv-nfs created
```

PV 리소스의 상태를 확인한다.
```
$ kubectl get persistentvolumes

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      CLAIM   STORAGECLASS   REASON   AGE
mynapp-pv-nfs   5Gi        RWX            Retain           Available
```
현재 mynapp-pv-nfs 볼륨의 접근 모드는 RWX(ReadWriteMany)이며, 회수 정책은 유지(Retain), 상태는 사용가능한(Available) 상태로 보고하고 있다.

표시되는 PV 리소스의 네 가지 상태(STATUS):
- Available: 다른 PVC에 연결되지 않은 상태, 사용가능 상태
- Bound: 특정 PVC에 연결됨
- Released: 연결되었던 PVC가 해제 되었으며, 리소스를 회수하지 않은 상태
- Failed: 회수 실패

### 3) PVC 생성 및 확인
다음은 PVC 리소스 정의 파일이다.

> mynapp-pvc-nfs.yml 파일
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mynapp-pvc-nfs
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 5Gi
  volumeName: mynapp-pv-nfs
```

PVC 리소스의 주요 필드는 다음과 같다.
- .spec.accessModes 필드: PV의 필드와 같다.
- .spec.requests.storage 필드: PV에 요청하는 크기
- .spec.volumeName 필드: PV 리소스 이름을 지정

PVC 리소스를 생성한다.
```
$ kubectl create -f mynapp-pvc-nfs.yml

persistentvolumeclaim/mynapp-pvc-nfs created
```

PVC 리소스를 확인한다.
```
$ kubectl get persistentvolumeclaims

NAME             STATUS   VOLUME          CAPACITY   ACCESS MODES   STORAGECLASS   AGE
mynapp-pvc-nfs   Bound    mynapp-pv-nfs   5Gi        RWX
```
상태를 확인하면 mynapp-pv-nfs PV 리소스에 연결된(Bound) 것을 확인 할수 있다. 혹시나 상태가 보류(Pending) 상태이면 연결중 또는 적절한 PV 리소스를 찾지 못한 경우다. 이런경우 kubectl describe 명령으로 PVC 리소스의 이벤트에 원인을 찾아서 해결한다.

PV 리소스의 상태도 확인 해보자.
```
$ kubectl get persistentvolumes

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                    STORAGECLASS   REASON   AGE
mynapp-pv-nfs   5Gi        RWX            Retain           Bound    default/mynapp-pvc-nfs
```
상태가 연결됨(Bound) 상태이고 요청(CLAIM) 필드에 mynapp-pvc-nfs PVC 리소스가 연결된 것을 확인 할수 있다.

### 4) 레플리카셋 컨트롤러 생성 및 확인
다음은 NFS PVC 볼륨을 사용할 레플리카셋 리소스 정의 파일이다.
> mynapp-rs-nfs.yml 파일
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-nfs
spec:
  replicas: 2
  selector:
    matchLabels:
      app: mynapp-rs-nfs
  template:
    metadata:
      labels:
        app: mynapp-rs-nfs
    spec:
      containers:
      - name: mynapp
        image: nginx
        volumeMounts:
        - name: nfs-share
          mountPath: /usr/share/nginx/html
        ports:
        - containerPort: 80
      volumes:
      - name: nfs-share
        persistentVolumeClaim:
          claimName: mynapp-pvc-nfs
```
사용할 볼륨을 PVC 볼륨으로 지정한다.

- .spec.template.spec.volumes.persistentVolumeClaim 필드: PVC 볼륨
  * claimName: PVC 리소스의 이름

레플리카셋 리소스를 생성한다.
```
$ kubectl create -f mynapp-rs-nfs.yml

replicaset.apps/mynapp-rs-nfs created
```

레플리카셋 리소스를 확인한다.
```
$ kubectl get replicasets.apps

NAME            DESIRED   CURRENT   READY   AGE
mynapp-rs-nfs   2         2         2       42s
```
두 개의 파드 모두 동작중이다.

파드의 목록도 확인 해보자.
```
$ kubectl get pods

NAME                  READY   STATUS    RESTARTS   AGE
mynapp-rs-nfs-28wfv   1/1     Running   0          45s
mynapp-rs-nfs-g6wnp   1/1     Running   0          45s
```
볼륨에 연결하기 까지 시간이 조금 걸릴수 있다.

### 5) PV 및 PVC 상세 정보 확인
PV 리소스의 상세 정보를 확인 해보자.
```
$ kubectl describe persistentvolumes mynapp-pv-nfs

Name:            mynapp-pv-nfs
...
Status:          Bound
Claim:           default/mynapp-pvc-nfs
Reclaim Policy:  Retain
Access Modes:    RWX
VolumeMode:      Filesystem
Capacity:        5Gi
...
Source:
    Type:      NFS (an NFS mount that lasts the lifetime of a pod)
    Server:    192.168.56.11
    Path:      /nfs-volume
    ReadOnly:  false
Events:        <none>
```
PVC 요청(Claim) 정보 및 연결된 스토리지(Source)의 정보를 확인 할수 있다.

PVC 리소스의 상세 정보를 확인 해보자.
```
$ kubectl describe persistentvolumeclaims mynapp-pvc-nfs
Name:          mynapp-pvc-nfs
...
Status:        Bound
Volume:        mynapp-pv-nfs
...
Capacity:      5Gi
Access Modes:  RWX
VolumeMode:    Filesystem
Mounted By:    mynapp-rs-nfs-28wfv
               mynapp-rs-nfs-g6wnp
...
```
어떤 파드가 마운드해서(Mounted By) 사용하고 있는지 확인 할수 있다.

### 6) 서비스 생성 및 확인
다름은 파드 및 볼륨의 데이터를 확인하기 위한 서비스 리소스다.

> mynapp-svc-nfs.yml 파일
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-nfs
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 80
  selector:
    app: mynapp-rs-nfs
```
레플리카셋 컨트롤러는 nginx 이미지를 사용하고 있고, 80번 포트를 사용한다.

서비스 리소스를 생성한다.
```
$ kubectl create -f mynapp-svc-nfs.yml

service/mynapp-svc-nfs created
```

```
$ kubectl run nettool -it --image=c1t1d0s7/network-multitool --generator=run-pod/v1 --rm=true bash

bash-5.0# curl http://mynapp-svc-nfs
hello NFS volume

bash-5.0# curl http://mynapp-svc-nfs
hello NFS volume

bash-5.0# exit

Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```

### 7) 볼륨 상태 전환 확인
레플리카셋을 삭제하여 파드를 중지한다.
```
$ kubectl delete replicasets.apps mynapp-rs-nfs

replicaset.apps "mynapp-rs-nfs" deleted
```

PVC 상세 정보를 확인하여 마운트 상태를 확인한다.
```
$ kubectl describe persistentvolumeclaims mynapp-pvc-nfs
...
Mounted By:    <none>
...
```

레플리카셋을 다시 생성 해보자.
```
kubectl create -f mynapp-rs-nfs.yml
```

PVC 상세 정보를 확인하여 다시 마운트 상태를 확인 해보자.
```
$ kubectl describe persistentvolumeclaims mynapp-pvc-nfs
...
Mounted By:    mynapp-rs-nfs-rdfvz
               mynapp-rs-nfs-zrxsg
...
```
새로 생성된 파드가 이상없이 마운트한 것을 확인할 수 있다.

레플리카셋 컨트롤러를 다시 삭제한다.
```
$ kubectl delete replicasets.apps mynapp-rs-nfs

replicaset.apps "mynapp-rs-nfs" deleted
```

PVC 리소스도 삭제한다.
```
$ kubectl delete persistentvolumeclaims mynapp-pvc-nfs

persistentvolumeclaim "mynapp-pvc-nfs" deleted
```

PVC 리소스가 삭제된 후 PV 리소스의 상태를 확인 해보자.
```
$ kubectl get persistentvolumes

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS     CLAIM                    STORAGECLASS   REASON   AGE
mynapp-pv-nfs   5Gi        RWX            Retain           Released   default/mynapp-pvc-nfs
```
상태가 연결이 해제된(Released) 상태인 것을 확인 할수 있다.

다시 PVC 리소스를 생성 해보자.
```
kubectl create -f mynapp-pvc-nfs.yml
```

생성된 PVC 리소스의 상태를 확인 해보자.
```
$ kubectl get persistentvolumeclaims

NAME             STATUS    VOLUME          CAPACITY   ACCESS MODES   STORAGECLASS   AGE
mynapp-pvc-nfs   Pending   mynapp-pv-nfs   0                                        84s
```
PV 리소스가 해제된(Released) 상태이기 때문에 PVC가 연결할 적절한 PV를 찾지 못했기 때문에 보류(Pending)로 보고하고 있다. 또는 볼륨에 따라서 실패(Failed)로 보고 할수도 있다.

PV 리소스를 삭제한다.
```
$ kubectl delete persistentvolume mynapp-pv-nfs

persistentvolume "mynapp-pv-nfs" deleted
```

PV 리소스를 다시 생성하자.
```
$ kubectl create -f mynapp-pv-nfs.yml

persistentvolume/mynapp-pv-nfs created
```

PV 및 PVC 리소스를 상태를 다시 확인 해보자.
```
$ kubectl get persistentvolume

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                    STORAGECLASS   REASON   AGE
mynapp-pv-nfs   5Gi        RWX            Retain           Bound    default/mynapp-pvc-nfs
```

```
$ kubectl get persistentvolumeclaims

NAME             STATUS   VOLUME          CAPACITY   ACCESS MODES   STORAGECLASS   AGE
mynapp-pvc-nfs   Bound    mynapp-pv-nfs   5Gi        RWX                           7m4s
```
다시 이상 없이 사용 할수 있다.

### 8) 컨트롤러, 서비스, PV, PVC 리소스 삭제
```
$ kubectl delete -f mynapp-svc-nfs.yml

service "mynapp-svc-nfs" deleted
```

```
$ kubectl delete -f mynapp-pvc-nfs.yml -f mynapp-pv-nfs.yml

persistentvolumeclaim "mynapp-pvc-nfs" deleted
persistentvolume "mynapp-pv-nfs" deleted
```
