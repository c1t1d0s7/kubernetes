## 5. 정적 볼륨 프로비저닝
정적 볼륨은 PV 및 PVC 리소스를 직접 생성하고 사용하는 방식이다.

### 1) NFS 스토리지 구성

#### (1) NFS 서버 구성(kube-master1)
이 작업은 NFS 서버로 구성할 kube-master1 호스트에서 작업한다.

NFS 서버 패키지를 설치한다.
```
$ sudo apt install -y nfs-kernel-server
```

NFS 공유 디렉터리를 설정한다.
```
$ sudo mkdir /nfs-volume
```

NFS 내보내기를 설정한다.
```
$ echo "/nfs-volume *(rw,sync,no_subtree_check)" | sudo tee /etc/exports
```

NFS 공유 디렉터리에 파드가 사용할 index.html 파일을 생성한다.
```
$ echo "hello NFS volume" | sudo tee /nfs-volume/index.html
```

NFS 공유 디렉터리에 적절한 권한을 부여한다.
```
$ sudo chown -R nobody:nogroup /nfs-volume
$ sudo chmod 777 /nfs-volume
```

NFS 서버 서비스를 재시작 한다.
```
$ sudo systemctl restart nfs-kernel-server
```

NFS 서버 서비스 상태를 확인한다.
```
$ sudo systemctl status nfs-kernel-server

● nfs-server.service - NFS server and services
   Loaded: loaded (/lib/systemd/system/nfs-server.service; enabled; vendor preset: enabled)
   Active: active (exited) since Wed 2020-03-18 02:59:22 UTC; 28min ago
...
```

NFS 서비스 포트인 2049번 포트를 방화벽에서 연다.
```
$ sudo iptables -A INPUT -p tcp --dport 2049 -j ACCEPT
$ sudo iptables -A INPUT -p udp --dport 2049 -j ACCEPT
```

#### (2) NFS 클라이언트 구성(kube-node1, kube-node2, kube-node3)
NFS 클라이언트로 설정할 kube-node1, kube-node2, kube-node3 노드에 NFS 클라이언트 패키지를 설치한다.

NFS 클라이언트 패키지인 nfs-common을 설치한다.
```
$ sudo apt install -y nfs-common
```
또는 Ansible을 이용하여 한 번에 모든 노드에 설치할 수 있다.
```
$ ansible kube-node -i ~/kubespray/inventory/mycluster/inventory.ini -m apt -a 'name=nfs-common' --become
```

### 2) PV 생성 및 확인
다음은 PV 리소스의 정의 파일이다.

> myapp-pv-nfs.yaml
```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: myapp-pv-nfs
spec:
  capacity:
    storage: 5Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  nfs:
    path: /nfs-volume
    server: 192.168.56.11
```

PV 리소스에서 정의할 주요 필드는 다음과 같다.
- .spec.capacity.storage: 스토리지 용량 지정
- .spec.accessMode: 접근 방식 관련 옵션 (스토리지에 따라 지원 옵션 다름)
  * ReadWriteOnce: 하나의 파드만 읽기/쓰기 가능
  * ReadWriteMany: 여러 파드 읽기/쓰기 가능
  * ReadOnlyMany: 여러 파드 읽기만 가능
- .spec.persistentVolumeReclaimPolicy: 회수 정책
  * Retain
  * Delete
  * Recycle 
- .spec.nfs: NFS 볼륨
  * 앞서 살펴본 emptyDir, hostPath와 같은 볼륨을 지정
  * 볼륨에 따른 하위 필드는 달라진다.

> 참고: 볼륨 형식에 따른 접근 모드(accessMode) 지원 표  
> https://kubernetes.io/docs/concepts/storage/persistent-volumes/#access-modes

PV 리소스를 생성한다.
```
$ kubectl create -f myapp-pv-nfs.yaml

persistentvolume/myapp-pv-nfs created
```

PV 리소스의 상태를 확인한다.
```
$ kubectl get persistentvolumes

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS      ...
mynapp-pv-nfs   5Gi        RWX            Retain           Available   ...
```
현재 mynapp-pv-nfs 볼륨의 접근 모드는 RWX(ReadWriteMany)이며, 회수 정책은 유지(Retain), 상태는 사용 가능한(Available) 상태로 보고하고 있다.

표시되는 PV 리소스의 네 가지 상태(STATUS):
- Available: 다른 PVC에 연결되지 않은 상태, 사용 가능 상태
- Bound: 특정 PVC에 연결됨
- Released: 연결되었던 PVC가 해제되었으며, 리소스를 회수하지 않은 상태
- Failed: 회수 실패

### 3) PVC 생성 및 확인
다음은 PVC 리소스 정의 파일이다.

> myapp-pvc-nfs.yaml
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: myapp-pvc-nfs
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 5Gi
  volumeName: myapp-pv-nfs
  storageClassName: ''
```

PVC 리소스의 주요 필드는 다음과 같다.
- .spec.accessModes: PV의 필드와 같다.
- .spec.requests.storage: PV에 요청하는 크기
- .spec.volumeName: PV 리소스 이름을 지정
- .spec.storageClassName: 스토리지 클래스 이름 (지금 지정하지 않음)

PVC 리소스를 생성한다.
```
$ kubectl create -f myapp-pvc-nfs.yaml

persistentvolumeclaim/myapp-pvc-nfs created
```

PVC 리소스를 확인한다.
```
$ kubectl get persistentvolumeclaims

NAME             STATUS   VOLUME          CAPACITY   ACCESS MODES   ...
mynapp-pvc-nfs   Bound    mynapp-pv-nfs   5Gi        RWX            ...
```
상태를 확인하면 mynapp-pv-nfs PV 리소스에 연결된(Bound) 것을 확인할 수 있다. 혹시나 상태가 보류(Pending) 상태이면 연결 중 또는 적절한 PV 리소스를 찾지 못한 경우다. 이런 경우 kubectl describe 명령으로 PVC 리소스의 이벤트에 원인을 찾아서 해결한다.

PV 리소스의 상태도 확인해보자.
```
$ kubectl get persistentvolumes

NAME            CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                   ...
mynapp-pv-nfs   5Gi        RWX            Retain           Bound    default/mynapp-pvc-nfs  ...
```
상태가 연결됨(Bound) 상태이고 요청(CLAIM) 필드에 mynapp-pvc-nfs PVC 리소스가 연결된 것을 확인할 수 있다.

### 4) 레플리카셋 컨트롤러 생성 및 확인
다음은 NFS PVC 볼륨을 사용할 레플리카셋 리소스 정의 파일이다.
> myapp-rs-nfs.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-nfs
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp-rs-nfs
  template:
    metadata:
      labels:
        app: myapp-rs-nfs
    spec:
      containers:
      - name: myapp
        image: nginx
        volumeMounts:
        - name: nfs-share
          mountPath: /usr/share/nginx/html
        ports:
        - containerPort: 80
      volumes:
      - name: nfs-share
        persistentVolumeClaim:
          claimName: myapp-pvc-nfs
```
사용할 볼륨을 PVC 볼륨으로 지정한다.

- .spec.template.spec.volumes.persistentVolumeClaim: PVC 볼륨
  * claimName: PVC 리소스의 이름

레플리카셋 리소스를 생성한다.
```
$ kubectl create -f myapp-rs-nfs.yaml

replicaset.apps/myapp-rs-nfs created
```

레플리카셋 리소스를 확인한다.
```
$ kubectl get replicasets.apps

NAME           DESIRED   CURRENT   READY   AGE
myapp-rs-nfs   2         2         2       12s
```
두 개의 파드 모두 동작 중이다.

파드의 목록도 확인해보자.
```
$ kubectl get pods

NAME                 READY   STATUS    RESTARTS   AGE
myapp-rs-nfs-dk5qv   1/1     Running   0          15s
myapp-rs-nfs-w8q4t   1/1     Running   0          15s
```
볼륨에 연결하기까지 시간이 조금 걸릴 수 있다.

### 5) PV 및 PVC 상세 정보 확인
PV 리소스의 상세 정보를 확인해보자.
```
$ kubectl describe persistentvolume myapp-pv-nfs

Name:            myapp-pv-nfs
...
Source:
    Type:      NFS (an NFS mount that lasts the lifetime of a pod)
    Server:    192.168.56.11
    Path:      /nfs-volume
    ReadOnly:  false
Events:        <none>
```
PVC 요청(Claim) 정보 및 연결된 스토리지(Source)의 정보를 확인할 수 있다.

PVC 리소스의 상세 정보를 확인해보자.
```
$ kubectl describe persistentvolumeclaims myapp-pvc-nfs

Name:          myapp-pvc-nfs
...
Mounted By:    myapp-rs-nfs-dk5qv
               myapp-rs-nfs-w8q4t
...
```
어떤 파드가 볼륨을 마운트 했는지 확인할 수 있다.

### 6) 서비스 생성 및 확인
다름은 파드 및 볼륨의 데이터를 확인하기 위한 서비스 리소스다.

> myapp-svc-nfs.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-nfs
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 80
  selector:
    app: myapp-rs-nfs
```
레플리카셋 컨트롤러는 nginx 이미지를 사용하고 있고, 80번 포트를 사용한다.

서비스 리소스를 생성한다.
```
$ kubectl create -f myapp-svc-nfs.yaml

service/myapp-svc-nfs created
```

서비스에 접근해서 컨텐츠를 확인해보자.
```
$ curl http://192.168.56.200

hello NFS volume
```

```
$ curl http://192.168.56.200

hello NFS volume
```

### 7) 볼륨 상태 전환 확인
레플리카셋을 삭제하여 파드를 중지한다.
```
$ kubectl delete replicasets.apps myapp-rs-nfs

replicaset.apps "myapp-rs-nfs" deleted
```

PVC 상세 정보를 확인하여 마운트 상태를 확인한다.
```
$ kubectl describe persistentvolumeclaims myapp-pvc-nfs

Name:          myapp-pvc-nfs
...
Mounted By:    <none>
...
```

레플리카셋을 다시 생성해보자.
```
kubectl create -f myapp-rs-nfs.yaml

replicaset.apps/myapp-rs-nfs created
```

PVC 상세 정보를 확인하여 다시 마운트 상태를 확인해보자.
```
$ kubectl describe persistentvolumeclaims myapp-pvc-nfs

Name:          myapp-pvc-nfs
...
Mounted By:    myapp-rs-nfs-jkshq
               myapp-rs-nfs-wbgpn
...
```
새로 생성된 파드가 이상 없이 마운트 한 것을 확인할 수 있다.

레플리카셋 컨트롤러를 다시 삭제한다.
```
$ kubectl delete replicasets.apps myapp-rs-nfs

replicaset.apps "myapp-rs-nfs" deleted
```

PVC 리소스도 삭제한다.
```
$ kubectl delete persistentvolumeclaims myapp-pvc-nfs

persistentvolumeclaim "myapp-pvc-nfs" deleted
```

PVC 리소스가 삭제된 후 PV 리소스의 상태를 확인해보자.
```
$ kubectl get persistentvolume

NAME           CAPACITY   ... STATUS     CLAIM                  ...
myapp-pv-nfs   5Gi        ... Released   default/myapp-pvc-nfs  ...
```
상태가 연결이 해제된(Released) 상태인 것을 확인할 수 있다.

다시 PVC 리소스를 생성해보자.
```
kubectl create -f myapp-pvc-nfs.yaml

persistentvolumeclaim/myapp-pvc-nfs created
```

생성된 PVC 리소스의 상태를 확인해보자.
```
$ kubectl get persistentvolumeclaims

NAME            STATUS    VOLUME         ...
myapp-pvc-nfs   Pending   myapp-pv-nfs   ...
```
PV 리소스가 해제된(Released) 상태이기 때문에 PVC가 연결할 적절한 PV를 찾지 못했기 때문에 보류(Pending)로 보고하고 있다. 또는 볼륨에 따라서 실패(Failed)로 보고 할 수도 있다.

PV 리소스를 삭제한다.
```
$ kubectl delete persistentvolume myapp-pv-nfs

persistentvolume "myapp-pv-nfs" deleted
```

PV 리소스를 다시 생성하자.
```
$ kubectl create -f myapp-pv-nfs.yaml

persistentvolume/myapp-pv-nfs created
```

PV 및 PVC 리소스를 상태를 다시 확인해보자.
```
$ kubectl get persistentvolume

NAME           CAPACITY   ... STATUS   CLAIM                   ...
myapp-pv-nfs   5Gi        ... Bound    default/myapp-pvc-nfs   ...
```

```
$ kubectl get persistentvolumeclaims

NAME            STATUS   VOLUME         ...
myapp-pvc-nfs   Bound    myapp-pv-nfs   ...
```
다시 이상 없이 사용할 수 있다.

### 8) 컨트롤러, 서비스, PV, PVC 리소스 삭제
```
$ kubectl delete -f myapp-svc-nfs.yaml

service "myapp-svc-nfs" deleted
```

```
$ kubectl delete -f myapp-pvc-nfs.yaml -f myapp-pv-nfs.yaml

persistentvolumeclaim "myapp-pvc-nfs" deleted
persistentvolume "myapp-pv-nfs" deleted
```
