## 3. hostPath 볼륨

### 1) hostPath 볼륨 소개
hostPath 볼륨은 파드가 동작하는 쿠버네티스 클러스터의 노드(호스트)의 로컬 파일시스템의 파일 및 디렉토리를 파드가 사용할 수 있는 볼륨으로 제공해 준다.

같은 노드에 배치된 파드 및 컨테이너에게 공유 디렉토리를 제공할 목적으로 사용될 수 있지만, 다른 노드에 배치된 파드 및 컨테이너에는 데이터를 공유 할 수 없다. 즉, 각 노드에 각각 볼륨이 생성된다.

또한 기존에 A 노드에 작동하고 있던 X 파드가, B 노드로 이동하면 더 이상 데이터를 접근할 수 없게 된다.

쿠버네티스 클러스터 내부에서 사용 용도는 쿠버네티스 클러스터 노드에 로그 파일을 저장하거나 CA 인증서 및 토큰을 파드에 제공하기 위한 용도도 있다.

### 2) hostPath 볼륨 생성
hostPath 볼륨 리소스를 정의하는 방법은 다음과 같다.
```yaml
volumes:
- name: <VOL-NAME>
  hostPath:
    type: <TYPE>
    path: <PATH>
```
- .volumes.hostPath.type: 볼륨의 종류 지정
  * DirectoryOrCreate: 지정된 경로의 디렉토리 사용 (없으면 만들어 줌)
  * Directory: 지정된 경로의 디렉토리 사용 (존재해야 함)
  * FileOrCreate: 지정된 경로의 파일 사용 (없으면 만들어 줌)
  * File: 지정된 경로의 파일 사용 (존재해야 함)
  * Socket: 지정된 경로의 UNIX 소켓 사용 (존재해야 함)
  * CharDevice: 지정된 경로의 캐릭터 장치 사용 (존재해야 함)
  * BlockDevice: 지정된 경로의 블록 장치 사용 (존재해야 함)
- .volumes.hostPath.path: 공유할 볼륨의 호스트 경로 지정

### 3) hostPath 볼륨에 사용할 노드의 디렉토리 준비
kube-node1에 hostPath 볼륨에 사용할 디렉토리를 생성해보자.

마스터에서 kube-node1으로 접속한다.
```
ssh vagrant@kube-node1
```

/web_contents 디렉토리를 생성한다.
```
vagrant@kube-node1:~$ sudo mkdir /web_contents
```

/web_contents 디렉토리에 index.html 파일을 생성한다.
```
vagrant@kube-node1:~$ echo "hello hostPath Volumes" | sudo tee /web_contents/index.html

hello hostPath Volumes
```

kube-node1 노드에서 마스터로 돌아온다.
```
vagrant@kube-node1:~$ exit
logout
Connection to kube-node1 closed.
```

### 4) hostPath 볼륨을 사용하는 레플리카셋 생성
다음은 hostPath 볼륨을 사용하는 레플라카셋 오브젝트 정의 파일이다.
> mynapp-rs-hostpath.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-hp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: mynapp-rs-hp
  template:
    metadata:
      labels:
        app: mynapp-rs-hp
    spec:
      containers:
      - name: web-server
        image: nginx:alpine
        volumeMounts:
        - name: web-content
          mountPath: /usr/share/nginx/html
        ports:
        - containerPort: 80
      volumes:
      - name: web-content
        hostPath:
          type: Directory
          path: /web_contents
```
.spec.template.spec.volumes 필드에 web-content hostPath 볼륨을 지정했으며, web-server 컨테이너가 해당 볼륨을 사용한다.

레플리카셋 컨트롤러에 의해 web-server 파드는 두 개의 복제본을 가진다. 즉, 두 개의 파드에 같은 웹 컨텐츠를 제공해 준다.

hostPath 볼륨을 사용하는 레플리카셋을 위한 서비스 오브젝트다.
> mynapp-svc-hostpath.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-hp
spec:
  ports:
  - port: 80
    targetPort: 80
  selector:
    app: mynapp-rs-hp
```

레플리카셋과 서비스를 생성한다.
```
$ kubectl create -f mynapp-rs-hostpath.yml -f mynapp-svc-hostpath.yml

replicaset.apps/mynapp-rs-hp created
service/mynapp-svc-hp created
```

### 5) hostPath 볼륨을 사용하는 리소스 확인
레플리카셋 컨트롤러와 파드의 상태를 확인한다.
```
$ kubectl get replicasets.apps,pods

NAME                           DESIRED   CURRENT   READY   AGE
replicaset.apps/mynapp-rs-hp   2         2         1       25s

NAME                     READY   STATUS              RESTARTS   AGE
pod/mynapp-rs-hp-gk6z7   0/1     ContainerCreating   0          25s
pod/mynapp-rs-hp-hrn82   1/1     Running             0          25s
```
하나의 파드(mynapp-rs-hp-gk6z7)는 생성되고 있지 않다.

mynapp-svc-hp 서비스와 엔드포인트를 확인한다.
```
$ kubectl get services,endpoints mynapp-svc-hp

NAME                    TYPE        CLUSTER-IP    EXTERNAL-IP   PORT(S)   AGE
service/mynapp-svc-hp   ClusterIP   10.233.17.4   <none>        80/TCP    2m43s

NAME                      ENDPOINTS          AGE
endpoints/mynapp-svc-hp   10.233.101.27:80   2m43s
```
당연히 하나의 파드가 준비되지 않았기 때문에 엔드포인트는 하나만 등록되어 있다.

파드가 배치된 노드를 확인해보자.
```
$ kubectl get pods -o wide
NAME                 READY   STATUS              RESTARTS   AGE     IP              NODE         ...
mynapp-rs-hp-gk6z7   0/1     ContainerCreating   0          3m42s   <none>          kube-node2   ...
mynapp-rs-hp-hrn82   1/1     Running             0          3m42s   10.233.101.27   kube-node1   ...
```
mynapp-rs-hp-gk6z7 파드는 kube-node2에 배치된 것을 확인할 수 있다. 즉, kube-node2는 볼륨을 연결할 수가 없어서 준비되고 있지 않는 것을 추정할 수 있다.

준비되지 않은 mynapp-rs-hp-gk6z7 파드의 정보를 더 자세히 보자.
```
$ kubectl describe pods mynapp-rs-hp-gk6z7

Name:           mynapp-rs-hp-gk6z7
...
Events:
  Type     Reason       Age                   From                 Message
  ----     ------       ----                  ----                 -------
  Normal   Scheduled    <unknown>             default-scheduler    Successfully assigned default/mynapp-rs-hp-gk6z7 to kube-node2
  Warning  FailedMount  3m25s                 kubelet, kube-node2  Unable to attach or mount volumes: unmounted volumes=[web-content], unattached volumes=[default-token-x4tb2 web-content]: timed out waiting for the condition
  Warning  FailedMount  78s (x10 over 5m28s)  kubelet, kube-node2  MountVolume.SetUp failed for volume "web-content" : hostPath type check failed: /web_contents is not a directory
  Warning  FailedMount  68s                   kubelet, kube-node2  Unable to attach or mount volumes: unmounted volumes=[web-content], unattached volumes=[web-content default-token-x4tb2]: timed out waiting for the condition
```
이벤트(Events) 항목을 확인 해보면 볼륨을 마운트 할수 없다고(FailedMount) 보고하고 있다.

### 6) hostPath 볼륨을 사용하는 리소스 변경
레플리카셋 컨트롤러를 삭제하자.
```
$ kubectl delete -f mynapp-rs-hostpath.yml

replicaset.apps "mynapp-rs-hp" deleted
```

레플리카셋 오브젝트를 수정한다.
```
$ vi mynapp-rs-hostpath.yml

...
    spec:
      nodeName: kube-node1
      containers:
...

:wq
```
.spec.template.spec.nodeName 필드를 추가한다. nodeName 필드는 앞서 데몬셋에서 살펴본 .spec.template.spec.nodeSelector 필드와 비슷한 기능으로 nodeSelector는 노드의 레이블을 매핑 시키지만, nodeName 필드는 배치할 노드를 직접 지정할 수 있다.

다시 레플리카셋 컨트롤러를 생성한다.
```
$ kubectl create -f mynapp-rs-hostpath.yml

replicaset.apps/mynapp-rs-hp created
```

파드의 배치 상태를 확인하자.
```
$ kubectl get pods -o wide
NAME                 READY   STATUS    RESTARTS   AGE   IP              NODE         ...
mynapp-rs-hp-d8ppz   1/1     Running   0          8s    10.233.101.29   kube-node1   ...
mynapp-rs-hp-vc4sd   1/1     Running   0          8s    10.233.101.28   kube-node1   ...
```
두 파드 모두 kube-node1에 배치된 것을 확인할 수 있고, 볼륨을 마운트 할수 있기 때문에 준비된 상태로 보고하고 있다.

### 7) hostPath 볼륨을 사용하는 리소스 재확인
두 파드에 index.html 파일의 내용이 같은지 확인해보자.
```
$ kubectl exec mynapp-rs-hp-d8ppz -- cat /usr/share/nginx/html/index.html

hello hostPath Volumes
```
```
$ kubectl exec mynapp-rs-hp-vc4sd -- cat /usr/share/nginx/html/index.html

hello hostPath Volumes
```
같은 볼륨을 마운트하고 있기 때문에 같은 내용이 확인된다.

마지막으로 네트워크 도구를 가진 파드를 생성해 서비스를 확인해보자.
```
$ kubectl run nettool -it --image=c1t1d0s7/network-multitool --generator=run-pod/v1 --rm=true bash

If you don't see a command prompt, try pressing enter.

bash-5.0# curl http://mynapp-svc-hp
hello hostPath Volumes

bash-5.0# curl http://mynapp-svc-hp
hello hostPath Volumes

bash-5.0# exit

Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
