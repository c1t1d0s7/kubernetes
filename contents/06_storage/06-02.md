## 2. emptyDir 볼륨

### 1) emptyDir 볼륨 소개
EmptyDir 볼륨은 아무 데이터도 없는 빈 디렉토리를 제공해주는 볼륨이다. 파드가 생성하는 데이터를 저장할 수 있으며, 동일한 파드 내의 컨테이너 간에 데이터를 공유할 때 유용하게 사용할 수 있다. 또한, 빅 데이터와 같은 큰 데이터 셋트를 처리하기 위한 임시 저장소로도 사용가능하다.

### 2) fortune 컨테이너 이미지 빌드
하나의 파드에 두 개의 컨테이너를 생성하여, EmptyDir 볼륨을 공유하고, 이 볼륨에 하나의 컨테이너는 데이터를 생성하고, 다른 하나의 컨테이너는 데이터를 사용하는 시나리오를 실행하기 위해 fortune 이미지를 생성하고자 한다.

이미지에 빌드 할 애플리케이션을 생성한다.
> fortune.sh
```bash
#!/bin/bash
trap "exit" SIGINT
mkdir /var/htdocs

while :
do
  echo $(date) Writing fortune to /var/htdocs/index.html
  /usr/games/fortune > /var/htdocs/index.html
  sleep 3
done
```
이 쉘 스크립트는 fortune 명령을 이용하여 3초마다 새로운 웹 컨텐츠를 생성하는 애플리케이션용 쉘 스크립트이다. fortune은 포춘쿠키 처럼 랜덤한 문장 및 문구를 랜덤으로 출력한다.

이미지를 빌드할 Dockerfile 파일이다.
> Dockerfile
```dockerfile
FROM ubuntu:latest

RUN apt-get update ; apt-get -y install fortune
ADD fortune.sh /bin/fortune.sh

ENTRYPOINT /bin/fortune.sh
```
베이스 이미지는 ubuntu를 사용하며, fortune 명령을 사용하기 위해 패키지를 설치하고, /bin 디렉토리에 이전에 작성한 애플리케이션인 fortune.sh을 복사한다. 이 이미지를 컨테이너로 동작시키면 fortune.sh 애플리케이션이 실행되며, 3초마다 index.html 파일의 내용을 변경하게 된다.

이미지를 빌드 한다.
```
$ docker build -t <ACCOUNT>/fortune .
```

이미지 이름에 자신의 도커 허브 계정을 사용한다.
```
$ docker login
```

빌드된 이미지를 도커 허브로 푸시한다.
```
$ docker push <ACCOUNT>/fortune
```
만약 도커 허브 계정이 없다면, c1t1d0s7/fortune 이미지를 사용한다.

### 3) emptyDir 볼륨 생성
다음은 파드나 컨트롤러에 볼륨을 정의하는 포맷이다.
```yaml
spec:
  containers:
  - volumeMounts:
    - name: <VOL-NAME>
        mountPath: <PATH>
  volumes:
  - name: <VOL-NAME>
    emptyDir: {}
```

- .spec.volumes: 
- .spec.volumes.emptyDir: emptyDir 타입의 볼륨정의
  * {}: 값을 비워두면 호스트의 로컬 디스크를 사용
  * { Memory }: RAM을 사용해 고성능의 볼륨을 제공
- .spec.volumes.name: 볼륨의 이름

- .spec.containers.volumeMounts: 컨테이너 마운트할 볼륨을 선언
- .spec.containers.volumeMounts.name: 사용할 볼륨 이름().spec.volumes 필드와 이름 참조)
- .spec.containers.volumeMounts.mountPath: 컨테이너 내의 마운트 포인트

다음과 같이 EmptyDir 볼륨을 사용하는 레플리카셋을 정의한다.
> mynapp-rs-emptydir.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-fortune
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mynapp-rs-fortune
  template:
    metadata:
      labels:
        app: mynapp-rs-fortune
    spec:
      containers:
      - name: web-server
        image: nginx:alpine
        volumeMounts:
        - name: web-fortune
          mountPath: /usr/share/nginx/html
          readOnly: true
        ports:
        - containerPort: 80
      - name: html-generator
        image: c1t1d0s7/fortune
        volumeMounts:
        - name: web-fortune
          mountPath: /var/htdocs
      volumes:
      - name: web-fortune
        emptyDir: {}
```
web-fortune이란 이름의 EmptyDir 볼륨을 정의했으며, web-server 컨테이너는 nginx 웹 서버로 이미지로 /usr/share/nginx/html 디렉토리에 볼륨을 마운트 하며, html-generator 컨테이너는 fortune 이미지를 사용하여 같은 볼륨을 /var/htdocs 디렉토리에 마운트 한다.

함께 사용할 서비스도 생성한다.
> mynapp-svc-emptydir.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-fortune
spec:
  ports:
  - port: 80
    targetPort: 80
  selector:
    app: mynapp-rs-fortune
```
포트는 80번 포트이다. nginx 이미지는 기본적으로 80번 포트로 서비스 한다.

### 5) gitRepo 볼륨
> DEPRECATED  
gitRepo 볼륨은 EmptyDir 볼륨에 지정한 Git 리포지토리의 파일을 채워서 제공해주는 볼륨이다.

다음은 gitRepo 볼륨을 정의하는 예제이다.
```yaml
volumes:
- name: <VOL-NAME>
  gitRepo:
    repository: <GIT-REPO-URL>
    revision: <BRANCH>
    directory: <PATH>
```
- .volumes.gitRepo: gitRepo 볼륨 정의
- .volumes.gitRepo.repository: Git 저장소의 주소
- .volumes.gitRepo.revision: 저장소의 브랜치
- .volumes.gitRepo.directory: 저장소의 특정 경로 지정

### 6) 초기화 컨테이너
gitRepo 볼륨 유형은 더이상 사용되지 않기 때문에 대신, Git 저장소가 있는 컨테이너를 프로비전 하려면 초기화 컨테이너(InitContainer)에 EmptyDir 볼륨을 마운트하고, 여기에 git 명령을 사용해서 Git 저장소를 복제하고, 파드 컨테이너에서 emptyDir 볼륨을 마운트 한다.

"초기화 컨테이너"는 파드의 "애플리케이션 컨테이너"가 실행되기 전에, 여러가지 작업을 수행하도록 설정할 수 있는 컨테이너다.

초기화 컨테이너는 애플리케이션 컨테이너가 실행되기 전 딱 한번만 실행되며, 초기화 컨테이너가 실패시 성공할 때 까지 재시도를 하게되며, 초기화 컨테이너가 성공하지 전까지는 애플리케이션 컨테이너는 실행되지 않는다.

> 참고  
> 초기화 컨테이너는 레디니스 프로브를 지원하지 않는다.

다음은 초기화 컨테이너를 사용하여 Git 저장소를 제공하는 예제이다.
> mynapp-pod-git.yml
```
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-git
spec:
  initContainers:
    - name: git-clone
      image: alpine/git
      args:
        - clone
        - --single-branch
        - --
        - https://github.com/kubernetes/kubernetes
        - /repo
      volumeMounts:
        - name: git-repository
          mountPath: /repo
  containers:
    - name: git-container
      image: busybox
      args: ['tail', '-f', '/dev/null']
      volumeMounts:
        - name: git-repository
          mountPath: /repo
  volumes:
    - name: git-repository
      emptyDir: {}
```
초기화 컨테이너(.spec.initContainers)는 쿠버네티스 저장소를 Git으로 클론하며 이를 /repo 디렉토리에 저장하게 된다. /repo 디렉토리는 emptyDir 타입의 볼륨인 git-repository로 제공된다. 초기화 컨테이너가 성공하면, git-container 컨테이너에서 똑같은 git-repository 볼륨을 마운트하게 된다.

초기화 컨테이너를 가지고 있는 파드를 생성 해보자.
```
$ kubectl create -f mynapp-pod-git.yml

pod/mynapp-pod-git created
```

곧바로 파드의 상태를 추적 해보자.
```
$ kubectl get pods --watch
NAME             READY   STATUS            RESTARTS   AGE
mynapp-pod-git   0/1     Init:0/1          0          4s
mynapp-pod-git   0/1     PodInitializing   0          107s
mynapp-pod-git   1/1     Running           0          111s
```
파드의 상태가 초기화(Init:0/1)에서 파드 초기화중(PodInitializing)로 전환되고, 마지막으로 초기화가 성공하면 애플리케이션 컨테이너가 실행(Running) 상태가 된다.

실제 Git 저장소가 클론 됬는지 확인해보자.
```
$ kubectl exec mynapp-pod-git -- ls -l /repo

total 216
lrwxrwxrwx    1 root     root            21 Mar 17 23:33 BUILD.bazel -> build/root/BUILD.root
drwxr-xr-x    2 root     root          4096 Mar 17 23:33 CHANGELOG
lrwxrwxrwx    1 root     root            19 Mar 17 23:33 CHANGELOG.md -> CHANGELOG/README.md
-rw-r--r--    1 root     root           493 Mar 17 23:33 CONTRIBUTING.md
drwxr-xr-x    2 root     root          4096 Mar 17 23:33 Godeps
-rw-r--r--    1 root     root         11358 Mar 17 23:33 LICENSE
...
```
/repo 디렉토리에 정상적으로 쿠버네티스 저장소가 클론 되었다.

### 7) 컨트롤러, 파드 및 서비스 삭제
```
$ kubectl delete -f mynapp-rs-emptydir.yml -f mynapp-svc-emptydir.yml -f mynapp-pod-git.yml

replicaset.apps "mynapp-rs-fortune" deleted
service "mynapp-svc-fortune" deleted
pod "mynapp-pod-git" deleted
```
