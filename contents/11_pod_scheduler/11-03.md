## 3. 어피니티
어피니티(Affinity)는 우리말로 쉽게 풀어쓰면 선호도라고 할 수 있다.

노드네임 또는 노드 셀렉터는 특정 파드를 배치할 때 특정 노드를 정확히 지칭하지만 어피니티는 선호하는 노드의 엄격함을 지정할 수 있다. 즉, 특정노드에 배치되는 것을 선호하지만 반드시 따라야하는 룰은 아닐 수도 있다는 의미다.

어피니티는 또한 파드간에도 선호도와 비-선호도를 지정해 같은 노드에 배치될것이냐, 다른 노드에 배치될것이냐를 지정할 수 있다. 즉, 자주 통신하는 파드사이에는 같은 노드에 배치되게 하고, 리소스를 많이 사용하는 파드들은 같은 노드에 배치되지 않도록 하는 방법이다.

- 노드 어피니티: 파드를 특정 노드에 배치할것을 선호
- 파드 어피니티: 파드간에 같은 노드에 배치할것을 선호
- 파드 안티 어피니티: 파드간에 다른 노드에 배치할것을 선호

### 1) 노드 어피니티
노드 어피니티는 노드 셀럭터와 비슷하게 파드를 특정 노드에 배치할 수 있다. 그러나 노드 셀렉터는 반드시 해당 레이블이 있는 노드에만 배치되지만, 노드 어피니티는 파드의 배치를 특정 노드에 강제하지 않을수도 있다.

#### (1) 노드 레이블 부여
만약 각 노드에 인공지능 애플리케이션을 위한 GPU 그래픽 카드의 모델을 레이블로 선언해보자.

kube-node1 노드에 gpu-model 키와 3080 값을 가지는 레이블을 선언한다.
```
$ kubectl label node kube-node1 gpu-model=3080

node/kube-node1 labeled
```

kube-node2 노드에 gpu-model 키와 2080 값을 가지는 레이블을 선언한다.
```
$ kubectl label node kube-node2 gpu-model=2080

node/kube-node2 labeled
```

kube-node3 노드에 gpu-model 키와 1660 값을 가지는 레이블을 선언한다.
```
$ kubectl label node kube-node3 gpu-model=1660

node/kube-node3 labeled
```

노드에 선언된 리이블을 확인해보자.
```
$ kubectl get nodes --show-labels

NAME                 ...   LABELS
kube-controlplane1   ...   ...
kube-node1           ...   ...gpu-model=3080,gpu=highend,  ...
kube-node2           ...   ...gpu-model=2080,gpu=midrange, ...
kube-node3           ...   ...gpu-model=1660,gpu=lowend,   ...
```

#### (2) 노드 어피니티 오브젝트
다음은 노드 어피니티 오브젝트의 예제다.
```yaml
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: gpu-model
            operator: In
            values:
            - '3080'
            - '2080'
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 10
        preference:
          matchExpressions:
          - key: gpu-model
            operator: In
            values:
            - titan
```

다음은 노드 어피니티의 주요 필드다.

- .spec.affinity.nodeAffinity
  - .requiredDuringSchedulingIgnoredDuringExecution: 파드가 스케줄링될때 반드시 조건에 만족(hard: 엄격함)
    - .nodeSelectorTerms
      - .matchExpressions
        - .key: 레이블의 키
        - .operator: 연산자
          - In, NotIn, Exists, DoesNotExist, Gt, Lt 
        - .values: 레이블의 값

- .spec.affinity.nodeAffinity
  - .preferredDuringSchedulingIgnoredDuringExecution: 파드가 스케줄링될때 선호되는 조건 (soft: 유연함)
    - .weight: 가중치(1-100)
    - .preference
      - .matchExpressions
        - .key
        - .operator
        - .values

#### (3) 노드 어피니티가 적용된 리소스 생성
다음은 노드 어피니티를 선언한 레플리카셋 오브젝트다.
> myapp-rs-nodeaff.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-nodeaff
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp-rs-nodeaff
  template:
    metadata:
      labels:
        app: myapp-rs-nodeaff
    spec:
      affinity:
        nodeAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: gpu-model
                operator: In
                values:
                - '3080'
                - '2080'
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 10
            preference:
              matchExpressions:
              - key: gpu-model
                operator: In
                values:
                - titan
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
```
노드에 gpu-model='3080' 또는 gpu-model='2080' 레이블이 반드시 있어야 하며, gpu=titan 레이블은 선호되나 강제사항은 아니다.

노드 어피니티가 적용된 레플리카셋 리소스를 생성하자.
```
$ kubectl create -f myapp-rs-nodeaff.yaml

replicaset.apps/myapp-rs-nodeaff created
```

#### (4) 노드 어피니티가 적용된 리소스 확인

레플리카셋 리소스를 확인해보자.
```
$ kubectl get replicasets.apps

NAME               DESIRED   CURRENT   READY   AGE
myapp-rs-nodeaff   2         2         2       37s
```

파드가 어떤 노드에 배치되었는지 확인해보자.
```
$ kubectl get pods -o wide

NAME                     ...   IP              NODE         ...
myapp-rs-nodeaff-v6452   ...   10.233.101.75   kube-node1   ...
myapp-rs-nodeaff-xjjjf   ...   10.233.103.76   kube-node2   ...
```
gpu-model='3080' 또는 gpu-model='2080' 레이블이 있는 kube-node1과 kube-node2에 배치되었다.

> 참고  
> nodeSelector와 nodeAffinity 둘 다 설정되었다면 두 조건 모두 만족해야한다.

#### (5) 리소스 삭제
```
$ kubectl delete replicasets.apps myapp-rs-nodeaff

```

### 2) 파드 어피니티 및 안티 어피니티
파드 어피니티 및 안티 어피니티는 파드간에 같은 노드에 배치할것인지, 다른 노드에 배치할것인지 선호도를 지정할 수 있다.

#### (1) 파드 어피니티 및 안티 어피니티 오브젝트
다음은 파드 어피니티 및 안티 어피니티 오브젝트의 예제다.
```yaml
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: tier
            operator: In
            values:
            - frontend
        topologyKey: "kubernetes.io/hostname"
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: tier
            operator: In
            values:
            - cache 
        topologyKey: "kubernetes.io/hostname"
```

대부분 노드 어피니티와 같은 구조를 가지고 있으나 몇가지 추가적인 필드가 있다.  
다음은 파드 어피니티 및 안티 어피니티의 주요 필드다.  

- .spec.affinity.podAffinity (*)
- .spec.affinity.podAntiAffinity (*) 
  - .requiredDuringSchedulingIgnoredDuringExecution: 파드가 스케줄링될때 반드시 조건에 만족(hard: 엄격함)
    - .labelSelctor: (*)
      - .matchExpressions
        - .key: 레이블의 키
        - .operator: 연산자
          - In, NotIn, Exists, DoesNotExist, Gt, Lt 
        - .values: 레이블의 값
    - topologyKey: 파드의 친화성 또는 비-친화성을 통해 배치 또는 분리할 기준 (일반적으로 파드를 구분할 수 있는 레이블을 지정)(*)

preferredDuringSchedulingIgnoredDuringExecution 필드도 설정이 가능하나 일반적으로 어피니티 및 안티 어피니티에서는 requiredDuringSchedulingIgnoredDuringExecution 필드를 사용하는게 일반적이다.

> 참고: 토폴로지 키
> 토폴로지 키는 파드의 고가용성 및 효율적인 리소스 관리를 위해 파드를 분산하기 위한 도메인과 같은 영역이다. 토폴로지 키는 지역, 영역, 노드 등을 구별할 수 있는 노드의 레이블 키다.
> ```
> kubectl get nodes --show-labels
> 
> NAME                 ...   LABELS
> kube-controlplane1   ...   ...kubernetes.io/hostname=kube-controlplane1...
> kube-node1           ...   ...kubernetes.io/hostname=kube-node1...
> kube-node2           ...   ...kubernetes.io/hostname=kube-node2...
> kube-node3           ...   ...kubernetes.io/hostname=kube-node3...
> ```

#### (2) 파드 어피니티 및 안티 어피니티 리소스 생성

다음은 파드 안티 어피니티를 선언한 오브젝트다.
> myapp-rs-podaff-cache.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-aff-cache
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp-rs-aff-cache
      tier: cache
  template:
    metadata:
      labels:
        app: myapp-rs-aff-cache
        tier: cache
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: tier
                operator: In
                values:
                - cache
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
...
```
해당 레플리카셋 리소스는 tier=cache 레이블을 설정하였고, 캐시 티어로 동작하는 애플리케이션이라고 가정한다. 또한 토폴로지키를 "kubernetes.io/hostname" 레이블을 지정하였고, 캐시 티어 파드간에는 같은 노드에 배치되지 않도록 구성하였다.

다음은 파드 안티 어피니티와 파드 어피니티를 선언한 오브젝트다.
> myapp-rs-podaff-front.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-aff-front
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp-rs-aff-front
      tier: frontend
  template:
    metadata:
      labels:
        app: myapp-rs-aff-front
        tier: frontend
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: tier
                operator: In
                values:
                - frontend
            topologyKey: "kubernetes.io/hostname"
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: tier
                operator: In
                values:
                - cache
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
```
해당 레플리카셋 리소스는 tier=frontend 레이블을 설정하였고, 프론트엔드 티어로 동작하는 애플리케이션이라고 가정한다. 또한 토폴로지키를 "kubernetes.io/hostname" 레이블을 지정하였고, 프론트엔드 티어 파드간에는 같은 노드에 배치되지 않고, 캐시 티어 파드와 같이 배치되도록 구성되었다.

두 레플리카셋 리소스를 생성하자.
```
$ kubectl create -f myapp-rs-podaff-cache.yaml -f myapp-rs-podaff-front.yaml

replicaset.apps/myapp-rs-aff-cache created
replicaset.apps/myapp-rs-aff-front created
```


프론트엔드 티어 레플리카셋 리소스의 복제본을 두 개로 줄인다.
```
$ kubectl scale replicaset myapp-rs-aff-front --replicas 2

replicaset.apps/myapp-rs-aff-front scaled
```

파드가 두 쌍이 남아있는지 확인한다.
