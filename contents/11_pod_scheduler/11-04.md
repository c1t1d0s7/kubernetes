## 4. 테인트 및 톨러레이션
테인트(Taint)는 쿠버네티스 클러스터의 특정 노드에 더 이상 추가적인 파드가 배치되지 않도록 파드를 스케줄링하지 않는다. 그러나 톨러레이션(Toleration)을 사용하면 테인트가 설정된 노드에 파드를 스케줄링할 수 있도록 구성할 수 있다.

테인트와 톨러레이션을 사용하는 주요 목적은 쿠버네티스 클러스터의 **노드에 특정 역할을 할당**하기 위해 사용한다. 예를 들어 노드에 GPU가 있고, GPU를 사용하지 않는 애플리케이션 파드는 배치하지 않고, GPU를 사용하는 애플리케이션은 해당 노드에 배치할 수 있다.

쿠버네티스 클러스터의 마스터 노드에는 마스터 역할의 테인트를 부여했기 때문에 특정 톨러레이션이 없는 경우 마스터 노드에 스케줄링하지 않는다.

### 1) 테인트
테인트는 레이블과 비슷하게 "Key[=Value]:Effect" 형태로 정의한다.

노드에 테인트를 설정하는 방법은 다음과 같다.
```
kubectl taint [NODE] [NODE-NAME] [KEY[=VALUE]:EFFECT] ...
```

테인트의 효과(Effect)는 세 가지가 있다.
- NoSchedule: 톨러레이션에 해당 레이블이 없는 파드는 스케줄링할 수 없다. (기존 파드에 적용되지 않음)
- PreferNoSchedule: 톨러레이션에 해당 레이블이 없는 파드는 스케줄링할 수 없으나, 리소스가 부족하면 배치될 수 있다. (기존 파드에 적용되지 않음)
- NoExecute: 톨러레이션에 해당 레이블이 없는 파드는 스케줄링할 수 없고, 기존 파드도 설정을 무시할 수 없다.

즉, "Key[=Value]:Effect"의 의미는 톨러레이션에 의해 허용될 레이블이며, 해당 레이블이 없으면 효과에 의해 스케줄링하지 않는다는 의미다.

#### (1) 노드에 설정된 테인트 확인
노드의 역할을 확인해보자.
```
$ kubectl get nodes

NAME                 STATUS   ROLES    AGE    VERSION
kube-controlplane1   Ready    master   2d3h   v1.18.10
kube-node1           Ready    <none>   2d3h   v1.18.10
kube-node2           Ready    <none>   2d3h   v1.18.10
kube-node3           Ready    <none>   2d3h   v1.18.10
```
kube-controlplane1 마스터 노드에만 master 역할이 부여되어 있으며, 지금까지 실습에 의해 마스터 노드에는 파드가 스케줄링이 되지 않는다는 것는 알 수 있다.

노드의 상세 정보중 테인트 관련 정보를 확인해보자.
```
$ kubectl describe nodes | grep Taints

Taints:             node-role.kubernetes.io/master:NoSchedule
Taints:             <none>
Taints:             <none>
Taints:             <none>
```

JsonPath를 이용한 방법도 있다.
```
$ kubectl get nodes -o=jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.taints}{"\n"}{end}'

kube-controlplane1	[map[effect:NoSchedule key:node-role.kubernetes.io/master]]
kube-node1
kube-node2
kube-node3
```

마스터 노드에는 "node-role.kubernetes.io/master" 레이블의 키가 없으면 파드를 스케줄링하지 않는다는 테인트가 설정되어 있으며, 나머지 노드에는 테인트가 없다.

#### (2) 노드에 테인트 설정
> 참고  
> 노드 어피니티 실습에서 캐시 티어 및 프론트엔드 티어 레플리카셋 파드가 배치되지 않은 노드에 테인트를 설정한다.  
> 여기서는 kube-node3다.

kube-node3에 env=production 레이블이 없으면 스케줄링하지 않도록 설정한다.
```
$ kubectl taint node kube-node3 env=production:NoSchedule

node/kube-node3 tainted
```

kube-node3 노드에 테인트가 설정되었는지 확인하자.
```
$ kubectl describe nodes | grep Taints

Taints:             node-role.kubernetes.io/master:NoSchedule
Taints:             <none>
Taints:             <none>
Taints:             env=production:NoSchedule
```

#### (3) 톨러레이션이 없는 레플리카셋
다음은 톨러레이션 설정이 없는 레플리카셋 리소스다.
> myapp-rs-notol.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-notol
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp-rs-notol
      tier: backend
  template:
    metadata:
      labels:
        app: myapp-rs-notol
        tier: backend
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: tier
                operator: In
                values:
                - cache
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
```
위 오브젝트는 톨러레이션 선언이 없으며, 복제본 개수는 하나, tier=backend 백엔드 레이블이 설정되어 있고, tire=cache 캐시 티어 레이블과 같이 배치되지 않는 안티 어피니가 설정되어 있다.

현재 kube-node3는 테인트가 설정되어 있어 톨러레이션이 없으면 배치되지 않을 것이고, kube-node1, kube-node2 노드는 캐시 티어 레이블을 가진 파드가 배치되어 있어, 이 리소스는 배치될 수 없다.

톨러레이션이 없는 레플리카셋 리소스를 생성해보자.
```
$ kubectl create -f myapp-rs-notol.yaml

replicaset.apps/myapp-rs-notol created
```

파드의 배치 및 상태를 확인 해보자.
```
$ kubectl get pods -o wide

NAME                       READY   STATUS  ... NODE        ...
myapp-rs-aff-cache-cwb9k   1/1     Running ... kube-node2  ...myapp-rs-aff-cache-p69s6   1/1     Running ... kube-node1  ...myapp-rs-aff-front-vpmxd   1/1     Running ... kube-node1  ...myapp-rs-aff-front-z5pjn   1/1     Running ... kube-node2  ...myapp-rs-notol-t8qbw       0/1     Pending ... <none>      ...
```
톨러레이션이 없는 파드는 배치되지 못한다.

파드의 상세 정보중 이벤트 정보를 확인해보자.
```
$ kubectl describe pod myapp-rs-notol-t8qbw

Name:           myapp-rs-notol-t8qbw
...
Events:
  Type     Reason            Age   From               Message
  ----     ------            ----  ----               -------
  Warning  FailedScheduling  85s   default-scheduler  0/4 nodes are available: 1 node(s) had taint {env: production}, that the pod didn't tolerate, 1 node(s) had taint {node-role.kubernetes.io/master: }, that the pod didn't tolerate, 2 node(s) didn't match pod affinity/anti-affinity, 2 node(s) didn't match pod anti-affinity rules.
...
```
두 개의 노드는 파드 어피니티 / 안티 어피니티로 매칭되지 않으며, 두 개의 노드는 테인트를 가지고 있고 톨러레이션이 없다고 보고하고 있다.

레플리카셋을 삭제한다.
```
$ kubectl delete -f myapp-rs-notol.yaml

replicaset.apps "myapp-rs-notol" deleted
```

### 2) 톨러레이션

#### (1) 톨러레이션 오브젝트 정의
다음은 톨레레이션을 오브젝트로 정의하는 예제다.
```yaml
spec:
  tolerations:
  - key: env
    operator: Equal
    value: production
    effect: NoSchedule
```

다음은 톨러레이션 설정의 주요 필드다.
- .spec.tolerations:
  - key: 레이블 키
  - operator: Equal(기본), Exists
  - value: 레이블 값
  - effect: NoSchedule, PreferNoSchedule, NoExecute

#### (2) 톨러레이션을 사용한 리소스 생성

다음은 톨러레이션을 사용한 레플리카셋 리소스다.
> myapp-rs-tol.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-tol
spec:
  replicas: 1
  selector:
    matchLabels:
      app: myapp-rs-tol
      tier: backend
  template:
    metadata:
      labels:
        app: myapp-rs-tol
        tier: backend
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: tier
                operator: In
                values:
                - cache
            topologyKey: "kubernetes.io/hostname"
      tolerations:
      - key: env
        operator: Equal
        value: production
        effect: NoSchedule
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
```
전에 생성했던 톨러레이션을 사용하지 않는 레플리카셋 컨트롤러와 설정은 같고, env=production:NoSchedule 톨러레이션이 설정되었다.

톨러레이션이 정의된 레플리카셋 리소스를 생성해보자.
```
$ kubectl create -f myapp-rs-tol.yaml

replicaset.apps/myapp-rs-tol created
```

#### (4) 리소스 및 테인트 삭제
사용했던 모든 레플리카셋 컨트롤러를 삭제한다.
```
$ kubectl delete replicasets.apps --all

replicaset.apps "myapp-rs-aff-cache" deleted
replicaset.apps "myapp-rs-aff-front" deleted
replicaset.apps "myapp-rs-tol" deleted
```

잊지 말고 kube-node3에 설정한 테인트도 삭제한다. 테인트 삭제는 "KEY:EFFECT-"로 정의할 수 있다.
```
$ kubectl taint node kube-node3 env-

node/kube-node3 untainted
```

테인트가 제거되었는지 확인하자.
```
$ kubectl describe nodes | grep Taints
Taints:             node-role.kubernetes.io/master:NoSchedule
Taints:             <none>
Taints:             <none>
Taints:             <none>
```
