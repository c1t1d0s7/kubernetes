## 5. 커든 및 드레인
커든(Cordon)은 특정 노드에 새로운 파드가 스케줄링되지 않도록 하는 기능이다. 테인트와 비슷하지만, 테인트는 특정 노드를 특정 역할로 사용하기 위해 톨러레이션과 함께 사용하지만, 커든은 새로운 파드가 스케줄링 되지 않도록 한다. 예외는 없다. 그러나 현재 배치되어 동작중인 파드에는 영향을 미치지 않는다. 당연하지만 새로 스케줄링되지 않도록만 하는 것이다.

드레인(Drain)은 관리적인 여러 이유, 특히 유지보수의 목적으로 해당 노드에 파드가 없어야 하는경우, 모든 파드는 퇴거(Evict)되고 다른 노드로 이전(복제본을 관리하는 컨트롤러만 해당)한다. 또한 드레인을 설정하면 모든 파드를 이전하기 전 커든이 먼저 적용된다.

### 1) 커든

#### (1) 커튼 명령
노드에 커든을 설정하고 해제하는 명령은 다음과 같다.
```
kubectl cordon [NODE]
kubectl uncordon [NODE]
```

#### (2) 커튼 설정
kube-node1과 kube-node2에 커든을 설정하자.
```
$ kubectl cordon kube-node1

node/kube-node1 cordoned
```
```
$ kubectl cordon kube-node2

node/kube-node2 cordoned
```

커든이 적용되었는지 노드의 상태를 확인해보자.
```
$ kubectl get nodes

NAME                 STATUS                     ROLES    AGE    VERSION
kube-controlplane1   Ready                      master   2d4h   v1.18.10
kube-node1           Ready,SchedulingDisabled   <none>   2d4h   v1.18.10
kube-node2           Ready,SchedulingDisabled   <none>   2d4h   v1.18.10
kube-node3           Ready                      <none>   2d4h   v1.18.10
```
kube-node1과 kube-node2의 상태에 SchedulingDisabled로 보고하는 것을 확인할 수 있다.

#### (3) 커든 확인을 위한 레플리카셋 생성
다음은 커든이 적용된 노드에 파드가 배치되지 않는지 확인하기 위한 일반적인 레플리카셋 리소스다.
> myapp-rs-cordon.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-cordon
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp-rs-cordon
  template:
    metadata:
      labels:
        app: myapp-rs-cordon
    spec:
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
```

레플리카셋을 생성해보자.
```
$ kubectl create -f myapp-rs-cordon.yaml

replicaset.apps/myapp-rs-cordon created
```

#### (4) 커든 확인을 위한 레플리카셋 확인
파드의 배치 상태를 확인해보자.
```
$ kubectl get pods -o wide

NAME                    ...   NODE         ...
myapp-rs-cordon-88sgl   ...   kube-node3   ...
myapp-rs-cordon-p5hvn   ...   kube-node3   ...
```
커든이 설정되지 않은 kube-node3 노드에만 두 개의 파드가 배치된것을 확인할 수 있다.

레플리카셋 리소스의 스케일을 확장해보자.
```
$ kubectl scale replicaset myapp-rs-cordon --replicas 4

replicaset.apps/myapp-rs-cordon scaled
```

다시 파드의 배치 상태를 확인해보자.
```
$ kubectl get pods -o wide

NAME                    ...   NODE         ...
myapp-rs-cordon-88sgl   ...   kube-node3   ...
myapp-rs-cordon-fh6js   ...   kube-node3   ...
myapp-rs-cordon-p5hvn   ...   kube-node3   ...
myapp-rs-cordon-rnr74   ...   kube-node3   ...
```
모든 파드가 kube-node3에만 배치된것을 확인할 수 있다.

#### (5) 리소스 삭제 및 커든 해제
레플리카셋을 삭제한다.
```
$ kubectl delete replicasets.apps myapp-rs-cordon

replicaset.apps "myapp-rs-cordon" deleted
```

잊지말고 노드에 설정된 커든도 해제한다.
```
$ kubectl uncordon kube-node1

node/kube-node1 uncordoned
```
```
$ kubectl uncordon kube-node2

node/kube-node2 uncordoned
```

커든이 해제되었는지 확인하자.
```
$ kubectl get nodes

NAME                 STATUS   ROLES    AGE    VERSION
kube-controlplane1   Ready    master   2d4h   v1.18.10
kube-node1           Ready    <none>   2d4h   v1.18.10
kube-node2           Ready    <none>   2d4h   v1.18.10
kube-node3           Ready    <none>   2d4h   v1.18.10
```

### 2) 드레인
드레인을 설정할 노드에 데몬셋 파드가 실행되고 있다면, 데몬셋이 실행한 파드는 삭제되어도 즉시 재실행되기 때문에, 데몬셋이 적용된 파드를 드레인할 수 없다. 이런경우 데몬셋 리소스를 무시하는 옵션을 사용해 드레인할 수 있다.

#### (1) 파드 및 데몬셋 확인
쿠버네티스 클러스터 전체에 데몬셋 리소스를 확인해보자.
```
$ kubectl get daemonsets.apps --all-namespaces

NAMESPACE        NAME                       ...
ingress-nginx    ingress-nginx-controller   ...
kube-system      calico-node                ...
kube-system      kube-proxy                 ...
kube-system      nodelocaldns               ...
metallb-system   speaker                    ...
rook-ceph        csi-cephfsplugin           ...
rook-ceph        csi-rbdplugin              ...
```
이런 데몬셋 때문에 기본적으로는 드레인을 할 수 없다.

kube-node3 노드에서 실행되고 있는 파드의 목록을 확인해보자.
```
$ kubectl describe node kube-node3

...

데몬셋 컨트롤러에의해 실행되고 있는 파드만 있는것은 아니지만, 총 16개의 파드가 동작하고 있다. 구성에 따라 동작하고있는 파드의 개수는 다를 수 있다.

#### (2) 드레인 명령
노드에 드레인을 설정하고 해제하는 명령은 다음과 같다.
```
kubectl drain [NODE] [OPTIONS]
kubectl uncordon [NODE]
```
드레인은 커든을 시행한 후 파드를 이전시킨다. 별도의 드레인을 해제하는 명령은 없고, 잊지말고 커든을 해제해야 한다.

#### (3) 드레인 설정
kube-node3 노드에 드레인을 설정해보자.
```
$ kubectl drain kube-node3

error: unable to drain node "kube-node3", aborting command...

 kube-node3
cannot delete Pods with local storage (use --delete-local-data to override): rook-ceph/csi-cephfsplugin-provisioner-c68f789b8-j4xxv, rook-ceph/csi-rbdplugin-provisioner-5bc97cdbb9-7hn48, rook-ceph/rook-ceph-mds-myfs-b-54d6c668bc-t85m6

```
kube-node3 노드는 커든되었으나, 드레인하지 못한다고 보고하고 있다.

첫 번째 이유는, 데몬셋이 관리하는 파드들이 있기 때문이다. 데몬셋은 복제본을 제공하지 않기 때문에 해당 파드가 종료되면 문제가 있기 때문이다. 데몬셋의 파드 종료를 무시하는 옵션은 --ignore-daemonsets=true 다.

두 번째 이유는, emptyDir 볼륨을 이용하고 있는 파드 때문이다. 만약 볼륨의 데이터가 삭제되어도 상관없다면 --delete-local-data=true 옵션을 사용해 파드를 삭제할 수 있다.

드레인을 설정한 노드의 유지보수 및 관리가 끝났다면 반드시 커든을 해제해준다.
```
$ kubectl uncordon kube-node3

node/kube-node3 uncordoned
```

노드의 상태를 확인해보자.
```
$ kubectl get nodes
