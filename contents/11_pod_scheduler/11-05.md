## 5. 커든 및 드레인
커든(Cordon)은 특정 노드에 파드가 스케줄링되지 않도록 하는 기능이다. 테인트와 비슷하지만, 테인트는 특정 노드를 특정 역할로 사용하기 위해 톨러레이션과 함께 사용하지만, 커든은 모든 파드의 스케줄링이 되지 않도록 한다. 예외는 없다. 그러나 현재 배치되어 동작중인 파드에는 영향을 미치지 않는다. 당연하지만 새로 스케줄링되지 않도록만 하는 것이다.

드레인(Drain)은 관리적인 여러 이유, 특히 유지보수의 목적으로 해당 노드에 파드가 없어야 하는경우, 모든 파드를 퇴거(Evict)되고 다른 노드로 이전(복제본을 관리하는 컨트롤러만 해당)한다. 또한 드레인을 설정하면 모든 파드를 이전하기 전 커든이 먼저 적용된다.

### 1) 커든

#### (1) 커튼 명령
노드에 커든을 설정하고 해제하는 명령은 다음과 같다.
```
kubectl cordon [NODE]
kubectl uncordon [NODE]
```

#### (2) 커튼 설정
kube-node1과 kube-node2에 커든을 설정하자.
```
$ kubectl cordon kube-node1

node/kube-node1 cordoned
```
```
$ kubectl cordon kube-node2

node/kube-node2 cordoned
```

커든이 적용됬는지 노드의 상태를 확인해보자.
```
$ kubectl get no

NAME           STATUS                     ROLES    AGE   VERSION
kube-master1   Ready                      master   8d    v1.16.8
kube-node1     Ready,SchedulingDisabled   <none>   8d    v1.16.8
kube-node2     Ready,SchedulingDisabled   <none>   8d    v1.16.8
kube-node3     Ready                      <none>   8d    v1.16.8
```
kube-node1과 kube-node2의 상태에 SchedulingDisabled로 보고하는 것을 확인할 수 있다.

#### (3) 커든 확인을 위한 레플리카셋 생성
다음은 커든이 적용된 노드에 파드가 배치되지 않는지 확인하기 위한 일반적인 레플리카셋 리소스다.
> mynapp-rs-cordon.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-cordon
spec:
  replicas: 2
  selector:
    matchLabels:
      app: mynapp-rs-cordon
  template:
    metadata:
      labels:
        app: mynapp-rs-cordon
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
```

레플리카셋을 생성해보자.
```
$ kubectl create -f mynapp-rs-cordon.yml

replicaset.apps/mynapp-rs-cordon created
```

#### (4) 커든 확인을 위한 레플리카셋 확인
파드의 배치 상태를 확인해보자.
```
$ kubectl get po -o wide

NAME                     READY   STATUS    RESTARTS   AGE   IP              NODE         ...
mynapp-rs-cordon-4sfj8   1/1     Running   0          29s   10.233.76.184   kube-node3   ...
mynapp-rs-cordon-cwsh9   1/1     Running   0          29s   10.233.76.185   kube-node3   ...
```
커든이 설정되지 않은 kube-node3 노드에만 두 개의 파드가 배치된것을 확인할 수 있다.

레플리카셋 리소스의 스케일을 확장해보자.
```
$ kubectl scale rs mynapp-rs-cordon --replicas=5

replicaset.apps/mynapp-rs-cordon scaled
```

다시 파드의 배치 상태를 확인해보자.
```
$ kubectl get po -o wide
NAME                     READY   STATUS    RESTARTS   AGE     IP              NODE         ...
mynapp-rs-cordon-29fmj   1/1     Running   0          23s     10.233.76.188   kube-node3   ...
mynapp-rs-cordon-4sfj8   1/1     Running   0          2m42s   10.233.76.184   kube-node3   ...
mynapp-rs-cordon-cwsh9   1/1     Running   0          2m42s   10.233.76.185   kube-node3   ...
mynapp-rs-cordon-ffvwz   1/1     Running   0          23s     10.233.76.187   kube-node3   ...
mynapp-rs-cordon-ppx8p   1/1     Running   0          23s     10.233.76.186   kube-node3   ...
```
모든 파드가 kube-node3에만 배치된것을 확인할 수 있다.

#### (5) 리소스 삭제 및 커든 해제
레플리카셋을 삭제한다.
```
$ kubectl delete rs mynapp-rs-cordon

replicaset.apps "mynapp-rs-cordon" deleted
```

잊지말고 노드에 설정된 커든도 해제한다.
```
$ kubectl uncordon kube-node1

node/kube-node1 uncordoned
```
```
$ kubectl uncordon kube-node2

node/kube-node2 uncordoned
```

커든이 해제되었는지 확인하자.
```
$ kubectl get no

NAME           STATUS   ROLES    AGE   VERSION
kube-master1   Ready    master   8d    v1.16.8
kube-node1     Ready    <none>   8d    v1.16.8
kube-node2     Ready    <none>   8d    v1.16.8
kube-node3     Ready    <none>   8d    v1.16.8
```

### 2) 드레인
드레인을 설정할 노드에 데몬셋 파드가 실행되고 있다면, 데몬셋이 실행한 파드는 삭제되어도 즉시 재실행되기 때문에, 데몬셋이 적용된 파드를 드레인할 수 없다. 이런경우 데몬셋 리소스를 무시하는 옵션을 사용해 드레인할 수 있다.

#### (1) 파드 및 데몬셋 확인
쿠버네티스 클러스터 전체에 데몬셋 리소스를 확인해보자.
```
$ kubectl get ds --all-namespaces
NAMESPACE        NAME                       DESIRED   CURRENT   READY   ...
ingress-nginx    ingress-nginx-controller   3         3         3       ...
kube-system      calico-node                4         4         4       ...
kube-system      kube-proxy                 4         4         4       ...
kube-system      nodelocaldns               4         4         4       ...
metallb-system   speaker                    3         3         3       ...
rook-ceph        csi-cephfsplugin           3         3         3       ...
rook-ceph        csi-rbdplugin              3         3         3       ...
rook-ceph        rook-discover              3         3         3       ...
```
이런 데몬셋 때문에 기본적으로는 드레인을 할 수 없다.

kube-node3 노드에서 실행되고 있는 파드의 목록을 확인해보자.
```
$ kubectl describe no kube-node3

  ingress-nginx              ingress-nginx-controller-cx7zt             ...

  kube-system                coredns-6dc4d777d6-ndhpn                   ...
  kube-system                kube-proxy-g4vzk                           ...
  kube-system                nginx-proxy-kube-node3                     ...

  rook-ceph                  csi-cephfsplugin-jrgzv                     ...
  rook-ceph                  csi-cephfsplugin-provisioner-7b8fbf88b4-xwv...
  rook-ceph                  csi-rbdplugin-jr88v                        ...

  rook-ceph                  rook-ceph-crashcollector-kube-node3-bcfdbdc...
```

데몬셋 컨트롤러에의해 실행되고 있는 파드만 있는것은 아니지만, 총 16개의 파드가 동작하고 있다. 구성에 따라 동작하고있는 파드의 개수는 다를 수 있다.

#### (2) 드레인 명령
노드에 드레인을 설정하고 해제하는 명령은 다음과 같다.
```
kubectl drain [NODE] [OPTIONS]
kubectl uncordon [NODE]
```
드레인은 커든을 시행한 후 파드를 이전시킨다. 별도의 드레인을 해제하는 명령은 없고, 잊지말고 커든을 해제해야 한다.

데몬셋을 무시하는 옵션은 --ignore-daemonsets=true 이다.

#### (3) 드레인 설정
kube-node3 노드에 드레인을 설정해보자.
```
$ kubectl drain kube-node3

node/kube-node3 cordoned
error: unable to drain node "kube-node3", aborting command...

There are pending nodes to be drained:
 kube-node3
cannot delete DaemonSet-managed Pods (use --ignore-daemonsets to ignore): ingress-nginx/ingress-nginx-controller-cx7zt, kube-system/calico-node-vg22q, kube-system/kube-proxy-g4vzk, kube-system/nodelocaldns-wppmz, metallb-system/speaker-pplsw, rook-ceph/csi-cephfsplugin-jrgzv, rook-ceph/csi-rbdplugin-jr88v, rook-ceph/rook-discover-hs255
cannot delete Pods with local storage (use --delete-local-data to override): rook-ceph/csi-cephfsplugin-provisioner-7b8fbf88b4-xwvkc, rook-ceph/csi-rbdplugin-provisioner-6b8b4d558c-ks6cj, rook-ceph/rook-ceph-operator-69f856fc5f-r85lr
```
kube-node3 노드는 커든되었으나, 드레인하지 못한다고 보고하고 있다.
첫 번째 이유는, 데몬셋이 관리하는 파드들이 있기 때문이다.
두 번째 이유는, Ceph 관련 파드가 로컬 스토리지를 사용하고 있는 파드가 있어 해당 파드를 삭제하지 못한다고 보고하고 있다. 만약 로컬 스토리지의 데이터가 삭제되어도 상관없다면 --delete-local-data=true 옵션을 사용해 파드를 삭제할 수 있다. Ceph 관련 파드가 어떤 노드에 배치되었는지에 따라 해당 오류는 나타나지 않을 수 있다.

지금은 스토리지를 사용하는 파드가 없기 때문에, 데몬셋을 무시하는 옵션과 데이터를 삭제하는 옵션을 사용해 드레인 해보자.

노드의 상태를 확인해보자.

kube-node3 노드가 커든 상태인것을 확인할 수 있다.

#### (4) 커든 해제
드레인을 설정한 노드의 유지보수 및 관리가 끝났다면 반드시 커든을 해제해준다.
```
$ kubectl uncordon kube-node3

node/kube-node3 uncordoned
```

노드의 상태를 확인해보자.
```
$ kubectl get no
NAME           STATUS   ROLES    AGE   VERSION
kube-master1   Ready    master   8d    v1.16.8
kube-node1     Ready    <none>   8d    v1.16.8
kube-node2     Ready    <none>   8d    v1.16.8
kube-node3     Ready    <none>   8d    v1.16.8
```
