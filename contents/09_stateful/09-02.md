## 2. 스테이트풀셋 관리

### 1) 스테이트풀셋 기본
다음은 스테이트풀셋에 사용할 헤드리스 서비스 예제다.
> myapp-svc-headless.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-headless
  labels:
    app: myapp-svc-headless
spec:
  ports:
  - name: http
    port: 80
  clusterIP: None
  selector:
    app: myapp-sts
```
서비스의 이름은 myapp-svc-headless이며, .spec.clusterIP를 None으로 설정해 헤드리스 서비스로 구성한다. 파드 레이블 셀렉터는 app=myapp-sts 레이블을 가지고 있는 파드를 지정한다.

다음은 스테이트풀셋 리소스의 기본 예제다.
> myapp-sts.yaml
```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: myapp-sts
spec:
  selector:
    matchLabels:
      app: myapp-sts
  serviceName: myapp-svc-headless
  replicas: 2
  template:
    metadata:
      labels:
        app: myapp-sts
    spec:
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb
        ports:
        - containerPort: 8080
```

.spec.serviceName: 스테이트풀셋에 사용할 헤드리스 서비스 이름 지정

앞서 선언한 헤드리스 서비스를 스테이트풀셋이 사용하도록 .spec.serviceName 필드에 지정한다.

헤드리스 서비스와 스테이트풀셋 리소스를 생성하자.
```
$ kubectl create -f myapp-sts.yaml -f myapp-svc-headless.yaml

statefulset.apps/myapp-sts created
service/myapp-svc-headless created
```

스테이트풀셋 리소스를 확인해보자.
```
$ kubectl get statefulsets.apps

NAME        READY   AGE
myapp-sts   2/2     9s
```

파드의 이름을 확인해보자.
```
$ kubectl get pods

NAME          READY   STATUS    RESTARTS   AGE
myapp-sts-0   1/1     Running   0          13s
myapp-sts-1   1/1     Running   0          10s
```
스테이트풀셋 컨트롤러의 이름에 0부터 시작하는 인텍스가 순서대로 붙게 된다.

스테이트풀셋의 복제본을 3개로 스케일링해보자.
```
$ kubectl scale statefulset myapp-sts --replicas=3

statefulset.apps/myapp-sts scaled
```

다시 파드의 목록을 확인해보자.
```
$ kubectl get pods

NAME          READY   STATUS    RESTARTS   AGE
myapp-sts-0   1/1     Running   0          26s
myapp-sts-1   1/1     Running   0          23s
myapp-sts-2   1/1     Running   0          6s
```
다음 순서인 2번 인덱스가 붙은 파드를 확인할 수 있다.

스테이트풀셋의 복제본을 2개로 스케일링하고 파드 목록을 확인해보자.
```
$ kubectl scale statefulset myapp-sts --replicas=2

statefulset.apps/myapp-sts scaled
```

```
$ kubectl get pods

NAME          READY   STATUS    RESTARTS   AGE
myapp-sts-0   1/1     Running   0          32s
myapp-sts-1   1/1     Running   0          29s
```
가장 마지막 인덱스인 2번이 종료된 것을 확인할 수 있다.

헤드리스 서비스에 접근하기 전에 파드의 IP를 확인해보자.
```
$ kubectl get pods -o wide

NAME          READY   STATUS    RESTARTS   AGE     IP              ...
myapp-sts-0   1/1     Running   0          8m17s   10.233.76.79    ...
myapp-sts-1   1/1     Running   0          8m14s   10.233.103.63   ...
```
파드의 IP는 10.233.76.79, 10.233.103.63이다. 

임시 파드를 생성해 헤드리스 서비스에 접근해보자.
```
$ kubectl run nettool -it --image=ghcr.io/c1t1d0s7/network-multitool --rm bash

bash-5.1# host myapp-svc-headless
myapp-svc-headless.default.svc.cluster.local has address 10.233.76.79
myapp-svc-headless.default.svc.cluster.local has address 10.233.103.63
```
헤드리스 서비스 이름으로 DNS에 질의하면 앞서 살펴본 대로, 서비스의 ClusterIP가 아닌(어차피 없음) 각 파드의 IP로 응답하는 것을 확인할 수 있다.

```
bash-5.1# host myapp-sts-0.myapp-svc-headless
myapp-sts-0.myapp-svc-headless.default.svc.cluster.local has address 10.233.76.79
```
첫 번째 파드 이름에 서비스 이름을 붙여 질의하면, 첫 번째 파드 IP만 응답하는 것을 확인할 수 있다.

```
bash-5.1# host myapp-sts-1.myapp-svc-headless
myapp-sts-1.myapp-svc-headless.default.svc.cluster.local has address 10.233.103.63
```
두 번째 파드 역시 동일하다.

curl 명령으로 두 파드에 각각 접근해보자.
```
bash-5.1# curl myapp-sts-0.myapp-svc-headless
Hello World!
myapp-sts-0

bash-5.1# curl myapp-sts-1.myapp-svc-headless
Hello World!
myapp-sts-1

bash-5.1# exit
exit

Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```

스테이트풀셋 컨트롤러와 헤드리스 서비스를 구성하면, 파드의 FQDN을 사용하여 각 파드를 구분할 수 있다.

### 2) 스테이트풀셋의 볼륨 클레임 템플릿
스테이트풀셋은 각 파드에 고유한 볼륨을 제공한다.

다음은 볼륨 클레임 템플릿을 사용한 예제다.
> myapp-sts-vol.yaml
```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: myapp-sts-vol
spec:
  selector:
    matchLabels:
      app: myapp-sts-vol
  serviceName: myapp-svc-headless
  replicas: 2
  template:
    metadata:
      labels:
        app: myapp-sts-vol
    spec:
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb:alpine
        ports:
        - containerPort: 8080
        volumeMounts:
        - name: myapp-data
          mountPath: /data
  volumeClaimTemplates:
  - metadata:
      name: myapp-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi
      storageClassName: rook-ceph-block
```

.spec.template: 파드의 템플릿
.spec.volumeClaimTemplates: 영구 볼륨 클레임(PVC) 템플릿

다른 컨트롤러와 다르게 파드별 고유 상태를 가지기 위해 각 파드에 연결할 볼륨 클레임 템플릿을 선언한다.

볼륨 클레임 템플릿을 사용하는 스테이트풀셋 리소스를 생성하자.
```
$ kubectl create -f myapp-sts-vol.yaml

statefulset.apps/myapp-sts-vol created
```

스테이트풀셋 리소스를 확인해보자.
```
$ kubectl get statefulsets.apps myapp-sts-vol

NAME            READY   AGE
myapp-sts-vol   2/2     49s
```

파드 목록을 확인해보자.
```
$ kubectl get pods -l app=myapp-sts-vol
NAME              READY   STATUS    RESTARTS   AGE
myapp-sts-vol-0   1/1     Running   0          58s
myapp-sts-vol-1   1/1     Running   0          45s
```

파드의 상세정보에서 볼륨 관련 정보를 확인해보자.
```
$ kubectl describe pods myapp-sts-vol-0

Name:         myapp-sts-vol-0
...
Volumes:
  myapp-data:
    Type:       PersistentVolumeClaim (a reference to a PersistentVolumeClaim in the same namespace)
    ClaimName:  myapp-data-myapp-sts-vol-0
    ReadOnly:   false
...
```
myapp-data-myapp-sts-vol-0라는 PVC에 연결된 것을 확인할 수 있다.

PVC 리소스를 확인해보자.
```
$ kubectl get persistentvolumeclaims
NAME                         STATUS   VOLUME                                     ...
myapp-data-myapp-sts-vol-0   Bound    pvc-f9424efb-2349-4ba4-986a-3ac2bea3ba3f   ...
myapp-data-myapp-sts-vol-1   Bound    pvc-b4be8ea1-de19-4148-b2d1-5ca7ec303e01   ...
```
각 파드에 제공할 두 개의 PVC 리소스가 생성되어 있다.

PV 리소스를 확인해보자.
```
$ kubectl get persistentvolume
NAME                                       ... STATUS   CLAIM                                STORAGECLASS      REASON   AGE
pvc-b4be8ea1-de19-4148-b2d1-5ca7ec303e01   ... Bound    default/myapp-data-myapp-sts-vol-1   rook-ceph-block            7m6s
pvc-f9424efb-2349-4ba4-986a-3ac2bea3ba3f   ... Bound    default/myapp-data-myapp-sts-vol-0   rook-ceph-block            7m19s
```
스토리지 클레스에 의해 PVC에 제공할 PV 두 개가 생성된 것을 확인할 수 있다.

복제본 3개로 스케일링해보자.
```
$ kubectl scale statefulset myapp-sts-vol --replicas=3

statefulset.apps/myapp-sts-vol scaled
```

스테이트풀셋과 파드 목록을 확인해보자.
```
$ kubectl get statefulsets.apps myapp-sts-vol

NAME            READY   AGE
myapp-sts-vol   3/3     9m20s
```

```
$ kubectl get pods -l app=myapp-sts-vol

NAME              READY   STATUS    RESTARTS   AGE
myapp-sts-vol-0   1/1     Running   0          9m24s
myapp-sts-vol-1   1/1     Running   0          9m11s
myapp-sts-vol-2   1/1     Running   0          16s
```
다음 인덱스인 2번으로 파드가 생성된 것을 확인할 수 있다.

PV 및 PVC 리소스도 확인해보자.
```
$ kubectl get persistentvolume
NAME                                       ...
pvc-4bfee1c1-e529-46ae-a08b-0bcb2ccb8c4f   ...
pvc-b4be8ea1-de19-4148-b2d1-5ca7ec303e01   ...
pvc-f9424efb-2349-4ba4-986a-3ac2bea3ba3f   ...
```

```
$ kubectl get persistentvolumeclaims
NAME                         STATUS   ...
myapp-data-myapp-sts-vol-0   Bound    ...
myapp-data-myapp-sts-vol-1   Bound    ...
myapp-data-myapp-sts-vol-2   Bound    ...
```
myapp-sts-vol-2 파드가 사용할 PV 및 PVC 리소스도 생성되었다.

컨트롤러 및 서비스 리소스를 정리하자.
```
$ kubectl delete -f myapp-sts.yaml -f myapp-sts-vol.yaml -f myapp-svc-headless.yaml

statefulset.apps "myapp-sts" deleted
statefulset.apps "myapp-sts-vol" deleted
service "myapp-svc-headless" deleted
```

컨트롤러 및 파드가 종료된 후 PV 및 PVC 리소스를 다시 확인해보자.
```
$ kubectl get persistentvolumeclaims
NAME                         ...
myapp-data-myapp-sts-vol-0   ...
myapp-data-myapp-sts-vol-1   ...
myapp-data-myapp-sts-vol-2   ...
```

```
$ kubectl get persistentvolume
NAME                                       ...
pvc-4bfee1c1-e529-46ae-a08b-0bcb2ccb8c4f   ...
pvc-b4be8ea1-de19-4148-b2d1-5ca7ec303e01   ...
pvc-f9424efb-2349-4ba4-986a-3ac2bea3ba3f   ...
```
컨트롤러 및 파드가 삭제된 후에도 여전히 PV 및 PVC 리소스는 남아있다. 데이터의 안전을 보장하기 위한 조치로 불필요한 경우 수동으로 삭제해야 한다.

PVC 리소스를 삭제하자.
```
$ kubectl delete persistentvolumeclaims --all
persistentvolumeclaim "myapp-data-myapp-sts-vol-0" deleted
persistentvolumeclaim "myapp-data-myapp-sts-vol-1" deleted
persistentvolumeclaim "myapp-data-myapp-sts-vol-2" deleted
```

> 참고
> 회수 정책이 Delete로 PVC만 삭제하면 PV도 같이 삭제된다.
