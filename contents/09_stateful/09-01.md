## 1. 스테이트풀셋 소개

### 1) 기존 컨트롤러의 문제점
여기서 기존 컨트롤러라고 하면 레플리케이션 컨트롤러나 레플리카셋과 같은 복제본을 가지고 있는 컨트롤러를 의미한다. 다음 설명에서 부터는 상징적으로 레플리카셋 컨트롤러로 대표해서 얘기한다.

레플리카셋은 파드 템플릿(replicaset.spec.template)에서 파드의 복제본을 생성한다. 다음은 레플리카셋의 파드 템플릿의 예다.
```yaml
...
  template:
    metadata:
      labels:
        app: cccr
    spec:
      containers:
      - name: web-server
        image: nginx:alpine
        volumeMounts:
        - name: web-content
          mountPath: /usr/share/nginx/html
      volumes:
      - name: web-content
        hostPath:
          type: Directory
          path: /web_contents
```
레플리카셋이 파드 복제본을 생성할 때 파드의 이름 및 IP 주소만 다를 뿐 나머지는 모두 똑같은 파드를 생성하게 된다. 만약 파드가 PVC를 참조한다면, 이 역시 똑같은 PVC를 연결하게 된다(가능하다면). 해당 PVC는 특정한 하나의 PV에 연결 되어 있다는 것은 이미 알고 있을것이다. 즉, 항상 똑같은 볼륨에 연결한다는 의미다.

기존 컨트롤러에서 각 파드가 분리된 저장소 볼륨을 사용해야 한다면 어떻게 해야 할까? (예: 데이터베이스 파드 및 스토리지)

#### (1) 여러 파드를 각각 별도로 관리
모든 파드를 수동 생성 및 관리하는 방법이 있을 수 있다. 각각 파드를 다른 볼륨 및 PVC를 참조하는 파드를 생성하는 방법이다.

| 파드   | PVC   | PV   |
|-------|-------|------|
| POD A | PVC A | PV A |
| POD B | PVC B | PV B |
| POD C | PVC C | PV C |

그러나 노드나 파드의 문제 발생 시 모든 대응은 수동으로 해야 한다. 이 방법은 적절한 해결책이 될 수 없다.

#### (2) 여러 컨트롤러 구성
첫 번째 방법의 문제를 해결하기 위해 각각 컨트롤러를 구성할 수 있다. 각각의 컨트롤러가 각각의 PVC를 사용하는 방식이다.

| RS   | POD   | PVC   | PV   |
|------|-------|-------|------|
| RS A | POD A | PVC A | PV A |
| RS B | POD B | PVC B | PV B |
| RS C | POD C | PVC C | PV C |

각 레플리카셋의 복제본 개수를 1로 설정하고 하나의 파드만 운용할 수 있도록 구성할 수 있다. 각 파드는 각각의 PVC에 연결 한다.
이 방법을 이용하면 노드의 문제나 파드의 장애에 대비할 수는 있지만, 각각의 레플리카셋은 별개의 리소스로 서로 연동되지 않는 레플리카셋을 운영해야한다는 단점이 있다. 규모를 늘리려면 별도의 레플리카셋을 생성해야 한다.

#### (3) 하나의 컨트롤러와 볼륨 디렉토리 분리
두 번째 방법의 문제를 해결하기 위해 하나의 컨트롤러만 운영하고, 각 파드가 하나의 PVC를 사용하는 대신 별도의 디렉토리를 마운트 하는 방법이다.

| RS   | POD   | PVC   | PV   | Dir |
|------|-------|-------|------|-----|
|      | POD A |       |      | /A  |
| RS A | POD B | PVC A | PV A | /B  |
|      | POD C |       |      | /C  |

하나의 레플리카셋은 하나의 파드 템플릿을 사용하기 때문에 같은 볼륨을 연결한다. 만약 가능하다면(?) 각 파드가 별도의 디렉토리에 별도의 데이터를 사용할 수 있도록 할 수 있다. 그러나 이는 일반적으로 가능하지 않을 뿐만 아니라, 가능하다 하더라도 수동으로 관리해줘야 한다.

### 2) 스테이트풀셋 소개
레플리케이션 컨트롤러나 레플리카셋이 제공하는 파드는 각 별도의 볼륨을 사용할 수 있는 방법을 제공해 주지 않아 모두 같은 볼륨으로 같은 상태를 가질수 밖에 없다. 파드마다 각각 다른 스토리지를 사용해 각각 다른 상태를 유지하기 위해서는 스테이트풀셋(StatefulSet) 리소스를 이용하면 된다.

#### (1) 가축과 애완동물
갑자기 쿠버네티스를 얘기하면서 가축과 애완동물이 나와서 이상할 수 있다. 그러나 스테이트풀셋의 초기 이름이 펫셋(PetSet)인걸 감안하면 이에 대한 얘기가 필요하다는 것을 느낄 수 있을 것이다.

레플리케이션 컨트롤러나 레플리카셋이 관리하는 파드는 가축에 비유할 수 있다.

레플리케이션 컨트롤러나 레플리카셋은 상태를 저장하지 않는 스테이트리스(Stateless)이기 때문에 항상 실행하는 파드의 정보가 똑같을 필요가 없다. 그래서 농부가 가축이 죽거나 병에 걸리면 필요시에 얼마든지 다른 가축을 사서 교체할 수 있는(슬프지만) 형태이다. 교체된 가축은 기존의 가축과 이름도 다르고(이름은 랜덤하게 붙음) 행동(IP)도 다를 수 있다.

스테이트풀셋이 관리하는 파드는 애완동물과 같아 애완동물이 죽으면 다른 애완동물로 대체 할 수 없다. 애완동물을 대체하기 위해서는(현실 세계에서 는 불가능하지만) 생김이나 행동 등 원래 있던 애완동물과 똑같아야한다. 스테이트풀셋은 문제가 생긴 파드와 완벽하게 똑같은 파드로 대체한다. 즉, 똑같은 이름과 똑같은 IP를 가진 파드로 교체한다는 의미다.

#### (2) 스테이트풀셋 이란?
스테이트풀셋은 컨테이너 애플리케이션의 상태를 관리하는데 사용하는 컨트롤러다.

디플로이먼트 컨트롤러와 같이 파드를 배포하고 복제본을 제공하며 스케일링을 관리할 수 있다. 그러나 디플로이먼트와 다른점은 파드의 순서 및 파드의 고유성을 보장하며, 동일한 스펙으로 생성되지만 각각 고유한 볼륨을 가지고 있다.

요약하면 다음과 같다.
- 파드 배포
- 파드의 복제본
- 스케일링
- 파드의 순서
- 파드의 고유성 (이름, 네트워크, 스토리지)
- 각 파드의 고유한 볼륨

#### (3) 스테이트풀셋의 주의사항
스테이트풀셋 컨트롤러 사용시 주의해야 할 사항은 다음과 같다.
- 파드에 사용할 스토리지는 PVC를 통해서만 가능
  * 미리 PV를 생성해 놓거나
  * StorageClass를 사용해 동적 프로비저닝 사용 
- 스테이트풀셋을 삭제하거나 파드를 삭제하더라도 볼륨은 삭제되지 않음
  * 데이터의 안전을 보장하기 위함 
- 헤드리스 서비스가 필요
  * 파드의 고유한 네트워크 신원을 제공하기 위함

#### (4) 스테이트풀셋의 파드 이름
스테이트풀셋의 각 파드 이름은 컨트롤러의 이름에 0 부터 시작하는 순서 색인이 붙게된다.

- 파드의 이름 형식
```
{StatefulSet-Name}-{Order}
```

- 파드 이름 예
```
mysts-0
mysts-1
```

#### (5) 스테이트풀셋의 파드 DNS 주소
헤드리스 서비스와 스테이트풀셋을 같이 사용하는 경우 파드의 DNS 주소는 다음과 같다.

- 파드 DNS 주소 형식
```
{Pod_Name}.{Governing_Service_Domain}.{Namespace}.svc.cluster.local
```

- 파드 DNS 주소 예
```
mysts-0.mysts.default.svc.cluster.local
mysts-1.mysts.default.svc.cluster.local
```

{Governing_Service_Domain}은 statefulset.spec.serviceName에 선언하며 해당 필드에 헤드리스 서비스의 이름을 지정한다.

#### (6) 스테이트풀셋의 스토리지 볼륨
스테이트풀셋의 파드는 각각 고유한 PVC를 생성해 고유한 PV를 가진다. statefulset.spec.volumeClaimTemplates 필드에 선언하며, 미리 PV를 준비하거나, StorageClass를 통해 PV를 생성할 수 있다.

스테이트풀셋 컨트롤러아 스테이트풀셋의 파드가 삭제되더라도 해당 볼륨은 안정적인 데이터 보존을 위해 자동으로 삭제되지 않는다.

#### (7) 스테이트풀셋의 스케일링
- 3개의 복제본이 있는 스테이트풀셋 파드는 0 -> 1 -> 2 순서대로 생성됨
- 파드를 스케일 아웃 하기 전 기존 파드는 Running 및 Ready 상태여햐 함
- 파드가 스케일 인에 의해 삭제될 때는 역순으로 삭제됨
 
#### (8) 스테이트풀셋 및 헤드리스 서비스 예시
```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  clusterIP: None
  ports:
  - name: web
    port: 80
  selector:
    app: nginx
```

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  selector:
    matchLabels:
      app: nginx 
  serviceName: "nginx"
  replicas: 3 
  template: # Pod Template
    metadata:
      labels:
        app: nginx 
    spec:
      terminationGracePeriodSeconds: 10
      containers:
      - name: nginx
        image: k8s.gcr.io/nginx-slim:0.8
        ports:
        - containerPort: 80
          name: web
        volumeMounts:
        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates: # Persistent Volume Claim Template
  - metadata:
      name: www
    spec:
      accessModes: [ "ReadWriteOnce" ]
      storageClassName: "my-storage-class"
      resources:
        requests:
          storage: 1Gi
```
