## 1. 파드의 리소스 관리
지금까지 생성한 파드는 리소스(CPU 및 메모리)에 관한 정의를 따로 하지 않았다. 따라서 파드(컨테이너)는 파드에서 실행된 프로세스가 사용하는 양만큼 CPU 및 메모리를 제한 없이 사용할 수 있다. 또한 여러 파드가 실제 가지고 있는 하드웨어 리소스 보다 더 많은 리소스를 요구하는 경우 프로세스는 서로 많은 리소스를 사용하기 위해 경합하게 되며, 이 때문에 전체 성능이 떨어질 수도 있다. 심지어 필요한 리소스 양을 받지 못해 종료되거나 동작을 못하는 파드도 생겨날 수 있다.

쿠버네티스는 파드 및 컨테이너에 최소 필요한 리소스의 양을 요청할 수 있는 "리소스 요청(Request)"을 구성할 수 있고, 최대 사용할 수 있는 리소스 양을 제한할 수 있는 "리소스 제한(Limit)"을 구성할 수 있다. 실제 쿠버네티스의 기능이라기 보단 컨테이너 런타임의 기능이다.

![파드의 리소스 요청과 제한](img/11_resource_reqlim.png)

### 1) 리소스 요청
리소스 요청(Request)은 특정 파드/컨테이너에만 전용으로 사용할 수 있는 리소스를 보장한다. 즉, 다른 리소스가 사용하지 못하게 예약된 리소스다.

#### (1) 현재 리소스 사용량 확인
쿠버네티스 클러스터 노드의 리소스 사용량 확인
```
$ kubectl top nodes

NAME           CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
kube-master1   290m         16%    1640Mi          48%
kube-node1     275m         14%    1051Mi          63%
kube-node2     220m         11%    1121Mi          68%
kube-node3     177m         9%     939Mi           56%
```

파드의 리소스 사용량 확인
```
$ kubectl top pod mynapp-pod-req

NAME             CPU(cores)   MEMORY(bytes)
mynapp-pod-req   0m           5Mi
```
아직 파드를 생성하지 않았기 때문에 조금 있다가 다시 확인해보자.

kubectl top 명령은 리소스 사용량을 확인할 수 있으며, 쿠버네티스 오브젝트 리소스가 생성되고 난 후 모니터링하기 까지 시간이 조금 걸린다. 즉, 새로 생성한 파드의 경우 시간적 여유를 두고 확인하자.

#### (2) 리소스 요청 정의
다음은 리소스 요청에 대한 오브젝트다.
```yaml
spec:
  containers: 
  - resources:
      requests:
        cpu: 200m
        memory: 10Mi
```

- spec.containers.resources.requests: 리소스 요청 선언
- spec.containers.resources.requests.cpu: CPU 리소스 요청
- spec.containers.resources.requests.memory: 메모리 리소스 요청


cpu 필드는 크게 두 가지 방식으로 선언할 수 있는데, 숫자만 표시하는 경우 코어(쓰레드)의 수를 지정할 수 있으며, 200m의 m은 밀리코어(millicore)를 의미하여 코어 하나가 1000m 밀리코어 이다. 즉, 200m은 코어 하나의 1/5을 의미한다. 숫자로 지정하는 경우 1은 코어 하나, 1.5는 코어 1.5개, 즉 밀리코어 단위로는 1500m이 된다.

memory 필드는 요청하는 메모리 양을 의미하며 아래와 같이 지정할 수 있다., M | G | T | P | E | Mi | Gi | Ti | Pi | Ei 로 지정할 수 있다.
- SI 표준 단위: k, M, G, T, P, E
- IEC 표준 단위: Ki, Mi, Gi, Ti, Pi, Ei

#### (3) 리소스 요청 생성
다음은 리소스 요청을 선언한 파드 리소스다.

> mynapp-pod-req.yml 파일
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-req
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
    resources:
      requests:
        cpu: 200m
        memory: 10Mi
```
CPU 요청은 코어 하나의 200m 밀리코어(0.2)를 요청하였고, 메모리는 10Mi 만큼을 요청하였다.

쿠버네티스는 스케줄러는 파드를 배포하기 전 요청 사항만큼 수용할 수 있는 노드를 찾아서 해당 노드에 파드를 생성하게 된다. 만약 요청 사항만큼을 실행할 수 있는 노드가 없다면 당연히 파드는 실행될 수 없다.

파드를 생성해 보자.
```
$ kubectl create -f mynapp-pod-req.yml

pod/mynapp-pod-req created
```

파드의 리소스 사용량을 확인해 보자.

- Allocated resources: 현재 노드에 배치된 모든 파드의 총 리소스 요청량 및 제한량을 확인할 수 있다.

#### (5) 많은 리소스 요청을 요구하는 파드
매우 큰 용량을 요청하는 파드를 생성해보자.

> mynapp-pod-huge-req.yml 파일
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-huge-req
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
    resources:
      requests:
        cpu: 4000m
        memory: 4Gi
```
CPU 리소스 요청은 4000m, 메모리는 4Gi 이다. 

현재 작업 하고있는 쿠버네티스 클러스터에서는 수용할 수 있는 노드가 없다. 만약 노드의 리소스가 충분하다면 리소스 요청량을 더 늘려서 테스트 한다.

파드를 생성해보자.
```
$ kubectl create -f mynapp-pod-huge-req.yml

pod/mynapp-pod-huge-req created
```

파드의 상태를 확인해 보자.
```
$ kubectl get pods

NAME                  READY   STATUS    RESTARTS   AGE
mynapp-pod-huge-req   0/1     Pending   0          63s
mynapp-pod-req        1/1     Running   0          12m
```
아무리 기다려도 준비 상태나 동작하지 않는다.

파드가 실행되지 않는 원인을 확인해보자.
```
$ kubectl describe pod mynapp-pod-huge-req

...
Events:
  Type     Reason            Age        From               Message
  ----     ------            ----       ----               -------

  Warning  FailedScheduling  <unknown>  default-scheduler  0/4 nodes are available: 4 Insufficient cpu, 4 Insufficient memory.
```
이벤트 필드에 확인해보면 4개 노드중 CPU 및 메모리 모두 충분하지 않다고 보고하고 있다.

### 2) 리소스 제한
리소스 제한(Limit)은 파드/컨테이너가 사용할 수 있는 한계를 지정한다. 

리소스 제한은 리소스 요청이 있는 경우 요청한 리소스와 같거나 그보다 더 커야한다. 
> 리소스 제한 >= 리소스 요청  

#### (1) 리소스 제한 정의
리소스 제한은 다음과 같이 선언할 수 있다.

```yaml
spec:
  containers: 
  - resources:
      limits:
        cpu: 1
        memory: 20Mi
```

- spec.containers.resources.limits: 리소스 제한 선언
- spec.containers.resources.limits.cpu: CPU 리소스 제한
- spec.containers.resources.limits.memory: 메모리 리소스 제한

#### (2) 리소스 제한 생성
다음은 리소스 제한을 선언한 파드 리소스이다.

> mynapp-pod-limits.yml 파일
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-lim
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
    resources:
      limits:
        cpu: 0.5
        memory: 20Mi
```
CPU는 최대 0.5 코어만큼 제한하였고, 메모리는 20Mi 만큼 제한을 설정했다.

> 참고  
> 리소스 요청 없이 리소스 제한만 선언하는 경우 리소스 요청은 리소스 제한만큼 할당된다.

파드를 생성해보자.
```
$ kubectl create -f mynapp-pod-limits.yml

pod/mynapp-pod-lim created
```

#### (3) 리소스 제한 확인
파드가 생성되었는지 확인해보자.
```
$ kubectl get pods mynapp-pod-lim

NAME             READY   STATUS    RESTARTS   AGE
mynapp-pod-lim   1/1     Running   0          19s
```

현재 파드가 사용하는 리소스를 확인해 보자.
```
$ kubectl top pods mynapp-pod-lim

NAME             CPU(cores)   MEMORY(bytes)
mynapp-pod-lim   0m           5Mi
```

별도의 터미널에서 다음 명령을 실행시켜 파드에 일부러 부하 주자.
```
$ kubectl exec mynapp-pod-lim -- sha1sum /dev/zero
```
컨테이너에서 실행할 명령은 sha1sum 해시를 계산하는 도구이며 /dev/zero 장치의 해시를 계산함으로 부하를 최대한 늘리게 된다.

다시 파드의 리소스 사용량을 확인해보자.
```
$ kubectl top pods mynapp-pod-lim

NAME             CPU(cores)   MEMORY(bytes)
mynapp-pod-lim   500m         5Mi
```
CPU 리소스 제한이 500m 밀리코어로 제한되어 있는데, 정확히 500m 밀리코어를 넘지않고 있다.

#### (4) 리소스 삭제
```
$ kubectl delete pods --all

pod "mynapp-pod-huge-req" deleted
```

### 3) 리소스 제한범위
리소스 제한범위(LimitRange)는 

#### (1) 제한범위 생성
> mynapp-limitrange.yml 파일
```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: mynapp-limitrange
spec:
  limits:
  - type: Pod
    min:
      cpu: 50m
      memory: 5Mi
    max:
      cpu: 1
      memory: 1Gi
  - type: Container
    defaultRequest:
      cpu: 100m
      memory: 10Mi
    default:
      cpu: 200m
      memory: 100Mi
    min:
      cpu: 50m
      memory: 5Mi
    max:
      cpu: 1
      memory: 1Gi
    maxLimitRequestRatio:
      cpu: 4
      memory: 10
  - type: PersistentVolumeClaim
    min:
      storage: 10Mi
    max:
      storage: 1Gi
```

- limitrange.spec.limits.type: 리소스 제한을 적용할 리소스 타입
- limitrange.spec.limits.default: 리소스 제한 기본 기본량 (명시적으로 선언하지 않는경우)
- limitrange.spec.limits.defaultRequest: 리소스 요청 기본 기본량 (명시적으로 선언하지 않는경우)
- limitrange.spec.limits.min: 리소스 제한을 요청할 수 있는 최소량
- limitrange.spec.limits.max: 리소스 제한을 요청할 수 있는 최대향
- limitrange.spec.limits.maxLimitRequestRatio: 리소스 요청 및 제한 간의 최대 비율

제한범위를 생성해 보자.
```
$ kubectl create -f mynapp-limitrange.yml

limitrange/mynapp-limitrange created
```

#### (2) 제한범위 확인
제한범위가 생성되었는지 확인해 보자.
```
$ kubectl get limitranges
NAME                CREATED AT
mynapp-limitrange   2020-03-19T06:19:25Z
```

제한범위의 상세정보를 확인해 보자.
```
$ kubectl describe limitranges mynapp-limitrange
Name:        mynapp-limitrange
Namespace:   default
Type         Resource  Min   Max  Default Request  Default Limit  Max Limit/Request Ratio
----         --------  ---   ---  ---------------  -------------  -----------------------
Pod          cpu       50m   1    -                -              -
Pod          memory    5Mi   1Gi  -                -              -
Container    cpu       50m   1    100m             200m           4
Container    memory    5Mi   1Gi  10Mi             100Mi          10
...
```

#### (3) 리소스 요청 및 제한이 없는 파드
다음은 리소스 요청 및 제한이 설정되지 않은 파드 리소스다.

> mynapp-pod-no-reqlim.yml
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: mynapp-pod-no-reqlim
spec:
  containers:
  - image: c1t1d0s7/myweb
    name: mynapp
```

파드를 생성해 보자.
```
$ kubectl create -f mynapp-pod-no-reqlim.yml

pod/mynapp-pod-no-reqlim created
```

파드의 자세한 정보를 확인해 보자.
```
$ kubectl describe pods mynapp-pod-no-reqlim

...
    Limits:
      cpu:     200m
      memory:  100Mi
    Requests:
      cpu:        100m
      memory:     10Mi
...
```
파드에 별도의 리소스 요청 및 제한 사항을 정의하지 않았지만, 제한범위의 기본 설정에 의해 리소스 제한 및 요청이 할당된 것을 확인할 수 있다.

#### (4) 많은 리소스 요청을 요구하는 파드
앞서 많은 리소스를 요청하는 파드를 다시 생성해보자.
```
$ kubectl create -f mynapp-pod-huge-req.yml

The Pod "mynapp-pod-huge-req" is invalid:
* spec.containers[0].resources.requests: Invalid value: "4": must be less than or equal to cpu limit
* spec.containers[0].resources.requests: Invalid value: "4Gi": must be less than or equal to memory limit
```
이전과 다르게 이번에는 리소스 생성시 부터 오류가 난다. 즉, 리소스 제한 및 요청량이 최대 정의할 수 있는 범위를 벗어났다는 의미이다.

#### (5) 리소스 삭제
```
$ kubectl delete limitranges mynapp-limitrange 

limitrange "mynapp-limitrange" deleted
```
```
$ kubectl delete pod mynapp-pod-no-reqlim

pod "mynapp-pod-no-reqlim" deleted
```

### 4) 리소스 쿼터
리소스 쿼터(ResourceQuota)는 네임스페이스에서 오브젝트 생성을 제한할 수 있다.

#### (1) CPU 및 메모리 쿼터
다음은 CPU 및 메모리 쿼터를 정의한 파일이다.

> mynapp-quota-cpumem.yml 파일
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: mynapp-quota-cpumem
spec:
  hard:
    requests.cpu: 500m
    requests.memory: 200Mi
    limits.cpu: 1000m
    limits.memory: 1Gi
```

#### (2) 오브젝트 쿼터
다음은 생성할 수 있는 오브젝트 개수에 대한 쿼터 정의 파일이다.

> mynapp-quota-object.yml 파일
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: mynapp-quota-object
spec:
  hard:
    pods: 10
    replicationcontrollers: 2
    secrets: 10
    configmaps: 10
    persistentvolumeclaims: 5
    services: 5
    services.loadbalancers: 1
    services.nodeports: 2
    glusterfile.storageclass.storage.k8s.io/persistentvolumeclaims: 2
```

#### (3) 스토리지 쿼터
스토리지 쿼터 정의 파일이다.

> mynapp-quota-storage.yml 파일
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: mynapp-quota-storage
spec:
  hard:
    requests.storage: 10Gi
    glusterfile.storageclass.storage.k8s.io/requests.storage: 5Gi
```

#### (4) 리소스 쿼터 생성 및 확인
리소스 쿼터를 생성해보자.
```
$ kubectl create -f mynapp-quota-cpumem.yml -f mynapp-quota-object.yml -f mynapp-quota-storage.yml

resourcequota/mynapp-quota-cpumem created
resourcequota/mynapp-quota-object created
resourcequota/mynapp-quota-storage created
```

생성된 리소스 쿼터를 확인해보자.
```
$ kubectl get resourcequotas
```
