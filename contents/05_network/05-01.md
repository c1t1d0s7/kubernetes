## 1. 클러스터 내부 서비스
파드는 클러스터 외부의 요청이나 클러스터 내부의 다른 파드의 요청에 응답해야 한다. 또한 파드가 다른 파드에서 제공하는 애플리케이션을 사용하기 위해서는 다른 파드를 찾을 수 있어야 한다.

쿠버네티스가 아닌 기존의 시스템은 애플리케이션이 동작하는 시스템의 호스트 이름이나 정적 IP를 할당하여 해당 애플리케이션을 찾을 수 있지만, 쿠버네티스 환경에서는 그렇게 할 수 없다.

이유는 다음과 같다.
- 파드는 일회성으로 동작하기 위해 설계되었다. 언제든지 제거될 수 있다.
- 특정 노드에 파드가 스케줄링 되고 IP 주소가 동적으로 할당된다. 클라이언트는 파드의 IP 주소를 알 수 없다.
- 분산 아키텍처 및 수평 스케일링의 경우 여러 파드가 같은 애플리케이션을 제공한다. 각 파드 마다 IP가 존재하고, 스케일링이 될 때마다 클라이언트가 해당 IP를 알 수 없다. 

이를 해결하려면 파드는 단일 고정 IP를 통해서 서비스를 제공해야 한다.

### 1) 서비스 소개
서비스(Service)는 쿠버네티스 시스템에서 같은 애플리케이션을 실행하고 있는 컨트롤러의 파드 그룹에 단일 네트워크 진입점을 제공하는 리소스다. 서비스에 부여된 IP는 해당 서비스가 종료될 때까지 변경되지 않는다.

클라이언트는 서비스가 제공하는 고정 IP 및 포트를 통해 파드에 접근하게 된다.

#### kubernetes 서비스
default 네임스페이스에 처음부터 항상 존재하던 서비스를 기억할 것이다.
```
$ kubectl get services
NAME         TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
kubernetes   ClusterIP   10.233.0.1   <none>        443/TCP   6m10s
```
이 서비스는 쿠버네티스 아키텍처에서 살펴본 마스터의 API 서버(kube-apiserver)로 접근할 수 있는 서비스다.

### 2) 서비스 생성
서비스 생성은 두 가지 방법이 있다. 첫 번째는 kubectl expose 명령을 이용하는 방법이고, 이는 앞서 이미 살펴보았다. 두 번째는 YAML 파일로 리소스를 생성하는 방법이다.

#### (1) 명령을 이용하는 방법
```
kubectl expose <CONTROLLER_TYPE> <CONTROLLER_NAME> [--type=<SVC_TYPE>] --name <SVC_NAME>
```

#### (2) YAML 파일을 이용하는 방법
> mynapp-svc.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc
spec:
  ports:
  - port: 80 # 서비스 포트
    targetPort: 8080 # 파드의 포트
  selector:
    app: mynapp-rs # 파드 레이블 셀렉터
```
서비스 오브젝트의 API는 v1을 사용하며 종류는 Service 이다. 

service.spec.ports.port 필드는 클라이언트가 접근할 서비스의 노출 포트이며, 해당 포트로 접근하면 서비스는 service.spec.ports.targetPort 필드에 지정된 파드의 서비스 포트로 포워딩하게 된다.

service.spec.selector 필드는 서비스에 연결할 파드를 레이블 셀렉터를 이용하여 매칭 한다.

서비스를 생성해보자.
```
$ kubectl create -f mynapp-svc.yml

service/mynapp-svc created
```

### 3) 서비스 및 엔드포인트 확인

#### (1) 서비스 목록 확인
kubectl get services 명령을 이용하여 서비스 목록을 확인할 수 있다.
```
$ kubectl get services
NAME         TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
...
mynapp-svc   ClusterIP   10.233.29.60   <none>        80/TCP    9s
```
서비스 타입은 ClusterIP 이며, 이는 쿠버네티스 클러스터 내에서만 접근할 수 있는 타입이다. 서비스에 할당된 클러스터 IP(CLUSTER-IP)는 10.233.29.60 이다.

클러스터 외부에서 접근할 때는 외부 IP(EXTERNAL-IP)를 사용해야하며 이는 서비스 타입에 따라서 할당되는 서비스 타입이 있다. 즉, 지금 현재 서비스는 클러스터 내부에서만 접근할 수 있는 서비스이다.

#### (2) 엔드포인트 확인
엔드포인트는 서비스의 레이블 셀렉터에 의해 연결된(포워딩 할) 파드의 IP 목록을 엔드포인트 목록이라고 한다. kubectl get endpoints 명령으로 엔드포인트를 확인할 수 있다.
```
$ kubectl get endpoints
NAME         ENDPOINTS            AGE
...
mynapp-svc   <none>               79s
```

> 참고  
> 엔드포인트 이름은 서비스의 이름과 동일하다.

현재 app=mynapp-rs 레이블을 가진 파드가 없기 때문에 엔드포인트 목록도 가지고 있지 않다. 즉, 서비스의 클러스터 IP로 접속하더라도 포워딩 할 파드가 없다.

### 4) 파드 생성 및 엔드포인트 연결
해당되는 레이블을 가진 파드를 생성하면 서비스는 자동으로 엔드포인트를 추가하게 된다. 즉, 나중에 파드가 추가되거나 제거되더라도 레이블 셀렉터에 의해 매칭되는 파드는 엔드포인트가 자동으로 추가 및 제거된다.

앞서 만든 mynapp-rs.yml 파일을 이용하여 파드를 생성해보자. 해당 컨트롤러 및 파드는 app=mynapp-rs 레이블을 사용하고 있다.
```
$ kubectl create -f mynapp-rs.yml

replicaset.apps/mynapp-rs created
```

파드를 생성한 이후 엔드포인트를 다시 확인해보자.
```
$ kubectl get endpoints mynapp-svc

NAME         ENDPOINTS                                                 AGE
mynapp-svc   10.233.101.14:8080,10.233.103.15:8080,10.233.76.14:8080   3m40s
```
10.233.101.14:8080, 10.233.103.15:8080, 10.233.76.14:8080 세 개의 엔드포인트가 구성되었으며, 이는 파드의 IP의 포트이다.

파드의 목록과 엔트포인트를 비교 해보자.
```
$ kubectl get pods -o wide

NAME              READY   STATUS    RESTARTS   AGE    IP              NODE         ...
mynapp-rs-m65m4   1/1     Running   0          109s   10.233.76.14    kube-node3   ...
mynapp-rs-t5wqf   1/1     Running   0          109s   10.233.103.15   kube-node2   ...
mynapp-rs-t8chb   1/1     Running   0          109s   10.233.101.14   kube-node1   ...
```
-o wide 옵션을 사용하면 파드에 할당된 IP를 확인할 수 있다.

### 5) 서비스 접근 테스트
이제 서비스와 서비스가 포워딩 할 파드가 준비되었다. 이제 클러스터 내의 서비스에 접근하기 위한 테스트용 파드를 생성해 서비스 접근을 확인해보자.

```
$ kubectl run nettool -it --image=c1t1d0s7/network-multitool --generator=run-pod/v1 --rm=true bash

If you don't see a command prompt, try pressing enter.
bash-5.0# curl http://10.233.29.60

Platform: linux
Uptime: 104738
IP: 10.233.101.14
DNS: 169.254.25.10

bash-5.0# curl http://10.233.29.60
Message: Hello World!
Hostname: mynapp-rs-t5wqf
Platform: linux
Uptime: 104698
IP: 10.233.103.15
DNS: 169.254.25.10

bash-5.0# curl http://10.233.29.60
Message: Hello World!
Hostname: mynapp-rs-m65m4
Platform: linux
Uptime: 104666
IP: 10.233.76.14
DNS: 169.254.25.10

bash-5.0# exit
exit
Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
kubectl run 명령을 이용하여 테스트용 파드를 실행한다. 파드의 이름은 nettool이고 사용한 이미지는 c1t1d0s7/network-multitool이다. 해당 이미지는 alpine 리눅스에 네트워크를 테스트하기 위한 여러 도구가 설치 되어있는 이미지다.

--generator=run-pod/v1 명령을 이용하여 파드 만 생성했으며, --rm=true 명령을 이용하여 애플리케이션(여기서는 bash)이 종료되면 파드를 자동 삭제하도록 했다. 여기서는 테스트를 하기위한 파드 이기 때문에 계속해서 작동시킬 필요가 없다.

> 참고  
> 앞서 살펴 봤듯이 kubectl run 명령에 --generator=run-pod/v1 옵션을 사용하지 않으면, 기본적으로 디플로이먼트 오브젝트로 파드를 생성하게 된다.

파드를 실행 후 bash 쉘이 실행되며 curl 명령으로 서비스에 할당된 IP로 접근하면 기본적으로 부하 분산이 이루어지며, 파드 그룹에 접근할 수 있다.

### 6) 서비스의 세션 어피니티 구성
앞서 살펴본 서비스 접근 테스트에서 특정한 하나의 클라이언트가 웹 요청을 할 때마다, 부하 분산을 통해 다른 파드로 연결되는 것을 확인할 수 있다.

그러나 클라이언트 요청을 매번 똑같은 파드로 연결하고 싶은 경우나 그렇게 해야 할 경우가 있다. 예를 들어 세션을 계속 유지하지 위한 이유이다. 이것을 세션 어피니티(Session Affinity)라고 한다.

서비스 오브젝트의 service.spec.sessionAffinity 필드로 구성할 수 있다.
> mynapp-svc-session-affinity.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-ses-aff
spec:
  sessionAffinity: ClientIP
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: mynapp-rs
```
service.spec.sessionAffinity 필드의 속성 값으로는 None과 ClientIP 두 가지 종류의 세션 어피니티 구성이 있으며, 기본 값은 None이다. 세션 어피니티 설정은 ClientIP로 설정하면, 쿠버네티스 클러스터의 프록시(kube-proxy)는 클라이언트의 IP를 확인하여 매번 같은 파드로 연결해 준다.

service.spec.sessionAffinity:
- None: (기본) 세션 어피니티 없음
- ClientIP: 클라이언트의 IP를 확인해 같은 파드로 연결됨

### 7) 서비스 다중 포트 구성
파드의 애플리케이션은 항상 하나의 포트로만 서비스 하는 것은 아니다. 즉 여러 포트를 사용하여 서비스를 할 수 있다. 서비스 역시 파드의 서비스 포 트가 다중 포트인 경우, 노출할 서비스 포트 역시 다중 포트로 구성할 수 있다.

예를 들어 웹 서버인 경우 HTTP는 80번, HTTPS는 443번 다중 포트를 가진 파드 및 서비스를 구성해야 할 경우도 있다.

다중 포트 구성은 이전 서비스 오브젝트 파일을 살펴보면, 서비스 포트가 단일 포트라고 하더라도, YAML의 리스트로 구성된 것을 확인할 수 있다. 즉 다중 포트는 리스트를 추가하여 구성할 수 있다. 다음은 다중 포트를 구성하기 위한 서비스 오브젝트 예제이다.

> mynapp-svc-multiport.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-mp
spec:
  ports:
  - name: mynapp-http
    port: 80
    targetPort: 8080
  - name: mynapp-https
    port: 443
    targetPort: 8443
  selector:
    app: mynapp-rs
```
다중 포트를 설정할 때는 반드시 포트의 이름을 부여해야 한다.

### 8) 포트 이름 참조
앞서 살펴 본대로 서비스의 포트에 이름을 부여할 수 있다. 파드나 파드를 생성하는 컨트롤러의 파드 템플릿의 컨테이너 포트에도 이름을 부여할 수 있다.

이렇게 하면 파드(컨테이너)의 포트 이름을 이용하여 서비스 생성 시 파드의 대상포트(targetPort)에 이름을 이용하여 연결할 수 있다. 이는 이름으로 참조하기 때문에 파드의 포트 이름만 변경되지 않았다면, 실제 파드의 포트번호가 변경되더라도, 스펙의 변경 없이 계속해서 연결할 수 있다. 즉, 엔드포인트도 변경된다.

#### (1) 포트 이름을 사용한 레플리카셋 및 서비스 생성
다음은 파드 및 컨트롤러에서 컨테이너의 포트에 이름을 부여하는 구성이다.

> mynapp-rs-named-port.yml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-nport
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-rs-nport
  template:
    metadata:
      labels:
        app: mynapp-rs-nport
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        ports:
        - name: mynapp-http
          containerPort: 8080
```

컨테이너의 8080 포트에 mynapp-http 라는 이름을 부여하였다. 이름을 서비스 생성 시 참조할 수 있다.

다음은 포트 이름을 참조하는 서비스 구성이다.
> mynapp-svc-named-port.yml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-nport
spec:
  ports:
  - name: mynapp-http
    port: 80
    targetPort: mynapp-http
  selector:
    app: mynapp-rs-nport
```
service.spec.ports.targetPort 필드에 포트번호 대신 이름을 지정하였다.

작성한 레플리카셋 컨트롤러와 서비스를 생성하자.
```
$ kubectl create -f mynapp-rs-named-port.yml -f mynapp-svc-named-port.yml

replicaset.apps/mynapp-rs-nport created
service/mynapp-svc-nport created
```
-f 옵션을 두 번 사용하여 한꺼번에 생성할 수 있다.

#### (2) 오브젝트 확인
레플리카셋 컨트롤러를 확인해보자.
```
$ kubectl get replicasets.apps

NAME              DESIRED   CURRENT   READY   AGE
...
mynapp-rs-nport   3         3         3       19s
```

엔드포인트를 확인해보자.
```
$ kubectl get endpoints mynapp-svc-nport

NAME               ENDPOINTS                                                 AGE
mynapp-svc-nport   10.233.101.16:8080,10.233.103.16:8080,10.233.76.15:8080   4m14s
```
정확하게 8080 포트를 참조하고 있는 것을 확인할 수 있다.

파드 목록 및 파드 IP를 확인해보자.
```
$ kubectl get pods -o wide -l app=mynapp-rs-nport

NAME                    READY   STATUS    RESTARTS   AGE     IP              NODE         ...
mynapp-rs-nport-6jv9f   1/1     Running   0          6m33s   10.233.76.15    kube-node3   ...
mynapp-rs-nport-c64z9   1/1     Running   0          6m33s   10.233.101.16   kube-node1   ...
mynapp-rs-nport-ljp8r   1/1     Running   0          6m33s   10.233.103.16   kube-node2   ...
```

#### (3) 포트 변경 및 확인
kubectl edit 명령을 이용하여 cccr 레플리카셋의 포트를 8081로 변경해 보자.
```
$ kubectl edit replicasets.apps mynapp-rs-nport

...
        ports:
        - containerPort: 8081
          name: cccr-http
          protocol: TCP
...
:wq

replicaset.apps/mynapp-rs-nport edited
```
replicaset.spec.template.spec.containers.ports.containerPort 필드를 8080 에서 8081 포트로 변경한다.

포트를 변경하고 난 후, 현재 동작중인 파드의 포트는 변경되지 않으므로, 기존의 파드를 삭제한다.
