## 6. 헤드리스 서비스

### 1) 헤드리스 서비스 소개
서비스는 기본적으로 파드 그룹을 서비스의 엔드포인트에 등록하고, 기본적으로 부하 분산을 한다.

그러나 항상 부하 분산이 좋은 것만은 아니다. 클라이언트 측 파드가 서버 측 서비스를 통해 각각 모든 개별 서버 파드로 접근해야할 필요도 있다.

서비스 리소스 자체의 IP(Cluster IP)가 아닌, 각각의 파드의 주소를 확인 하여 개별적으로 접근할 필요가 있다는 의미이다. 이를 헤드리스(Headless) 서비스라 한다.

예를 들어 데이터베이스를 실행하고 있는 컨트롤러에 의해 두 개의 데이터베이스 파드가 있고, 이 데이터베이스를 실행하고 있는 파드는 고가용성을 이유로 마스터(읽기/쓰기) 파드와 슬레이브(읽기 전용) 파드로 나눠져 있는 경우를 가정해 보자.

기존의 서비스는 부하 분산기능을 가지고 있기 때문에 어떤 파드(마스터 및 슬레이브)에 접근할 것인지 우리가 결정할 수 없고, 서비스에 의해 연결된 랜덤한 파드로 접근하게 된다. 이 경우 데이터베이스에 쓰기를 해야하는 경우는 어떻게 할 것인가? 이 경우에 헤드리스 서비스가 필요하다.

### 2) 헤드리스 서비스 생성
헤드리스 서비스 리소스 정의 파일이다.


> mynapp-svc-headless.yml 파일
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-headless
spec:
  clusterIP: None
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: mynapp-rs-headless
```
service.spec.clusterIP 필드를 None으로 설정하여 서비스 자체 IP를 가지지 않도록 한다.

다음은 헤드리스 서비스에 사용할 레플리케이션 컨트롤러이다.
> mynapp-rs-headless.yml 파일
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-headless
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-rs-headless
  template:
    metadata:
      labels:
        app: mynapp-rs-headless
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        ports:
        - containerPort: 8080
```

헤드리스 서비스와 컨트롤러를 생성하자.
```
$ kubectl create -f mynapp-rs-headless.yml -f mynapp-svc-headless.yml

replicaset.apps/mynapp-rs-headless created
service/mynapp-svc-headless created
```

### 3) 헤드리스 서비스 확인
서비스 목록 및 상태를 확인하자.
```
$ kubectl get services

NAME                  TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
kubernetes            ClusterIP   10.233.0.1   <none>        443/TCP   4h42m
mynapp-svc-headless   ClusterIP   None         <none>        80/TCP    22s
```
mynapp-svc-headless 서비스의 내부 IP(CLUSTER-IP)가 할당되어 있지 않은것을 확인할 수 있다.

헤드리스 서비스의 엔드포인트 목록을 확인해 보자.
```
$ kubectl get endpoints mynapp-svc-headless

NAME                  ENDPOINTS                                                 AGE
mynapp-svc-headless   10.233.101.22:8080,10.233.103.23:8080,10.233.76.23:8080   76s
```

네트워크 도구를 가진 테스트 파드를 생성해 서비스 주소를 조회 해보자.
```
$ kubectl run nettool -it --image=c1t1d0s7/network-multitool --generator=run-pod/v1 --rm=true bash

bash-5.0# nslookup mynapp-svc-headless
Server:		169.254.25.10
Address:	169.254.25.10#53

Name:	mynapp-svc-headless.default.svc.cluster.local
Address: 10.233.76.23
Name:	mynapp-svc-headless.default.svc.cluster.local
Address: 10.233.101.22
Name:	mynapp-svc-headless.default.svc.cluster.local
Address: 10.233.103.23

bash-5.0# exit
exit
Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
mynapp-svc-headless 서비스를 조회해 보면 각 파드의 주소가 각각 조회됨을 알수 있다.

조금더 자세한 내용은 "스테이트풀셋" 파트에서 확인 해보면 명확하게 이해할 수 있다.

### 4) 컨트롤러 및 서비스 삭제
```
$ kubectl delete -f mynapp-rs-headless.yml -f mynapp-svc-headless.yml

replicaset.apps "mynapp-rs-headless" deleted
service "mynapp-svc-headless" deleted
```
