## 3. 클러스터 외부 서비스
쿠버네티스 클러스터에서 웹의 프론트엔드 서비스를 실행하는 파드의 경우 쿠버네티스 클러스터의 외부로 노출시켜 접근 가능하도록 구성해야 한다.

지금까지 살펴본 서비스는 쿠버네티스 클러스터 내부에서 사용 가능하도록 구성된 서비스이다.

쿠버네티스 클러스터 외부에서 접근 가능한 서비스 리소스 출력의 필드를 살펴보자.
```
NAME         TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
kubernetes   ClusterIP   10.233.0.1   <none>        443/TCP   128m
```
서비스의 종류(TYPE)가 ClusterIP이고, 클러스터 내부 IP(CLUSTER-IP)만 할당되어 있고, 클러스터 외부 IP(EXTERNAL-IP)는 할당되어 있지 않다.

서비스의 종류는 다음과 같다. 
- ClusterIP
  * 클러스터 내부용 서비스
- NodePort
  * NodePort + ClusterIP 
  * 쿠버네티스 모든 노드(호스트)에 외부 접근용 포트를 할당함
  * 노드의 포트를 사용하여 외부에서 접근 가능
  * 노드의 포트로 접근하면 서비스에 의해 파드로 리다이렉션 함
  * 파드를 실행하고 있지 않는 노드에도 포트가 할당되고, 접근 가능함
- LoadBalancer
  * LoadBalancer + NodePort + ClusterIP 
  * NodePort의 확장
  * 클러스터 외부의 로드 밸런서를 사용하여 외부에서 접근 가능
  * 외부 로드 밸런서로 접근하면 서비스를 통해 파드로 리다이렉션 함
  * 클라우드 공급업체(AWS, GCP 등)나 MetalLB에서 지원
- ExternalName
  * 외부에서 접근하기 위한 종류 아님
  * 외부의 특정 FQDN에 대한 CNAME 매핑을 제공
  * 파드가 CNAME을 이용해 특정 FQDN과 통신하기 위함

### 1) 외부 서비스용 레플리카셋 생성 및 확인
기존에 작성된 myapp-rs 파일을 이용하여 컨트롤러 및 파드를 생성한다.
```
$ kubectl create -f myapp-rs.yaml

replicaset.apps/myapp-rs created
```

컨트롤러 및 파드가 생성되었는지 확인한다.
```
$ kubectl get replicasets.apps

NAME       DESIRED   CURRENT   READY   AGE
myapp-rs   3         3         3       14s
```

```
$ kubectl get pods

NAME             READY   STATUS    RESTARTS   AGE
myapp-rs-gqf7c   1/1     Running   0          28s
myapp-rs-hlsgp   1/1     Running   0          28s
myapp-rs-ltqf2   1/1     Running   0          28s
```

### 2) NodePort 서비스 생성
> myapp-svc-np.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-np
spec:
  type: NodePort
  ports:
  - port: 80
    targetPort: 8080
    nodePort: 31111
  selector:
    app: myapp-rs
```

.spec.type: 서비스 타입 (기본: ClusterIP)
.spec.ports.nodePort: 30000-32767 포트, 포트를 지정하지 않으면 랜덤한 포드 지정

서비스 타입은 NodePort이며, 노드에 노출할 포트는 31111 포트이다. 포트는 기본적으로 30000-32767 포트 내에서만 사용 가능하다.

노드 포트 서비스를 생성하자.
```
$ kubectl create -f myapp-svc-np.yaml

service/myapp-svc-np created
```

### 3) NodePort 서비스 확인
서비스 목록 및 상태를 확인해보자.
```
$ kubectl get service myapp-svc-np

NAME           TYPE       CLUSTER-IP     EXTERNAL-IP   PORT(S)        AGE
myapp-svc-np   NodePort   10.233.13.27   <none>        80:31111/TCP   15s
```
노드의 31111 포드로 접근하면 서비스의 80 포트로 리다이렉션 된다.

서비스의 엔드포인트를 확인해보자.
```
$ kubectl get endpoints myapp-svc-np

NAME           ENDPOINTS                                                 AGE
myapp-svc-np   10.233.101.33:8080,10.233.103.36:8080,10.233.76.45:8080   29s
```
서비스의 80 포트는 파드의 8080 포트로 리다이렉션 된다.

노드의 IP를 확인해보자.
```
$ kubectl get nodes -o wide

NAME                 STATUS   ... EXTERNAL-IP     ...
kube-controlplane1   Ready    ... 192.168.56.11   ...
kube-node1           Ready    ... 192.168.56.21   ...
kube-node2           Ready    ... 192.168.56.22   ...
kube-node3           Ready    ... 192.168.56.23   ...
```
노드의 IP는 192.168.56.11, 192.168.56.21, 192.168.56.22, 192.168.56.23이다.

노드 IP와 노드 포트로 접근해보자.
```
$ curl http://192.168.56.11:31111

Hello World!
myapp-rs-ltqf2
```
```
$ curl http://192.168.56.21:31111

Hello World!
myapp-rs-ltqf2
```
```
$ curl http://192.168.56.22:31111

Hello World!
myapp-rs-ltqf2
```
```
$ curl http://192.168.56.23:31111

Hello World!
myapp-rs-ltqf2
```
외부에서 192.168.56.X:31111로 접근하면, 서비스 포트인 80번 포트로 리다이렉션 되며, 이는 다시 내부 파드의 8080번 포트로 리다이렉션 된다.

### 4) LoadBalancer 서비스 생성
클라우드 인프라에서 LoadBalancer 서비스를 생성하면 클라우드 인프라의 로드 밸런서를 자동으로 프로비저닝 하게 되며, 이 로드 밸런서를 통해 서비스와 파드에 접근할 수 있다.

> 참고  
> 온프레미스(On-Premise)의 베어메탈 또는 가상머신에 배포하거나, 퍼블릭 클라우드의 관리형 쿠버네티스가 아닌 가상머신에 직접 설치한 경우.  
> [MetalLB](https://metallb.universe.tf/)  
> 오픈소스를 이용해 쿠버네티스 리소스로 로드 밸런서 기능을 제공할 수 있다.  

> myapp-svc-lb.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-lb
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: myapp-rs
```

로드 밸런서 서비스를 생성하자.
```
$ kubectl create -f myapp-svc-lb.yaml

service/myapp-svc-lb created
```

### 5) LoadBalancer 서비스 확인
```
$ kubectl get service myapp-svc-lb

NAME           TYPE           CLUSTER-IP     EXTERNAL-IP      PORT(S)        AGE
myapp-svc-lb   LoadBalancer   10.233.61.67   192.168.56.200   80:30176/TCP   15s
```
로드 밸런서의 외부 IP가 192.168.56.200으로 정상적으로 할당된 것을 확인할 수 있다.

클라우드의 관리형 쿠버네티스 서비스 및 MetalLB를 사용하지 않는 경우, 외부 로드 밸런서가 프로비저닝 되지 않기 때문에 외부 IP(로드 밸런서 IP)가 할당되지 않고 <pending> 상태로 보고한다.

실제 외부 로드 밸런서를 사용할 수 있는 경우, 외부의 로드 밸런서가 생성되고 IP가 할당되기까지 시간이 필요하다.

로드 밸런서 서비스의 IP로 접근해 본다.
```
$ curl http://192.168.56.200

Hello World!
myapp-rs-hlsgp
```
```
$ curl http://192.168.56.200

Hello World!
myapp-rs-ltqf2
```
```
$ curl http://192.168.56.200

Hello World!
myapp-rs-gqf7c
```
로드 밸런서를 통해 부하 분산이 이루어지는 것을 확인할 수 있다.

### 6) ExternalName 서비스 생성
ExternalName 서비스는 NodePort 및 LoadBalancer와 다르게 외부에서 접근하기 위한 서비스 종류가 아닌 내부 파트가 외부의 특정 FQDN에 쉽게 접근하기 위한 서비스다.

즉, 쿠버네티스 클러스터의 coredns 서비스가 특정 FQDN에 대한 CNAME(서비스의 FQDN)을 제공함에 따라 해당 CNAME을 이용하여 쉽게 통신할 수 있다. 심지어 접속하기 위한 외부 FQDN 주소가 바뀌더라도, CNAME은 그대로 유지할 수 있어 애플리케이션을 다시 작성하거나 빌드 하지 않아도 된다.

> myapp-svc-extname.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-extname
spec:
  type: ExternalName
  externalName: www.google.com
```
외부 FQDN은 www.google.com이며 이에 대한 CNAME은 myapp-svc-extname 이름을 제공하는 서비스이다. 파드와 매칭하지 않기 때문에 레이블 셀렉터는 없다.

외부 이름 서비스를 생성해보자.
```
$ kubectl create -f myapp-svc-extname.yaml

service/myapp-svc-extname created
```

### 7) ExternalName 서비스 확인
생성된 서비스를 확인해보자.
```
$ kubectl get service myapp-svc-extname

NAME                TYPE           CLUSTER-IP   EXTERNAL-IP      PORT(S)   AGE
myapp-svc-extname   ExternalName   <none>       www.google.com   <none>    13s
```
서비스 타입은 ExternalName이며, 외부 IP에는 www.google.com의 FQDN이 할당되어 있다.

즉, 파드는 myapp-svc-extername CNAME을 사용하면 www.google.com의 주소를 알 수 있다.
