## 4. 인그레스
앞서 서비스를 외부에 노출시키는 방법인 NodePort와 LoadBalancer 서비스 타입이 대해 살펴보았다. 마지막으로 서비스를 노출시키는 방법이 하나가 더 있다. 이것이 인그레스(Ingress) 컨트롤러다.

쿠버네티스 외부에 노출 시켜야할 서비스가 많을 경우, NodePort는 각 서비스 마다 전용의 노드의 포트를 할당해야하고, LoadBalancer의 경우 외부 로드밸런서가 각 서비스 마다 프로비저닝 되어야 한다. 그러나 인그레스는 HTTP 요청의 주소부분을 구분해 하나의 인그레스 컨트롤러를 이용해 각 서비스에 연결할 수 있다.

```
클라이언트 <-> 인그레스 컨트롤러 <-foo.example.com-> 서비스A <-> 파드
                          <-bar.example.com-> 서비스A <-> 파드
                          <-www.example.com/web1-> 서비스B <-> 파드
                          <-www.example.com/web2-> 서비스B <-> 파드
```


NodePort 및 LoadBalancer 서비스는 OSI 레이어 4(TCP/UDP)에서 작동하지만, 인그레스 컨트롤러는 OSI 레이어 7(HTTP/HTTPS)에서 작동한다. 세션 쿠키 기반의 세션 친화성을 가지고 있다.

> 참고  
> 인그레스 컨트롤러는 Nginx HTTP 서버와 리버스 프록시 기능을 통해 제공되며, ingress 컨트롤러를 노드의 개수만큼 실행시킨다.
```
$ kubectl get all -n ingress-nginx

NAME                                 READY   STATUS    RESTARTS   AGE
pod/ingress-nginx-controller-cx7zt   1/1     Running   0          33h
pod/ingress-nginx-controller-lgmgf   1/1     Running   0          33h
pod/ingress-nginx-controller-ltvdv   1/1     Running   0          33h

NAME                                      DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR                 AGE
daemonset.apps/ingress-nginx-controller   3         3         3       3            3           beta.kubernetes.io/os=linux   33h
```

### 1) 인그레스 생성
인그레스 API 그룹은 networking.k8s.io이며, 버전은 v1beta1이다. 오브젝트 종류는 Ingress로 선언한다.

> 참고  
> 현재는 networking.k8s.io API 그룹과 함께 extensions API 그룹도 똑같이 사용할 수 있으나, extensions API 그룹은 더이상 사용되지(DEPRECATED) 않는다.

> mynapp-ing.yml
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: mynapp-ing
spec:
  rules:
  - host: mynapp.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: mynapp-svc-ext-np
          servicePort: 80
```
외부에서 mynapp.example.com FQDN 주소를 통해 인그레스 컨트롤러에 접속하면, 인그레스 컨트롤러는 백엔드 서비스인 mynapp-svc-ext-np 서비스에 리다이렉션하며, mynapp-svc-ext-np 서비스는 레이블 셀렉터에 매핑된 파드로 연결 해준다.

레플리카셋 컨트롤러와 NodePort 서비스를 생성한다.
```
$ kubectl create -f mynapp-rs.yml -f mynapp-svc-ext-nodeport.yml

replicaset.apps/mynapp-rs created
service/mynapp-svc-ext-np created
```

인그레스 컨트롤러를 생성한다.
```
$ kubectl create -f mynapp-ing.yml

ingress.extensions/mynapp-ing created
```

### 2) 인그레스 컨트롤러 확인
인그레스 컨트롤러를 확인해보자.
```
$ kubectl get ingresses.networking.k8s.io

NAME         HOSTS                ADDRESS   PORTS   AGE
mynapp-ing   mynapp.example.com             80      24s
```
인그레스 컨트롤러가 확인되지만, 아직 인그레스 컨트롤러의 주소(ADDRESS)는 아직 할당되지 않았다. 주소가 할당되는데 시간이 조금 걸린다.

```
$ kubectl get ingresses.networking.k8s.io
NAME         HOSTS                ADDRESS                                     PORTS   AGE
mynapp-ing   mynapp.example.com   192.168.56.21,192.168.56.22,192.168.56.23   80      63s
```
할당된 인그레스 컨트롤러 IP는 쿠버네티스 클러스터의 노드 IP 이다.

인그레스 컨트롤러에 요청할 때에는 mynapp.example.com 주소로 접근해야 해당 mynapp-svc-ext-np 서비스로 리다이렉션 하게 된다. 

그러나 현재 외부 DNS가 없기 때문에 mynapp.example.com FQDN의 A 레코드를 반환하지 못한다. 그러므로 curl 명령의 --resolve 옵션을 사용하여 해당 FQDN의 주소를 IP로 변환하는 옵션을 사용해야 한다.

```
$ curl --resolve mynapp.example.com:80:192.168.56.21 http://mynapp.example.com

Message: Hello World!
Hostname: mynapp-rs-jsldw
Platform: linux
Uptime: 124051
IP: 10.233.101.20
DNS: 169.254.25.10
```

> 참고  
> curl 명령의 --resolve 옵션을 사용할 수도 있고, 호스트 시스템에서 /etc/hosts 파일을 이용할 수도 있다.

### 3) 다중 경로 인그레스
지금 까지는 하나의 인그레스를 하나의 서비스만 노출시켰다. 그러나 인그레스는 하나의 인그레스 컨트롤러로 다중 서비스를 노출 시킬 수 있다.

#### (1) 같은 FQDN의 다중 경로 인그레스 컨트롤러
- http://mynapp.example.com/web1
- http://mynapp.example.com/web2

다중 서비스용 인그레스 구성 예제이다.
> mynapp-ing-multi-paths.yml 파일
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: mynapp-ing-mpath
spec:
  rules:
  - host: mynapp.example.com
    http:
      paths:
        - path: /web1 # http://mynapp.example.com/web1
          backend:
            serviceName: mynapp-svc-ext-np1
            servicePort: 80
        - path: /web2 # http://mynapp.example.com/web2
          backend:
            serviceName: mynapp-svc-ext-np2
            servicePort: 80
```

| FQDN/PATH               | 서비스               |
|-------------------------|--------------------|
| mynapp.example.com/web1 | mynapp-svc-ext-np1 |
| mynapp.example.com/web2 | mynapp-svc-ext-np2 |

인그레스 컨트롤러를 생성해보자.
```
$ kubectl create -f mynapp-ing-multi-paths.yml

ingress.extensions/mynapp-ing-mpath created
```
mynapp-ing-mpath 인그레스 컨트롤러는 실제 작동하지 않는다. 이유는 백엔드 서비스를 만들어 놓지 않았기 때문이다.

인그레스 컨트롤러를 확인해보자.
```
$ kubectl get ingresses.networking.k8s.io

NAME               HOSTS                ADDRESS                                     PORTS   AGE
mynapp-ing         mynapp.example.com   192.168.56.21,192.168.56.22,192.168.56.23   80      11m
mynapp-ing-mpath   mynapp.example.com   192.168.56.21,192.168.56.22,192.168.56.23   80      4m15s
```

인그레스 컨트롤러를 자세하게 확인하면 설정된 룰을 확인할 수 있다.
```
$ kubectl describe ingresses.networking.k8s.io mynapp-ing-mpath

Name:             mynapp-ing-mpath
...
Rules:
  Host                Path  Backends
  ----                ----  --------
  mynapp.example.com
                      /web1   mynapp-svc-ext-np1:80 (<none>)
                      /web2   mynapp-svc-ext-np2:80 (<none>)
...
```

#### (2) 다른 FQDN의 다중 호스트 인그레스 컨트롤러
- http://web1.example.com
- http://web2.example.com

> mynapp-ing-multi-hosts.yml 파일
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: mynapp-ing-mhost
spec:
  rules:
  - host: web1.example.com
    http:
      paths:
      - path: / # http://web1.example.com/
        backend:
          serviceName: mynapp-svc-ext-np1
          servicePort: 80
  - host: web2.example.com
    http:
     paths:
      - path: / # http://web2.example.com/
        backend:
          serviceName: mynapp-svc-ext-np2
          servicePort: 80
```

| FQDN/PATH        | 서비스               |
|------------------|--------------------|
| web1.example.com | mynapp-svc-ext-np1 |
| web2.example.com | mynapp-svc-ext-np2 |

인그레스 컨트롤러를 생성해보자.
```
$ kubectl create -f mynapp-ing-multi-hosts.yml

ingress.extensions/mynapp-ing-mhost created
```

인그레스 컨트롤러를 확인해보자.
```
$ kubectl get ingresses.networking.k8s.io

NAME               HOSTS                               ADDRESS                                     PORTS   AGE
mynapp-ing         mynapp.example.com                  192.168.56.21,192.168.56.22,192.168.56.23   80      15m
mynapp-ing-mhost   web1.example.com,web2.example.com   192.168.56.21,192.168.56.22,192.168.56.23   80      74s
mynapp-ing-mpath   mynapp.example.com                  192.168.56.21,192.168.56.22,192.168.56.23   80      8m55s
```

인그레스 컨트롤러를 자세하게 확인하면 설정된 룰을 확인할 수 있다.
```
$ kubectl describe ingresses.networking.k8s.io mynapp-ing-mhost

Name:             mynapp-ing-mhost
...
Rules:
  Host              Path  Backends
  ----              ----  --------
  web1.example.com
                    /   mynapp-svc-ext-np1:80 (<none>)
  web2.example.com
                    /   mynapp-svc-ext-np2:80 (<none>)
...
```
