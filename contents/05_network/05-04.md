## 4. 인그레스
앞서 서비스를 외부에 노출시키는 방법인 NodePort와 LoadBalancer 서비스 타입이 대해 살펴보았다. 마지막으로 서비스를 노출시키는 방법이 하나가 더 있다. 이것이 인그레스(Ingress) 컨트롤러다.

쿠버네티스 외부에 노출 시켜야할 서비스가 많을 경우, NodePort는 각 서비스 마다 전용의 노드의 포트를 할당해야 하고, LoadBalancer의 경우 외부 로드밸런서가 각 서비스 마다 프로비저닝 되어야 한다. 그러나 인그레스는 HTTP 요청의 주소를 구분해 하나의 인그레스 리소스를 이용해 각 서비스에 라우팅할 수 있다.

```
클라이언트 -> 인그레스 -> 라우팅 규칙 -> 서비스 -> 파드
```
```
-> foo.example.com -> 서비스A -> 파드
-> bar.example.com -> 서비스B -> 파드
-> www.example.com/web1 -> 서비스C -> 파드
-> www.example.com/web2 -> 서비스D -> 파드
```

NodePort 및 LoadBalancer 서비스는 OSI 레이어 4(TCP/UDP)에서 작동하지만, 인그레스 리소스는 OSI 레이어 7(HTTP/HTTPS)에서 작동한다. 세션 쿠키 기반의 세션 어피니티를 가지고 있다.

> 참고  
> 인그레스 컨트롤러는 Nginx HTTP 서버와 리버스 프록시 기능을 통해 제공되며, 인그레스 컨트롤러는 노드의 개수만큼 각 노드에 실행한다.
```
$ kubectl get all -n ingress-nginx

NAME                                 READY   STATUS    RESTARTS   AGE
pod/ingress-nginx-controller-cx7zt   1/1     Running   0          33h
pod/ingress-nginx-controller-lgmgf   1/1     Running   0          33h
pod/ingress-nginx-controller-ltvdv   1/1     Running   0          33h

NAME                                      DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR                 AGE
daemonset.apps/ingress-nginx-controller   3         3         3       3            3           beta.kubernetes.io/os=linux   33h
```

### 1) 인그레스 생성
인그레스 API 그룹은 networking.k8s.io이며, 버전은 v1beta1이다. 오브젝트 종류는 Ingress로 선언한다.

> 참고  
> 현재는 networking.k8s.io API 그룹과 함께 extensions API 그룹도 똑같이 사용할 수 있으나, extensions API 그룹은 더이상 사용되지(DEPRECATED) 않는다.

> myapp-ing.yaml
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: myapp-ing
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: myapp-svc-np
          servicePort: 80
```

.spec.rules.host: URL 호스트
.spec.rules.host.http.paths.path: URL 경로
.spec.rules.host.http.paths.path.backend: 라우팅 할 서비스 밴엔드
.spec.rules.host.http.paths.path.backend.serviceName: 서비스 리소스 이름
.spec.rules.host.http.paths.path.backend.servicePort: 서비스 포트

외부에서 myapp.example.com FQDN 주소를 통해 인그레스 리소스에 접속하면, 인그레스 리소스는 백엔드 서비스인 mynapp-svc-np 서비스에 라우팅하며, mynapp-svc-np 서비스는 레이블 셀렉터에 매핑된 파드로 라우팅 해준다.

레플리카셋 컨트롤러와 NodePort 서비스를 생성한다.
```
$ kubectl create -f myapp-rs.yaml -f myapp-svc-np.yaml

replicaset.apps/myapp-rs created
service/myapp-svc-np created
```

인그레스 리소스를 생성한다.
```
$ kubectl create -f myapp-ing.yaml

ingress.networking.k8s.io/myapp-ing created
```

### 2) 인그레스 리소스 확인
인그레스 리소스를 확인해보자.
```
$ kubectl get ingresses.networking.k8s.io

NAME         HOSTS                ADDRESS   PORTS   AGE
mynapp-ing   mynapp.example.com             80      24s
```
인그레스 리소스가 확인되지만, 아직 인그레스 리소스의 주소(ADDRESS)는 아직 할당되지 않았다. 주소가 할당되는데 시간이 조금 걸린다.

인그레스 리소스를 확인 해보자.
```
$ kubectl get ingresses.networking.k8s.io

NAME        CLASS    HOSTS               ADDRESS                                     PORTS   AGE
myapp-ing   <none>   myapp.example.com   192.168.56.21,192.168.56.22,192.168.56.23   80      2m16s
```
할당된 인그레스 리소스 IP는 쿠버네티스 클러스터의 노드 IP다.

인그레스 리소스의 상세 정보를 확인 해보자.
```
$ kubectl describe ingresses.networking.k8s.io myapp-ing

Name:             myapp-ing
Namespace:        default
Address:          192.168.56.21,192.168.56.22,192.168.56.23
Default backend:  default-http-backend:80 (<error: endpoints "default-http-backend" not found>)
Rules:
  Host               Path  Backends
  ----               ----  --------
  myapp.example.com
                     /   myapp-svc-np:80 (10.233.101.34:8080,10.233.103.37:8080,10.233.76.48:8080)
...
```

인그레스 리소스에 myapp.example.com 주소로 접근하면 myapp-svc-np 서비스로 라우팅 하게된다.

그러나 현재 외부 DNS가 없기 때문에 mynapp.example.com FQDN의 A 레코드를 반환하지 못한다. 그러므로 curl 명령의 --resolve 옵션을 사용하여 해당 FQDN의 주소를 IP로 변환하는 옵션을 사용해야 한다.

```
$ curl --resolve myapp.example.com:80:192.168.56.21 http://myapp.example.com/\?detail\=header
Hello World!
myapp-rs-v69f9

[Request Headers]
	GET /?detail=header HTTP/1.1
	Accept: [*/*]
	X-Request-Id: [85938a0f7b15cff6e126ad9b38bf5208]
	X-Forwarded-Proto: [http]
	X-Forwarded-Host: [myapp.example.com]
	X-Forwarded-Port: [80]
	X-Scheme: [http]
	X-Real-Ip: [192.168.56.11]
	X-Forwarded-For: [192.168.56.11]
	User-Agent: [curl/7.58.0]
```

> 참고  
> curl 명령의 --resolve 옵션을 대신 /etc/hosts 파일을 이용할 수 있다.

> 참고
> 와일드카드 DNS 서비스를 이용해 테스트할 수 있다.
> nip.io
> xip.io
> sslip.io

### 3) 다중 경로 인그레스
지금 까지는 하나의 인그레스를 하나의 서비스만 노출시켰다. 그러나 인그레스는 하나의 인그레스 리소스로 다중 서비스를 노출 시킬 수 있다.

#### (1) 같은 FQDN의 다중 경로 인그레스 
- http://myapp.example.com/web1
- http://myapp.example.com/web2

다중 서비스용 인그레스 구성 예제이다.
> myapp-ing-multi-paths.yaml
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: myapp-ing-mpath
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
        - path: /web1
          backend:
            serviceName: myapp-svc-ext-np1
            servicePort: 80
        - path: /web2
          backend:
            serviceName: myapp-svc-ext-np2
            servicePort: 80
```

| FQDN/PATH              | 서비스              |
|------------------------|-------------------|
| myapp.example.com/web1 | myapp-svc-ext-np1 |
| myapp.example.com/web2 | myapp-svc-ext-np2 |

인그레스 리소스를 생성해보자.
```
$ kubectl create -f myapp-ing-multi-paths.yaml

ingress.networking.k8s.io/myapp-ing-mpath created
```
mynapp-ing-mpath 인그레스 리소스는 실제 작동하지 않는다. 이유는 백엔드 서비스를 만들어 놓지 않았기 때문이다.

인그레스의 룰을 확인해보자.
```
$ kubectl describe ingresses.networking.k8s.io myapp-ing-mpath
Name:             myapp-ing-mpath
...
Rules:
  Host               Path  Backends
  ----               ----  --------
  myapp.example.com
                     /web1   myapp-svc-ext-np1:80 (<error: endpoints "myapp-svc-ext-np1" not found>)
                     /web2   myapp-svc-ext-np2:80 (<error: endpoints "myapp-svc-ext-np2" not found>)
...
```

#### (2) 다른 FQDN의 다중 호스트 인그레스
- http://web1.example.com
- http://web2.example.com

> myapp-ing-multi-hosts.yaml
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: myapp-ing-mhost
spec:
  rules:
  - host: web1.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: myapp-svc-ext-np1
          servicePort: 80
  - host: web2.example.com
    http:
     paths:
      - path: /
        backend:
          serviceName: myapp-svc-ext-np2
          servicePort: 80
```

| FQDN/PATH        | 서비스              |
|------------------|-------------------|
| web1.example.com | myapp-svc-ext-np1 |
| web2.example.com | myapp-svc-ext-np2 |

인그레스 리소스를 생성해보자.
```
$ kubectl create -f myapp-ing-multi-hosts.yaml

ingress.networking.k8s.io/myapp-ing-mhost created
```

인그레스의 룰을 확인해보자.
```
$ kubectl describe ingresses.networking.k8s.io myapp-ing-mhost
Name:             myapp-ing-mhost
...
Rules:
  Host              Path  Backends
  ----              ----  --------
  web1.example.com
                    /   myapp-svc-ext-np1:80 (<error: endpoints "myapp-svc-ext-np1" not found>)
  web2.example.com
                    /   myapp-svc-ext-np2:80 (<error: endpoints "myapp-svc-ext-np2" not found>)
...
```
