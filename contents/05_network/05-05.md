## 5. 레디니스 프로브

### 1) 레디니스 프로브 소개
앞서 불안전한 파드를 재시작 하는 라이브니스 프로브를 학습하였다. 이와 비슷 하지만 서비스 리소스와 관련된 레디니스 프로브에 대해 알아보자.

레디니스 프로브(Rediness Probe)는 주기적으로 파드를 모니터링 하여 파드의 애플리케이션의 동작이 준비가 되었는지 확인한다. 동작 준비가 되었다면 연결된 서비스의 엔드포인트에 파드 주소를 등록하고, 동작 준비가 되지 않았다면 연결된 서비스의 엔드포 인트에 주소를 등록하지 않는다.

즉, 엔드포인트에 주소가 등록되지 않았다는 것은, 파드의 애플리케이션이 준비가 안 되었다는 것이고, 서비스를 통해 파드에 접근하지 못한다. 준비가 되지 않는 파드는 서비스에 의해 노출하지 않는다는 의미이다.

지금까지 파드나 컨트롤러에 의해 생성된 파드는 레디니스 프로브가 설정되지 않았기 때문에 컨테이너가 실행되면, 바로 준비되었는지 확인하지 않고, 바로 엔드포인트에 등록하게 된다. 이는 실제로 애플리케이션이 동작하지 않음에도 불구하고 엔드포인트에 등록되기 때문에, 특정 파드에 접근하는 클라이언트는 서비스를 사용하지 못하는 경우가 발생할 수 있다.


레디니스 프로브의 종류는 라이브니스 프로브와 같이 세 가지가 있다.
- HTTP GET 프로브
  * 특정 경로에 HTTP GET 요청
  * HTTP 응답코드가 2XX 또는 3XX 인지 확인 함
- TCP 소켓 프로브
  * 특정 TCP 포트 연결을 시도 함
- Exec 프로브
  * 컨테이너 내부의 바이너리(명령)을 실행하고 종료 코드 확인

### 2) 레디니스 프로브 생성
다음은 레디니스 프로브를 적용한 레플리카셋 오브젝트이다.

> mynapp-rs-readiness.yml 파일
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: mynapp-rs-readiness
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mynapp-rs-readiness
  template:
    metadata:
      labels:
        app: mynapp-rs-readiness
    spec:
      containers:
      - name: mynapp
        image: c1t1d0s7/myweb
        readinessProbe:
          exec:
            command:
              - ls
              - /var/ready
        ports:
        - containerPort: 8080
```
Exec 프로브를 이용 하여 엔드포인트 등록 여부를 확인해 보자. 실제 이미지에는 /var/ready 파일이 없으며 ls 명령으로 확인하면 초기에는 파일이 없어 엔드포인트에 등록이 되지 않을 것으로 예상된다. (일부러 오류 상황을 만들었다.)

다음은 mynapp-rs-readiness 레플리카셋 컨트롤러에 연결할 서비스다.
> mynapp-svc-readiness.yml 파일
```yaml
apiVersion: v1
kind: Service
metadata:
  name: mynapp-svc-readiness
spec:
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: mynapp-rs-readiness
```

레디니스 프로브를 위한 레플리카셋 컨트롤러와 서비스를 생성한다.
```
$ kubectl create -f mynapp-rs-readiness.yml -f mynapp-svc-readiness.yml

replicaset.apps/mynapp-rs-readiness created
service/mynapp-svc-readiness created
```

### 3) 레디니스 프로브 확인
컨트롤러의 상태를 확인 해보자.
```
$ kubectl get replicasets.apps

NAME                  DESIRED   CURRENT   READY   AGE
mynapp-rs-readiness   3         3         0       49s
```
현재 아무 파드로 준비(READY)되지 않았다.

파드의 상태를 확인 해보자.
```
$ kubectl get pods

NAME                        READY   STATUS    RESTARTS   AGE
mynapp-rs-readiness-8qvw6   0/1     Running   0          76s
mynapp-rs-readiness-fr6hv   0/1     Running   0          76s
mynapp-rs-readiness-jsmzn   0/1     Running   0          76s
```
역시 모든 파드가 준비되지 않았다.


서비스의 상태를 확인 해보자.
```
$ kubectl get services

NAME                   TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
kubernetes             ClusterIP   10.233.0.1      <none>        443/TCP   4h27m
mynapp-svc-readiness   ClusterIP   10.233.50.140   <none>        80/TCP    101s
```
mynapp-svc-readiness 서비스는 이상없이 생성되어 있다.

mynapp-svc-readiness 서비스의 엔드포인트의 상태를 확인해보자.
```
$ kubectl get endpoints mynapp-svc-readiness

NAME                   ENDPOINTS   AGE
mynapp-svc-readiness               2m44s
```
엔드포인트에 아무런 파드의 주소가 없음을 확인할 수 있다.

kubectl exec 명령을 이용하여 하나의 파드에 /var/ready 파일을 만들어 보자.
```
$ kubectl exec mynapp-rs-readiness-8qvw6 -- touch /var/ready
```

그리고 파드의 준비 상태와 엔드포인트 목록을 확인해보자.
```
$ kubectl get pods,endpoints

NAME                            READY   STATUS    RESTARTS   AGE
pod/mynapp-rs-readiness-8qvw6   1/1     Running   0          3m35s
pod/mynapp-rs-readiness-fr6hv   0/1     Running   0          3m35s
pod/mynapp-rs-readiness-jsmzn   0/1     Running   0          3m35s

NAME                             ENDPOINTS            AGE
endpoints/kubernetes             192.168.56.11:6443   4h29m
endpoints/mynapp-svc-readiness   10.233.101.21:8080   3m35s
```
하나의 파드가 준비 상태인 것을 확인할 수 있고, 해당 파드의 주소가 엔드 포인트에 등록된 것을 확인할 수 있다.

나머지 두 파드도 파일을 생성해서 준비되도록 하자.
```
$ kubectl exec mynapp-rs-readiness-fr6hv -- touch /var/ready
$ kubectl exec mynapp-rs-readiness-jsmzn -- touch /var/ready
```

다시 한번 파드 준비 상태와 엔드포인트 목록을 확인해 보자.
```
$ kubectl get pods,endpoints

NAME                            READY   STATUS    RESTARTS   AGE
pod/mynapp-rs-readiness-8qvw6   1/1     Running   0          4m57s
pod/mynapp-rs-readiness-fr6hv   1/1     Running   0          4m57s
pod/mynapp-rs-readiness-jsmzn   1/1     Running   0          4m57s

NAME                             ENDPOINTS                                                 AGE
endpoints/kubernetes             192.168.56.11:6443                                        4h30m
endpoints/mynapp-svc-readiness   10.233.101.21:8080,10.233.103.22:8080,10.233.76.22:8080   4m57s
```
모든 파드가 준비되었고, 모든 파드의 주소가 엔드포인트에 등록 되었음을 확인할 수 있다.

### 4) 컨트롤러 및 서비스 삭제
```
$ kubectl delete -f mynapp-rs-readiness.yml -f mynapp-svc-readiness.yml

replicaset.apps "mynapp-rs-readiness" deleted
service "mynapp-svc-readiness" deleted
```
