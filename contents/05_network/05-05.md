## 5. 레디니스 프로브

### 1) 레디니스 프로브 소개
레디니스 프로브(Rediness Probe)는 주기적으로 파드를 모니터링하여 애플리케이션이 요청을 처리할 준비가 되었는지 확인한다. 준비가 되었다면 연결된 서비스의 엔드포인트에 파드 주소를 등록하고, 준비가 되지 않았다면 연결된 서비스의 엔드포인트에 주소를 등록하지 않는다.

즉, 엔드포인트에 주소가 등록되지 않았다는 것은, 파드의 애플리케이션이 준비가 되지 않았다는 것이고, 서비스를 통해 파드에 접근하지 못한다. 준비가 되지 않은 파드는 서비스에 의해 노출하지 않는다는 의미이다.

지금까지 파드나 컨트롤러에 의해 생성된 파드는 레디니스 프로브가 설정되지 않았기 때문에 컨테이너가 실행되면, 바로 준비되었는지 확인하지 않고, 바로 엔드포인트에 등록하게 된다. 이는 실제로 애플리케이션이 동작하지 않음에도 불구하고 엔드포인트에 등록되기 때문에, 특정 파드에 접근하는 클라이언트는 서비스를 사용하지 못하는 경우가 발생할 수 있다.

### 2) 레디니스 프로브 생성
다음은 레디니스 프로브를 적용한 레플리카셋 오브젝트이다.

> myapp-rs-readiness.yaml
```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-rs-readiness
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp-rs-readiness
  template:
    metadata:
      labels:
        app: myapp-rs-readiness
    spec:
      containers:
      - name: myapp
        image: ghcr.io/c1t1d0s7/go-myweb:alpine
        readinessProbe:
          exec:
            command:
              - ls
              - /var/ready
        ports:
        - containerPort: 8080
```
Exec 프로브를 이용하여 엔드포인트 등록 여부를 확인해보자. 실제 이미지에는 /var/ready 파일이 없으며 ls 명령으로 확인하면 초기에는 파일이 없어 엔드포인트에 등록이 되지 않을 것으로 예상된다. 이미지는 쉘을 사용하기 위해 Alpine 리눅스가 포함된 이미지를 사용한다.

다음은 myapp-rs-readiness 레플리카셋 컨트롤러에 연결할 서비스다.
> myapp-svc-readiness.yaml
```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc-readiness
spec:
  ports:
  - port: 80
    targetPort: 8080
  selector:
    app: myapp-rs-readiness
```

레디니스 프로브를 위한 레플리카셋 컨트롤러와 서비스를 생성한다.
```
$ kubectl create -f myapp-rs-readiness.yaml -f myapp-svc-readiness.yaml

replicaset.apps/myapp-rs-readiness created
service/myapp-svc-readiness created
```

### 3) 레디니스 프로브 확인
컨트롤러의 상태를 확인해보자.
```
$ kubectl get replicasets.apps

NAME                 DESIRED   CURRENT   READY   AGE
myapp-rs-readiness   3         3         0       14s
```
현재 모든 파드가 준비(READY) 되지 않았다.

파드의 상태를 확인해보자.
```
$ kubectl get pods

NAME                       READY   STATUS    RESTARTS   AGE
myapp-rs-readiness-52c4n   0/1     Running   0          9s
myapp-rs-readiness-5qrfc   0/1     Running   0          9s
myapp-rs-readiness-js24r   0/1     Running   0          9s
```
역시 모든 파드가 준비되지 않았다.

서비스의 상태를 확인해보자.
```
$ kubectl get service myapp-svc-readiness

NAME                  TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
myapp-svc-readiness   ClusterIP   10.233.40.23   <none>        80/TCP    34s
```
myapp-svc-readiness 서비스는 이상없이 생성되어 있다.

myapp-svc-readiness 서비스의 엔드포인트의 상태를 확인해보자.
```
$  kubectl get endpoints myapp-svc-readiness

NAME                  ENDPOINTS   AGE
myapp-svc-readiness               46s
```
엔드포인트에 파드의 주소가 없음을 확인할 수 있다.

kubectl exec 명령을 이용하여 하나의 파드에 /var/ready 파일을 만들어 보자.
```
$ kubectl exec myapp-rs-readiness-52c4n -- touch /var/ready
```

그리고 파드의 준비 상태와 엔드포인트 목록을 확인해보자.
```
$ kubectl get pods,endpoints
NAME                           READY   STATUS    RESTARTS   AGE
pod/myapp-rs-readiness-52c4n   1/1     Running   0          84s
pod/myapp-rs-readiness-5qrfc   0/1     Running   0          84s
pod/myapp-rs-readiness-js24r   0/1     Running   0          84s

NAME                            ENDPOINTS            AGE
...
endpoints/myapp-svc-readiness   10.233.103.41:8080   84s
```
하나의 파드가 준비 상태인 것을 확인할 수 있고, 해당 파드의 주소가 엔드 포인트에 등록된 것을 확인할 수 있다.

나머지 두 파드도 파일을 생성해서 준비되도록 하자.
```
$ kubectl exec myapp-rs-readiness-5qrfc -- touch /var/ready
$ kubectl exec myapp-rs-readiness-js24r -- touch /var/ready
```

다시 한번 파드 준비 상태와 엔드포인트 목록을 확인해보자.
```
$ kubectl get pods,endpoints

NAME                           READY   STATUS    RESTARTS   AGE
pod/myapp-rs-readiness-52c4n   1/1     Running   0          2m40s
pod/myapp-rs-readiness-5qrfc   1/1     Running   0          2m40s
pod/myapp-rs-readiness-js24r   1/1     Running   0          2m40s

NAME                            ENDPOINTS                                                 AGE
...
endpoints/myapp-svc-readiness   10.233.101.38:8080,10.233.103.41:8080,10.233.76.52:8080   2m40s
```
모든 파드가 준비되었고, 모든 파드의 주소가 엔드포인트에 등록되었음을 확인할 수 있다.

### 4) 컨트롤러 및 서비스 삭제
```
$ kubectl delete -f myapp-rs-readiness.yaml -f myapp-svc-readiness.yaml

replicaset.apps "myapp-rs-readiness" deleted
service "myapp-svc-readiness" deleted
```
