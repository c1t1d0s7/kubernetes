## 2. 서비스 탐색

| 클러스터 외부 | 클러스터 내부                                      |
|------------|------------------------------------------------|
| 클라이언트 <--|--> 앱 서비스 <--> 앱 파드 <--> DB 서비스 <--> DB 파드 |

일반적인 웹의 구조이다. 클라이언트는 앱 서비스의 IP 및 포트(사실 DNS를 이용한 FQDN)를 통해 접근할 수 있다. 앱 파드는 DB 서비스의 IP 및 포트를 통해 접근할 수 있다.

쿠버네티스 클러스터의 서비스는 파드에서 직접 탐색하고 접근할 수 있다. 첫 번째는 환경 변수를 이용한 탐색 방법이 있고, 두 번째는 DNS를 이용한 탐색 방법이 있다.

### 1) 환경 변수를 이용한 서비스 탐색
쿠버네티스 클러스터는 파드가 시작될 때 현재 존재하는 서비스를 파드 내의 쉘 환경 변수로 설정한다. 쉘이 없는 이미지는 환경 변수를 설정하지 못한다.

하나의 서비스에 설정되는 여러 형태의 쉘 환경 변수:
- {SVC_NAME}_SERVICE_HOST={SVC_IP}
- {SVC_NAME}_SERVICE_PORT={SVC_PORT}
- {SVC_NAME}_PORT={PROTOCOL}://{SVC_IP}:{SVC_PORT}
- {SVC_NAME}_PORT_{PORT_NUMBER}_{PROTOCOL}={PROTOCOL}://{SVC_IP}:{SVC_PORT}
- {SVC_NAME}_PORT_{PORT_NUMBER}_{PROTOCOL}_ADDR={SVC_IP}
- {SVC_NAME}_PORT_{PORT_NUMBER}_{PROTOCOL}_PORT={SVC_PORT}
- {SVC_NAME}_PORT_{PORT_NUMBER}_{PROTOCOL}_PROTO={PROTOCOL}

임시 확인용 파드를 생성해서 확인해보자. 쉘 환경 변수는 env 명령으로 확인할 수 있다.
```

KUBERNETES_SERVICE_PORT=443
MYAPP_SVC_SES_AFF_SERVICE_HOST=10.233.5.3
MYAPP_SVC_SES_AFF_PORT_80_TCP=tcp://10.233.5.3:80
MYAPP_SVC_NAMEDPORT_SERVICE_HOST=10.233.24.88
HOSTNAME=nettool
MYAPP_SVC_PORT_80_TCP_PROTO=tcp
MYAPP_SVC_SES_AFF_PORT=tcp://10.233.5.3:80
PWD=/
MYAPP_SVC_NAMEDPORT_PORT=tcp://10.233.24.88:80
HOME=/root
MYAPP_SVC_SERVICE_HOST=10.233.24.226
KUBERNETES_PORT_443_TCP=tcp://10.233.0.1:443
MYAPP_SVC_SES_AFF_PORT_80_TCP_ADDR=10.233.5.3
MYAPP_SVC_NAMEDPORT_PORT_80_TCP_PORT=80
MYAPP_SVC_SERVICE_PORT=80
MYAPP_SVC_NAMEDPORT_SERVICE_PORT=80
TERM=xterm
MYAPP_SVC_PORT_80_TCP_PORT=80
MYAPP_SVC_PORT_80_TCP_ADDR=10.233.24.226
SHLVL=1
MYAPP_SVC_SES_AFF_PORT_80_TCP_PORT=80
MYAPP_SVC_SES_AFF_SERVICE_PORT=80
KUBERNETES_PORT_443_TCP_PROTO=tcp
KUBERNETES_PORT_443_TCP_ADDR=10.233.0.1
MYAPP_SVC_PORT_80_TCP=tcp://10.233.24.226:80

KUBERNETES_PORT_443_TCP_PORT=443
MYAPP_SVC_NAMEDPORT_SERVICE_PORT_MYAPP_HTTP=80
MYAPP_SVC_NAMEDPORT_PORT_80_TCP_ADDR=10.233.24.88
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

MYAPP_SVC_PORT=tcp://10.233.24.226:80
MYAPP_SVC_NAMEDPORT_PORT_80_TCP=tcp://10.233.24.88:80

```
출력되는 대부분의 환경 변수는 앞서 생성한 kubernetes, myapp-svc, myapp-svc-namedport, myapp-svc-ses-aff 서비스의 환경 변수다.

대표적으로 myapp-svc 서비스에 대한 다음 환경 변수를 참조할 수 있다.
- MYAPP_SVC_SERVICE_HOST=10.233.24.226
- MYAPP_SVC_SERVICE_PORT=80

애플리케이션은 서비스 이름만 알고 있다면 적절한 환경 변수를 참조해 접근하고자 하는 서비스의 IP와 포트를 확인할 수 있다.

### 2) DNS를 이용한 서비스 탐색
쿠버네티스 클러스터는 클러스터 시스템 내부에서 사용할 수 있는 DNS(CoreDNS) 서버가 동작하고 있다.

우리는 이미 kube-system 네임스페이스에 DNS 관련 파드 및 컨트롤러 서비스가 존재하는 것을 알고 있다.

#### (1) DNS 관련 리소스 확인
kube-system 네임스페이스에 DNS 관련 파드를 확인해보자. DNS 관련 파드는 k8s-app=kube-dns 레이블을 확인하면 된다.
```
$ kubectl get all -n kube-system -l k8s-app=kube-dns

NAME                           READY   STATUS    RESTARTS   AGE
pod/coredns-6dc4d777d6-5f8sj   1/1     Running   0          29h
pod/coredns-6dc4d777d6-bfxp7   1/1     Running   0          29h

NAME              TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)                  AGE
service/coredns   ClusterIP   10.233.0.3   <none>        53/UDP,53/TCP,9153/TCP   29h

NAME                          DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
daemonset.apps/nodelocaldns   4         4         4       4            4           <none>          29h

NAME                      READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/coredns   2/2     2            2           29h

NAME                                 DESIRED   CURRENT   READY   AGE
replicaset.apps/coredns-6dc4d777d6   2         2         2       29h
```
DNS 관련 서비스인 service/coredns 서비스가 있으며 10.233.0.3 IP로 접근할 수 있다. 그리고 표준 DNS 포트인 53번으로 서비스하고 있다.

#### (2) 파드 내부 DNS 설정 확인
임시 파드를 생성해 파드 내부의 DNS 설정을 확인해보자.
```
$ kubectl run nettool -it --image=ghcr.io/c1t1d0s7/network-multitool --rm bash

bash-5.1# cat /etc/resolv.conf
nameserver 169.254.25.10
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5

bash-5.1# exit
exit

Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
예상과는 다르게 DNS 서버는 169.254.25.10으로 설정되어 있다. 이는 coredns 서비스의 IP인 10.233.0.3과 어떤 연관이 있을까?

#### (3) 노드 로컬 DNS 캐시
기존에는 파드 내의 DNS 서버 설정도 coredns 서비스의 IP가 부여되어 있었다. 즉, 파드가 직접적으로 coredns 서비스에 DNS 쿼리/응답을 받게 되었는데, 이는 대규모 환경에서 병목현상을 만들어 성능이 저하된다는 단점이 있었다.

그래서 쿠버네티스 1.15 버전부터 추가된 노드 로컬 DNS 캐시(NodeLocal DNSCache) 기능이 도입되었다. 노드 로컬 DNS 캐시 기능은 쿠버네티스의 각 노드에 DNS 캐시 기능을 가지고 있는 파드를 데몬셋으로 배치하고, 파드의 DNS 쿼리 및 응답을 coredns가 대신하게 되는 구조이다.

NodeLocal

Pod <--> NodeLocal DNSCache(169.254.25.10) <--> iptables <--> coredns(10.233.0.3)

> 참고: 노드 로컬 DNS 캐시
> https://kubernetes.io/docs/tasks/administer-cluster/nodelocaldns/

노드 로컬 DNS 캐시 기능에 대해 간단하게 살펴보자.

kube-system 네임스페이스에 nodelocaldns 데몬셋 컨트롤러가 존재한다.
```
$ kubectl get daemonsets.apps -l k8s-app=kube-dns -n kube-system

NAME           DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
nodelocaldns   4         4         4       4            4           <none>          30h
```
현재 우리 구성은 마스터 1대, 노드 3대로 총 4대의 구성이기 때문에 4개의 파드가 배포된다.

nodelocaldns 관련 파드 목록을 확인해보자.
```
$ kubectl get pods -l k8s-app=nodelocaldns -n kube-system

NAME                 READY   STATUS    RESTARTS   AGE
nodelocaldns-gnrcf   1/1     Running   0          30h
nodelocaldns-kwnbf   1/1     Running   0          30h
nodelocaldns-pptjw   1/1     Running   0          30h
nodelocaldns-wppmz   1/1     Running   0          30h
```

nodelocaldns 파드의 아규먼트 설정이다. 이는 각 노드에서 169.254.25.10 IP인 링크로컬 주소 대역을 사용해 DNS 캐시 서비스를 제공한다는 의미이다.
```
$ kubectl get pods -l k8s-app=nodelocaldns -n kube-system -o jsonpath='{.items[*].spec.containers[*].args[*]}'

-localip 169.254.25.10 -conf /etc/coredns/Corefile -upstreamsvc coredns 
-localip 169.254.25.10 -conf /etc/coredns/Corefile -upstreamsvc coredns 
-localip 169.254.25.10 -conf /etc/coredns/Corefile -upstreamsvc coredns 
-localip 169.254.25.10 -conf /etc/coredns/Corefile -upstreamsvc coredns
```

#### (4) FQDN을 이용한 서비스 탐색
다시 본론으로 돌아가서, nettool 파드를 이용해 서비스의 FQDN을 이용해 접근해보자.
```
$ kubectl run nettool -it --image=ghcr.io/c1t1d0s7/network-multitool --rm bash

bash-5.1# curl http://myapp-svc
Hello World!
myapp-rs-9xq7l

bash-5.1# curl http://myapp-svc.default
Hello World!
myapp-rs-c6vgs

bash-5.1# curl http://myapp-svc.default.svc
Hello World!
myapp-rs-xs7sd

bash-5.1# curl http://myapp-svc.default.svc.cluster.local
Hello World!
myapp-rs-9xq7l

bash-5.1# exit
exit

Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
myapp-svc 서비스에 접근하기 위한 FQDN 주소는 mynapp-svc.default.svc.cluster.local 이다.
- mynapp-svc: 서비스의 이름
- default: 서비스 리소스의 네임스페이스 이름
- svc: 서비스 그 자체를 의미함
- cluster.local: 쿠버네티스 클러스터 도메인

FQDN의 주소 형식은 다음과 같다.
<서비스 이름>.<네임스페이스>.<리소스 종류>.<클러스터 도메인>
mynapp-svc.default.svc.cluster.local

#### (5) 컨트롤러 및 서비스 리소스 삭제
레플리카셋 컨트롤러와 서비스의 모든 리소스를 삭제한다.
```
$ kubectl delete replicasets.apps,services --all
replicaset.apps "myapp-rs" deleted
replicaset.apps "myapp-rs-namedport" deleted
service "kubernetes" deleted
service "myapp-svc" deleted
service "myapp-svc-namedport" deleted
service "myapp-svc-ses-aff" deleted
```

> 참고  
> service/kubernetes 서비스는 삭제되더라도 다시 생성된다.
