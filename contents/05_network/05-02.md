## 2. 서비스 탐색

| 클러스터 외부 | 클러스터 내부                                     |
|------------|-----------------------------------------------|
| 클라이언트 <--|--> 웹서비스 <--> 웹 파드 <--> DB 서비스 <--> DB 파드 |

일반적인 웹의 구조이다. 클라이언트는 웹 서비스의 고정 IP 및 고정 포트(사실 DNS를 이용한 FQDN)를 통해 접근할 수 있다. 웹 파드는 DB 서비스의 고정 IP 및 고정 포트를 통해 접근할 수 있다.

그러면 웹 파드의 애플리케이션은 DB 파드에 접근하기 위해 DB 서비스의 IP 및 포트를 알아야 하는가(?)에 대한 문제이다.

쿠버네티스 클러스터의 서비스는 파드에서 직접 탐색하고 접근할 수 있다. 첫 번째는 환경변수를 이용한 탐색 방법이 있고, 두 번째는 DNS를 이용한 탐색 방법이 있다.

### 1) 환경변수를 이용한 서비스 탐색
쿠버네티스 클러스터는 파드가 시작될 때 존재하는 서비스를 파드 내의 쉘 환경 변수로 설정한다.

현재 실행중인 파드에 kubectl exec 명령을 이용해 환경 변수를 확인해보자.
```
$ kubectl exec mynapp-rs-m65m4 env

PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
HOSTNAME=mynapp-rs-m65m4
KUBERNETES_SERVICE_HOST=10.233.0.1
KUBERNETES_SERVICE_PORT=443
KUBERNETES_PORT_443_TCP_PORT=443
MYNAPP_SVC_PORT_80_TCP=tcp://10.233.29.60:80
MYNAPP_SVC_PORT_80_TCP_PORT=80
MYNAPP_SVC_PORT_80_TCP_ADDR=10.233.29.60
KUBERNETES_PORT_443_TCP=tcp://10.233.0.1:443
KUBERNETES_PORT_443_TCP_PROTO=tcp
MYNAPP_SVC_SERVICE_PORT=80
KUBERNETES_SERVICE_PORT_HTTPS=443
KUBERNETES_PORT_443_TCP_ADDR=10.233.0.1
MYNAPP_SVC_PORT=tcp://10.233.29.60:80
MYNAPP_SVC_PORT_80_TCP_PROTO=tcp
KUBERNETES_PORT=tcp://10.233.0.1:443
MYNAPP_SVC_SERVICE_HOST=10.233.29.60
NODE_VERSION=13.7.0
YARN_VERSION=1.21.1
HOME=/root
```
환경 변수 중 많은 부분이 kubernetes 서비스 및 mynaap 서비스의 IP, 포트에 관한 변수와 값으로 구성되어 있음을 알 수 있다.

대표적으로 다음과 같은 환경 변수를 참조할 수 있다.
- MYNAPP_SVC_SERVICE_HOST=10.233.29.60
- MYNAPP_SVC_SERVICE_PORT=80
MYNAPP_SVC_, KUBERNETES_SVC_ 이름은 실제 서비스의 이름과 매핑된다.

애플리케이션은 서비스 이름만 알고 있다면 적절한 환경 변수를 참조해 접근하고자 하는 서비스의 IP와 포트를 확인할 수 있다.

### 2) DNS를 이용한 서비스 탐색
쿠버네티스 클러스터는 클러스터 시스템 내부에서 사용할 수 있는 DNS(CoreDNS) 서버가 동작하고 있다.

우리는 이미 kube-system 네임스페이스에 DNS 관련 파드 및 컨트롤러 서비스가 존재하는 것을 알고 있다.

#### (1) DNS 관련 리소스 확인
kube-system 네임스페이스에 DNS 관련 파드를 확인해보자. DNS 관련 파드는 k8s-app=kube-dns 레이블을 확인하면된다.
```
$ kubectl get all -n kube-system -l k8s-app=kube-dns

NAME                           READY   STATUS    RESTARTS   AGE
pod/coredns-6dc4d777d6-5f8sj   1/1     Running   0          29h
pod/coredns-6dc4d777d6-bfxp7   1/1     Running   0          29h

NAME              TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)                  AGE
service/coredns   ClusterIP   10.233.0.3   <none>        53/UDP,53/TCP,9153/TCP   29h

NAME                          DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
daemonset.apps/nodelocaldns   4         4         4       4            4           <none>          29h

NAME                      READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/coredns   2/2     2            2           29h

NAME                                 DESIRED   CURRENT   READY   AGE
replicaset.apps/coredns-6dc4d777d6   2         2         2       29h
```
DNS 관련 서비스인 service/coredns 서비스가 있으며 10.233.0.3 IP로 접근할 수 있다. 그리고 표준 DNS 포트인 53번으로 서비스 하고 있다.

#### (2) 파드 내부 DNS 설정 확인
파드 내부의 DNS 설정도 확인해보자.
```
$ kubectl exec mynapp-rs-m65m4 -- cat /etc/resolv.conf

nameserver 169.254.25.10
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5
```
예상과는 다르게 DNS 서버는 169.254.25.10 으로 설정되어 있다. 이는 coredns 서비스의 IP인 10.233.0.3와 어떤 연관이 있을까?

#### (3) NodeLocal DNSCache
기존에는 파드 내의 DNS 서버 설정도 coredns 서비스의 IP가 부여되어 있었다. 즉, 파드가 직접적으로 coredns 서비스에 DNS 쿼리/응답을 받게 되었는데, 이는 대규모 환경에서 병목현상을 만들어 성능이 저하 된다는 단점이 있었다.

그래서 쿠버네티스 1.15 버전 부터 추가된 NodeLocal DNSCache 기능이 도입 되었다. NodeLocal DNSCache 기능은 쿠버네티스의 각 노드에 DNS 캐시 기능을 가지고 있는 파드를 데몬셋으로 배치 하고, 파드의 DNS 쿼리 및 응답을 coredns가 대신하게 되는 구조이다.

Pod <--> NodeLocal DNSCache(169.254.25.10) <--> iptables <--> coredns(10.233.0.3)

> 참고: NodeLocal DNSCache
> https://kubernetes.io/docs/tasks/administer-cluster/nodelocaldns/

NodeLocal DNSCache 기능에 대해 간단하게 살펴보자.

kube-system 네임스페이스에 nodelocaldns 데몬셋 컨트롤러가 존재한다.
```
$ kubectl get daemonsets.apps -l k8s-app=kube-dns -n kube-system

NAME           DESIRED   CURRENT   READY   UP-TO-DATE   AVAILABLE   NODE SELECTOR   AGE
nodelocaldns   4         4         4       4            4           <none>          30h
```
현재 우리 구성은 마스터 1대, 노드 3대로 총 4대의 구성이기 때문에 4개의 파드가 배포된다.

nodelocaldns 관련 파드 목록을 확인해보자.
```
$ kubectl get pods -l k8s-app=nodelocaldns -n kube-system

NAME                 READY   STATUS    RESTARTS   AGE
nodelocaldns-gnrcf   1/1     Running   0          30h
nodelocaldns-kwnbf   1/1     Running   0          30h
nodelocaldns-pptjw   1/1     Running   0          30h
nodelocaldns-wppmz   1/1     Running   0          30h
```

nodelocaldns 파드의 아규먼트 설정이다. 이는 각 노드에서 169.254.25.10 IP인 링크로컬 주소 대역을 사용해 DNS 캐시 서비스를 제공한다는 의미이다.
```
$ kubectl describe pods -l k8s-app=nodelocaldns -n kube-system | grep -A 2 Args

    Args:
      -localip
      169.254.25.10
--
    Args:
      -localip
      169.254.25.10
--
    Args:
      -localip
      169.254.25.10
--
    Args:
      -localip
      169.254.25.10
```

#### (4) FQDN을 이용한 서비스 탐색
다시 본론으로 돌아가서, nettool 파드를 이용해 서비스의 FQDN을 이용해 접근해보자.
```
$ kubectl run nettool -it --image=c1t1d0s7/network-multitool --generator=run-pod/v1 --rm=true bash

bash-5.0# curl http://mynapp-svc
Message: Hello World!
Hostname: mynapp-rs-t8chb
Platform: linux
Uptime: 109829
IP: 10.233.101.14
DNS: 169.254.25.10

bash-5.0# curl http://mynapp-svc.default
Message: Hello World!
Hostname: mynapp-rs-t5wqf
Platform: linux
Uptime: 109791
IP: 10.233.103.15
DNS: 169.254.25.10

bash-5.0# curl http://mynapp-svc.default.svc.cluster.local
Message: Hello World!
Hostname: mynapp-rs-m65m4
Platform: linux
Uptime: 109760
IP: 10.233.76.14
DNS: 169.254.25.10

bash-5.0# exit
exit
Session ended, resume using 'kubectl attach nettool -c nettool -i -t' command when the pod is running
pod "nettool" deleted
```
mynapp-svc 서비스에 접근 하기위한 FQDN 주소는 mynapp-svc.default.svc.cluster.local 이다.
- mynapp-svc: 서비스의 이름
- default: 서비스 리소스의 네임스페이스 이름
- svc: 서비스 그 자체를 의미함
- cluster.local: 쿠버네티스 클러스터 도메인

FQDN의 주소 형식은 다음과 같다.
<리소스(서비스) 이름>.<네임스페이스>.<리소스 종류>.<클러스터 도메인>
mynapp-svc.default.svc.cluster.local

#### (5) 컨트롤러 및 서비스 리소스 삭제
지금까지 생성했던 리소스를 확인한다.
```
$ kubectl get all
NAME                        READY   STATUS    RESTARTS   AGE
pod/mynapp-rs-m65m4         1/1     Running   0          99m
pod/mynapp-rs-nport-bfqlk   1/1     Running   0          59m
pod/mynapp-rs-nport-mclc4   1/1     Running   0          59m
pod/mynapp-rs-nport-v979s   1/1     Running   0          59m
pod/mynapp-rs-t5wqf         1/1     Running   0          99m
pod/mynapp-rs-t8chb         1/1     Running   0          99m

NAME                       TYPE        CLUSTER-IP      EXTERNAL-IP   PORT(S)   AGE
service/kubernetes         ClusterIP   10.233.0.1      <none>        443/TCP   30h
service/mynapp-svc         ClusterIP   10.233.29.60    <none>        80/TCP    102m
service/mynapp-svc-nport   ClusterIP   10.233.44.249   <none>        80/TCP    68m

NAME                              DESIRED   CURRENT   READY   AGE
replicaset.apps/mynapp-rs         3         3         3       99m
replicaset.apps/mynapp-rs-nport   3         3         3       68m
```

레플리카셋 컨트롤러와 서비스의 모든 리소스를 삭제한다.
```
$ kubectl delete replicasets.apps,services --all

replicaset.apps "mynapp-rs" deleted
replicaset.apps "mynapp-rs-nport" deleted
service "kubernetes" deleted
service "mynapp-svc" deleted
service "mynapp-svc-nport" deleted
```

> 참고  
> service/kubernetes 서비스는 삭제되더라도 다시 생성된다.
