## 4. 컨피그 맵을 이용한 사용자화
컨피그 맵(ConfigMap)은 간단한 설정, 변수, 설정 파일의 값을 키/값 쌍 형태 로 저장하고 사용할 수 있는 쿠버네티스 저장소다.

설정 값을 앞서 살펴본 환경 변수나 명령의 인수로 제공할 수도 있고, 설정 파일(예: nginx.conf) 및 간단한 파일을 저장했다가 볼륨으로 제공할 수도 있다.

### 1) 컨피그 맵 정의 및 확인
컨피그 맵 생성을 위해 YAML 형식의 오브젝트를 작성하여 생성할 수도 있지만, 간단하게 kubectl create configmap 명령을 이용할 수도 있다.

#### (1) 컨피그 맵 생성 사용법
```
kubectl create configmap <NAME> [--from-file=[key=]source][--from-literal=key1=value1]
```
- --from-file 옵션
  * 키/값이 있는 파일 지정
  * 키를 별도로 지정하지 않으면 파일명이 키가 됨
- --from-literal 옵션
  * 키=값 문자로 지정 

#### (2) --from-literal=key=value 형식 사용법
```
$ kubectl create configmap my-config1 --from-literal=key1=value1
```
- 키: key1
- 값: value1

#### (3) --from-file=source 형식 사용법
```
$ echo value2 > key2
```

```
$ kubectl create configmap my-config2 --from-file=key2
```
- 키: key2
- 값: value2


#### (4) --from-file=key=source 형식 사용법
```
$ echo value3 > key3.conf
```

```
kubectl create configmap my-config3 --from-file=key3=key3.conf
```
- 키: key3
- 값: value3

#### (5) 컨피그 맵 확인
컨피그 맵 목록을 확인하자.
```

```

```

...
```

```
$ kubectl describe configmap my-config2

```

```
$ kubectl describe configmap my-config3

```

#### (6) 오브젝트 파일을 이용한 컨피그 맵 정의
다음은 YAML 오브젝트를 이용한 컨피그 맵 정의 예제이다.
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config4
data:
  key4: value4
```

### 2) 컨피그 맵 생성 및 확인
파드에게 제공할 포트 정보와 메시지 정보를 담은 파일을 생성한다.

configmap 디렉토리를 생성한다.
```
$ mkdir configmap
```

configmap 디렉토리에 nodeport 파일과 message 파일을 생성한다.
```
$ echo "Hello World by ConfigMap" > configmap/message
```

컨피그 맵에 사용할 파일과 디렉토리 및 데이터는 다음과 같다.
```
configmap/
└── message [Hello World by ConfigMap]
```

해당 파일을 이용하여 컨피그 맵을 생성 해보자.
```
$ kubectl create configmap myapp-message --from-file=configmap/message
```
디렉토리를 지정하면 하나의 컨피그 맵에 여러 키를 저장할 수 있다.

생성한 myapp-message 컨피그 맵을 확인해보자.
```
$ kubectl get configmaps myapp-message

NAME            DATA   AGE
myapp-message   1      9s
```

컨피그 맵에 등록된 키/값을 확인한다.
```
$ kubectl describe configmap myapp-message

Name:         myapp-message
Namespace:    default
Labels:       <none>
Annotations:  <none>

Data
====
message:
----
Hello World by ConfigMap

Events:  <none>
```

### 3) 컨피그 맵을 참조하는 파드 생성
다음은 mynapp-pod-options 컨피그 맵을 참조하는 파드 오브젝트 정의 파일이다.

> myapp-pod-cm.yaml
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod-cm
spec:
  containers:
  - image: ghcr.io/c1t1d0s7/go-myweb
    name: myapp
    env:
    - name: MESSAGE
      valueFrom:
        configMapKeyRef:
          name: myapp-message
          key: message
    args:
    - $(MESSAGE)
    ports:
    - containerPort: 8080
      protocol: TCP
```
- .spec.containers.env: 환경변수
- .spec.containers.env.valueFrom.configMapKeyRef: 컨피그 맵 리소스에서 키를 참조하여 매핑
  * name: 컨피그 맵 리소스의 이름
  * key: 키 이름
- .spec.containers.args 필드
  * 인수를 지정하는 필드
  * $(MESSAGE)로 환경 변수를 참조하고 있다.

파드를 생성 해보자.
```
$ kubectl create -f myapp-pod-cm.yaml

pod/myapp-pod-cm created
```

확인을 위해 포트 포워딩 한다.
```
$ kubectl port-forward myapp-pod-cm 8080:8080

Forwarding from 127.0.0.1:8080 -> 8080
Forwarding from [::1]:8080 -> 8080
```

새로운 터미널에서 확인해보자.
```
$ curl http://localhost:8080

Hello World by ConfigMap
myapp-pod-cm
```
MESSAGE 환경 변수에 의해 변경된 메시지를 확인할 수 있다.

### 4) 컨피그 맵의 볼륨 사용
컨피그 맵에 설정 파일을 저장하고, 이를 볼륨으로 제공할 수 있다.

#### (1) 컨피그 맵에 사용할 설정파일
다음은 컨피그 맵에 등록할 Nginx 설정 파일인 nginx-gzip.conf 파일이다.

설정파을 저장하기 위해 디렉토리를 생성한다.
```
mkdir conf
```

> conf/nginx-gzip.conf
```
server {
    listen              80;
    server_name         myapp.example.com;
    gzip on;
    gzip_types text/plain application/xml;
    location / {
        root   /usr/share/nginx/html;
        index  index.html;
    }
}
```
Nginx 설정파일로 기본 설정에 gzip 압축 기능을 활성화 하였다.

설정 파일을 컨피그 맵으로 생성하자.
```
$ kubectl create configmap nginx-gzip-config --from-file=conf/nginx-gzip.conf

configmap/nginx-gzip-config created
```

컨피그 맵 내용은 다음과 같다. 파일명이 키가 된다.
```
$ kubectl describe configmaps nginx-gzip-config

...
Data
====
nginx-gzip.conf:
----
server {
    listen              80;
    server_name         myapp.example.com;
    gzip on;
    gzip_types text/plain application/xml;
    location / {
        root   /usr/share/nginx/html;
        index  index.html;
    }
}
...
```

#### (2) 컨피그 맵을 볼륨으로 사용하는 파드 생성
다음은 컨피그 맵을 볼륨으로 사용하는 파드 오브젝트이다.

> nginx-pod-compress.yaml
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod-compress
spec:
  containers:
  - image: nginx
    name: nginx-compress
    volumeMounts:
    - name: nginx-compress-config
      mountPath: /etc/nginx/conf.d
    ports:
    - containerPort: 80
      protocol: TCP
  volumes:
  - name: nginx-compress-config
    configMap:
      name: nginx-gzip-config
```
nginx-gzip-config 컨피그 맵을 볼륨으로 선언하고 /etc/nginx/conf.d 디렉토리에 마운트 한다.

> 참고  
> /etc/nginx.conf 기본 설정 파일  
> /etc/nginx/conf.d 추가 설정 디렉토리  

컨피그 맵을 볼륨으로 사용하는 파드를 생성하자.
```
$ kubectl create -f nginx-pod-compress.yaml

pod/nginx-pod-compress created
```

파드의 상세정보를 확인해보자.
```
$ kubectl describe pod nginx-pod-compress

Name:         nginx-pod-compress
...
Containers:
...
    Mounts:
      /etc/nginx/conf.d from nginx-compress-config (rw)
...
Volumes:
  nginx-compress-config:
    Type:      ConfigMap (a volume populated by a ConfigMap)
    Name:      nginx-gzip-config
    Optional:  false
...
```
볼륨이 마운트되어 있으며, 볼륨은 nginx-compress-config 컨피그 맵에 의해 제공된 것을 확인할 수 있다.

확인을 위해 포트 포워딩을 한다. nginx 이미지는 기본 포트로 80번을 사용한다. (호스트에서 80번 포트는 이미 사용 중이므로 열 수 없다.)
```
$ kubectl port-forward nginx-pod-compress 8080:80

Forwarding from 127.0.0.1:8080 -> 80
Forwarding from [::1]:8080 -> 80
```

새로운 터미널에서 curl 명령으로 확인해보자. -H 옵션은 HTTP 요청 해더를 추가하는 부분이며, "Accept-Encodong: gzip"은 gzip 압축을 요청을 한다. -I 옵션은 HTTP 해더만 보겠다는 의미이다.
``` 
$ curl -H "Accept-Encoding: gzip" -I http://localhost:8080

HTTP/1.1 200 OK
Server: nginx/1.17.9
Date: Wed, 18 Mar 2020 11:39:22 GMT
Content-Type: text/html
Last-Modified: Tue, 03 Mar 2020 14:32:47 GMT
Connection: keep-alive
ETag: W/"5e5e6a8f-264"
Content-Encoding: gzip
```
응답 해더에 "Content-Encoding: gzip" 항목이 보인다면 응답이 압축 되었다는 의미다.

압축 요청을 하지 않은 경우와 비교해 보면 알 수 있다.
```
$ curl -I http://localhost:8080

HTTP/1.1 200 OK
Server: nginx/1.17.9
Date: Wed, 18 Mar 2020 11:44:45 GMT
Content-Type: text/html
Content-Length: 612
Last-Modified: Tue, 03 Mar 2020 14:32:47 GMT
Connection: keep-alive
ETag: "5e5e6a8f-264"
Accept-Ranges: bytes
```

> 참고  
> .spec.containers.volumeMounts.subPath 필드  
> 방금 생성한 파드의 볼륨 마운트 설정은, /etc/nginx/conf.d/ 디렉토리에 컨피그 맵 파일을 마운트 하게 된다.  
> 기존에 /etc/nginx/conf.d/ 디렉토리에 기존 파일이 있었다면, 기존 파일이 삭제되는 것은 아니지만 마운트된 상태에서는 사용할 수 없다. 즉, 파일을 디렉토리 전체에 마운트 하지 말고, 개별 파일로 마운트 할 수 있다.
> ```
> spec:
>   containers:
>     volumeMounts:
>     - name: nginx-config
>       mountPath: /etc/nginx/conf.d/nginx-gzip.conf
>       subPath: nginx-gzip.conf
> ```
> 이것은 쿠버네티스의 기능이 아니라 리눅스 시스템 자체 성질에 기인한 것이다.  

### 5) 리소스 삭제
```
$ kubectl delete -f myapp-pod-cm.yaml -f nginx-pod-compress.yaml

pod "myapp-pod-cm" deleted
pod "nginx-pod-compress" deleted
```
