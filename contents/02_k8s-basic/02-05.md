## 5. 명령형 명령어를 사용한 애플리케이션 실행
명령형 명령어를 이용하여 쿠버네티스 오브젝트를 생성해 애플리케이션을 관리해 보자.

### 1) 레플리케이션 컨트롤러(파드) 생성
```
$ kubectl run myweb-1st-app --image=c1t1d0s7/myweb --port=8080 --generator=run/v1

kubectl run --generator=run/v1 is DEPRECATED and will be removed in a future version. Use kubectl run --generator=run-pod/v1 or kubectl create instead.
replicationcontroller/myweb-1st-app created
```

kubectl run 명령을 사용하여 레플리케이션 컨트롤러를 생성할 수 있다. 컨트롤러의 이름은 myweb-1st-app 이며, 컨테이너에 사용할 이미지는 이전에 사용자 지정으로 빌드 한 myweb 이미지를 사용하며, 애플리케이션 포트는 8080 으로 접근한다.

일반적으로 --generator 옵션은 사용하지 않지만, 이 옵션을 사용하여 복제 컨트롤러를 사용하여 파드를 생성하도록 한다. --generator 옵션을 사용하지 않으면 디플로이먼트 오브젝트가 생성된다. 지금은 디플로이먼트 오브젝트를 생성하지 않으려고 한다. 이와 관련된 오브젝트의 자세한 내용은 나중에 알아본다.

> 참조  
> generator 옵션  
> https://kubernetes.io/docs/reference/kubectl/conventions/#generators

### 2) 파드 확인
```
$ kubectl get pods

NAME                  READY   STATUS    RESTARTS   AGE
myweb-1st-app-lqr42   1/1     Running   0          2m6s
```
현재 myweb-1st-app-lqr42 파드에 하나의 컨테이너가 동작하고 있다.

### 3) 레플리케이션 컨트롤러 확인
```
$ kubectl get replicationcontrollers

NAME            DESIRED   CURRENT   READY   AGE
myweb-1st-app   1         1         1       6m10s
```
현재 myweb-1st-app 레플리케이션 컨트롤러가 하나의 파드를 생성하였다.

### 4) 서비스 생성
외부에서 클러스터 내부의 컨테이너에 접근할 수 있도록 서비스 오브젝트를 생성해서 노출해야 한다.
```
$ kubectl expose replicationcontroller myweb-1st-app --type=LoadBalancer --name myweb-svc

service/myweb-svc exposed
```
생성하는 서비스 이름은 myweb-svc 이다. 서비스 타입은 나중에 알아본다.

### 5) 서비스 확인
```
$ kubectl get services

NAME         TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
kubernetes   ClusterIP      10.233.0.1     <none>        443/TCP          4h36m
myweb-svc    LoadBalancer   10.233.12.29   <pending>     8080:31289/TCP   14s
```
생성된 서비스는 myweb-svc 이고, kubernetes 서비스는 기본적으로 존재하는 서비스이다.

> 참고  
> myweb-svc 서비스에 접근하기 위한 EXTERNAL-IP의 상태가 pending 인데 이는 외부 로드밸런서가 없어서 외부 IP를 얻지 못해서이다.

### 6) 서비스 접근
```
$ curl http://192.168.56.11:31289
Message: Hello World!
Hostname: myweb-1st-app-lqr42
Platform: linux
Uptime: 17848
IP: 10.233.103.3
DNS: 169.254.25.10
```
주소는 ```kubectl get nodes -o wide``` 명령의 아무 노드나 지정하면 되고, 포트는 myweb-svc의 포트를 지정한다.

### 7) 파드 스케일링
```
$ kubectl scale replicationcontroller myweb-1st-app --replicas=3

replicationcontroller/myweb-1st-app scaled
```
앞서 살펴보았지만 현재 myweb-1st-app 컨트롤러는 하나의 파드를 가지고 있었다. 이를 3개의 복제본으로 스케일링한다.

### 8) 스케일링 결과 확인
```
$ kubectl get pods

NAME                  READY   STATUS    RESTARTS   AGE
myweb-1st-app-lqr42   1/1     Running   0          30m
myweb-1st-app-ph9jg   1/1     Running   0          49s
myweb-1st-app-rxjdb   1/1     Running   0          49s
```
AGE가 비교적 최근인 2개의 파드가 스케일링에 의해 최근에 생성된 파드다.

```
$ kubectl get replicationcontrollers

NAME            DESIRED   CURRENT   READY   AGE
myweb-1st-app   3         3         3       30m
```

### 9) 서비스 요청 시 분산된 접근 확인
조금의 시간차를 두고 같은 주소를 여러번 확인해 본다.

```
$ curl http://192.168.56.11:31289

Message: Hello World!
Hostname: myweb-1st-app-ph9jg
Platform: linux
Uptime: 18111
IP: 10.233.76.3
DNS: 169.254.25.10
```

```
$ curl http://192.168.56.11:31289

Message: Hello World!
Hostname: myweb-1st-app-rxjdb
Platform: linux
Uptime: 18192
IP: 10.233.101.2
DNS: 169.254.25.10
```

```
$ curl http://192.168.56.11:31289

Message: Hello World!
Hostname: myweb-1st-app-lqr42
Platform: linux
Uptime: 18154
IP: 10.233.103.3
DNS: 169.254.25.10
```
동일한 서비스 주소에 접근하더라도 서로 다른 결과를 보여준다. 즉, 다른 파드의 컨테이너에 접근하는 것을 확인할 수 있다.

### 10) 생성한 리소스 삭제
이제까지 명령으로 생성한 리소스를 확인하고 삭제한다.

```
$ kubectl get all

NAME                      READY   STATUS    RESTARTS   AGE
pod/myweb-1st-app-lqr42   1/1     Running   0          32m
pod/myweb-1st-app-ph9jg   1/1     Running   0          3m33s
pod/myweb-1st-app-rxjdb   1/1     Running   0          3m33s

NAME                                  DESIRED   CURRENT   READY   AGE
replicationcontroller/myweb-1st-app   3         3         3       32m

NAME                 TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
service/kubernetes   ClusterIP      10.233.0.1     <none>        443/TCP          4h44m
service/myweb-svc    LoadBalancer   10.233.12.29   <pending>     8080:31289/TCP   8m2s
```
전체 오브젝트의 목록을 확인한다.

```
$ kubectl delete pod myweb-1st-app-lqr42

pod "myweb-1st-app-lqr42" deleted
```
특정(myweb-1st-app-lqr42) 파드를 삭제 해 본다.

myweb-1st-app-lqr42 파드를 삭제한 후 다시 전체 목록을 확인한다.
```
$ kubectl get all

NAME                      READY   STATUS    RESTARTS   AGE
pod/myweb-1st-app-lcnxp   1/1     Running   0          44s
pod/myweb-1st-app-ph9jg   1/1     Running   0          5m39s
pod/myweb-1st-app-rxjdb   1/1     Running   0          5m39s

NAME                                  DESIRED   CURRENT   READY   AGE
replicationcontroller/myweb-1st-app   3         3         3       34m

NAME                 TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
service/kubernetes   ClusterIP      10.233.0.1     <none>        443/TCP          4h46m
service/myweb-svc    LoadBalancer   10.233.12.29   <pending>     8080:31289/TCP   10m
```
myweb-1st-app-lcnxp 파드가 새로 생성된 것을 확인할 수 있다.

```
$ kubectl delete replicationcontrollers myweb-1st-app

replicationcontroller "myweb-1st-app" deleted
```
myweb-1st-app 레플리케이션 컨틀롤러를 삭제한다.

```
$ kubectl get all

NAME                      READY   STATUS        RESTARTS   AGE
pod/myweb-1st-app-lcnxp   1/1     Terminating   0          2m42s
pod/myweb-1st-app-ph9jg   1/1     Terminating   0          7m37s
pod/myweb-1st-app-rxjdb   1/1     Terminating   0          7m37s

NAME                 TYPE           CLUSTER-IP     EXTERNAL-IP   PORT(S)          AGE
service/kubernetes   ClusterIP      10.233.0.1     <none>        443/TCP          4h48m
service/myweb-svc    LoadBalancer   10.233.12.29   <pending>     8080:31289/TCP   12m
```
컨트롤러를 삭제하면 컨트롤러에 연결된 파드도 같이 삭제된다.

```
$ kubectl delete service myweb-svc

service "myweb-svc" deleted
```
myweb-svc 서비스를 삭제한다.

```
$ kubectl get all

NAME                 TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
service/kubernetes   ClusterIP   10.233.0.1   <none>        443/TCP   4h49m
```
삭제가 되었는지 전체 목록을 확인한다. kubernetes 서비스는 원래 존재하는 서비스다.
