## 4. 명령형 커맨드를 사용한 애플리케이션 실행
명령형 커맨드를 이용하여 쿠버네티스 오브젝트를 생성해 애플리케이션을 관리해 보자.

### 1) 디플로이먼트 생성
```
$ kubectl create deployment myapp --image=ghcr.io/c1t1d0s7/go-myweb

deployment.apps/myapp created
```

kubectl create 명령을 사용하여 디플로이먼트 리소스를 생성할 수 있다. 디플로이먼트 컨트롤러의 이름은 myapp 이며, 컨테이너에 사용할 이미지는 ghcr.io/c1t1d0s7/go-myweb 이미지를 사용한다.

### 2) 리소스 확인
```
$ kubectl get deployments,replicasets,pods
NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/myapp   1/1     1            1           2m1s

NAME                               DESIRED   CURRENT   READY   AGE
replicaset.apps/myapp-79877ddf67   1         1         1       2m1s

NAME                         READY   STATUS    RESTARTS   AGE
pod/myapp-79877ddf67-zmjb4   1/1     Running   0          2m1s
```

현재 myapp 디플로이먼트 컨트롤러 리소스가 배포되었으며, 디플로이먼트에 의해 myapp-79877ddf67 레플리카셋 컨트롤러 리소스가 생성되었으며, 레플리카셋 리소스에 의해 myapp-79877ddf67-zmjb4 파드가 생성되었다. 별도로 복제본 개수를 지정하지 않았기 때문에 파드는 하나가 생성되었다.

### 3) 서비스 생성
외부에서 클러스터 내부의 컨테이너에 접근할 수 있도록 서비스 오브젝트를 생성해서 노출해야 한다.
```
$ kubectl expose deployment myapp --port=80 --protocol=TCP --target-port=8080 --name myapp-svc --type=LoadBalancer

service/myapp-svc exposed
```
생성하는 서비스 이름은 myapp-svc 이다. 서비스 타입은 나중에 알아본다.

### 4) 서비스 확인
```
$ kubectl get services myapp-svc

NAME        TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)        AGE
myapp-svc   LoadBalancer   10.233.19.224   192.168.56.200   80:31266/TCP   32s
```
생성된 서비스는 myapp-svc 이고, LoadBalancer 타입의 외부 주소는 192.168.56.200 이며, 80번 포트로 접근한다.

### 5) 서비스 접근

#### (1) LoadBalancer 타입의 서비스 접근
IP는 LoadBalancer의 EXTERNAL-IP로 접근한다.
```
$ curl http://192.168.56.200

Hello World!
myapp-79877ddf67-zmjb4
```

#### (2) NodePort 타입의 서비스 접근
노드의 IP(192.168.56.21~23)와 PORT(S)의 콜론 구분자의 두번째 포트로 접근한다.
```
curl http://192.168.56.21:31266

Hello World!
myapp-79877ddf67-zmjb4
```

노드의 주소는 ```kubectl get nodes -o wide``` 명령으로 확인할 수 있다.

### 6) 파드 스케일링
```
$ kubectl scale deployment myapp --replicas=3

deployment.apps/myapp scaled
```
앞서 살펴보았지만 현재 myapp 디플로이먼트 컨트롤러는 하나의 파드를 가지고 있었다. 이를 3개의 복제본으로 스케일링한다.

### 7) 스케일링 결과 확인
```
$ kubectl get deployments,replicasets,pods

NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/myapp   3/3     3            3           12m

NAME                               DESIRED   CURRENT   READY   AGE
replicaset.apps/myapp-79877ddf67   3         3         3       12m

NAME                         READY   STATUS    RESTARTS   AGE
pod/myapp-79877ddf67-br8t2   1/1     Running   0          34s
pod/myapp-79877ddf67-mdh4v   1/1     Running   0          34s
pod/myapp-79877ddf67-zmjb4   1/1     Running   0          12m
```
AGE가 비교적 최근인 2개의 파드가 스케일링에 의해 최근에 생성된 파드다.

### 8) 서비스 요청 시 분산된 접근 확인
조금의 시간차를 두고 같은 주소를 여러번 확인해 본다.

```
$ curl http://192.168.56.200

Hello World!
myapp-79877ddf67-mdh4v
```

```
$ curl http://192.168.56.200

Hello World!
myapp-79877ddf67-br8t2
```

```
$ curl http://192.168.56.200

Hello World!
myapp-79877ddf67-zmjb4
```

동일한 서비스 주소에 접근하더라도 서로 다른 결과를 보여준다. 즉, 다른 파드의 컨테이너에 접근하는 것을 확인할 수 있다.

### 9) 생성한 리소스 삭제
이제까지 명령으로 생성한 리소스를 확인하고 삭제한다.

전체 오브젝트의 목록을 확인한다.
```
$ kubectl get all

NAME                         READY   STATUS    RESTARTS   AGE
pod/myapp-79877ddf67-br8t2   1/1     Running   0          2m31s
pod/myapp-79877ddf67-mdh4v   1/1     Running   0          2m31s
pod/myapp-79877ddf67-zmjb4   1/1     Running   0          14m

NAME                 TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)        AGE
service/kubernetes   ClusterIP      10.233.0.1      <none>           443/TCP        83m
service/myapp-svc    LoadBalancer   10.233.19.224   192.168.56.200   80:31266/TCP   8m29s

NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/myapp   3/3     3            3           14m

NAME                               DESIRED   CURRENT   READY   AGE
replicaset.apps/myapp-79877ddf67   3         3         3       14m
```

특정(myapp-79877ddf67-zmjb4) 파드를 삭제 해 본다.
```
$ kubectl delete pod myapp-79877ddf67-zmjb4

pod "myapp-79877ddf67-zmjb4" deleted
```

myapp-79877ddf67-zmjb4 파드를 삭제한 후 다시 전체 목록을 확인한다.
```
$ kubectl get all

NAME                         READY   STATUS    RESTARTS   AGE
pod/myapp-79877ddf67-br8t2   1/1     Running   0          3m19s
pod/myapp-79877ddf67-mdh4v   1/1     Running   0          3m19s
pod/myapp-79877ddf67-xtpf7   1/1     Running   0          12s

NAME                 TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)        AGE
service/kubernetes   ClusterIP      10.233.0.1      <none>           443/TCP        83m
service/myapp-svc    LoadBalancer   10.233.19.224   192.168.56.200   80:31266/TCP   9m17s

NAME                    READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/myapp   3/3     3            3           15m

NAME                               DESIRED   CURRENT   READY   AGE
replicaset.apps/myapp-79877ddf67   3         3         3       15m
```
pod/myapp-79877ddf67-xtpf7 파드가 새로 생성된 것을 확인할 수 있다.

myapp 디플로이먼트 컨트롤러를 삭제한다.
```
$ kubectl delete deployments myapp

deployment.apps "myapp" deleted
```

컨트롤러를 삭제하면 컨트롤러가 관리하던 파드도 같이 삭제된다.
```
$ kubectl get all

NAME                 TYPE           CLUSTER-IP      EXTERNAL-IP      PORT(S)        AGE
service/kubernetes   ClusterIP      10.233.0.1      <none>           443/TCP        85m
service/myapp-svc    LoadBalancer   10.233.19.224   192.168.56.200   80:31266/TCP   10m
```

myapp-svc 서비스를 삭제한다.
```
$ kubectl delete service myapp-svc

service "myapp-svc" deleted
```

삭제가 되었는지 전체 목록을 확인한다. kubernetes 서비스는 원래 존재하는 서비스다.
```
$ kubectl get all

NAME                 TYPE        CLUSTER-IP   EXTERNAL-IP   PORT(S)   AGE
service/kubernetes   ClusterIP   10.233.0.1   <none>        443/TCP   86m
```
